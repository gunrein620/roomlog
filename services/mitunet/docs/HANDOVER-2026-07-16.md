## ⚠️ 자주 터지는 것

1. **GPU 서버 IP가 임시라 VM 재시작마다 바뀜.** 증상: 편집기에서 업로드 버튼이 아예 안 보임
   (healthz 성공해야 업로드 UI가 켜지는 구조, `viewer/index.html`의 `enableLiveMode`).
   해결: `gcloud compute instances list`로 새 IP 확인 →
   `roomlog/.env.local`의 `MITUNET_INTERNAL_SERVICE_URL` 수정 →
   `docker compose up -d --force-recreate web`. (사용자가 고정 IP는 안 쓰기로 결정)
2. **web 컨테이너는 `.env.local`을 `env_file`로 읽음** (docker-compose.yml에 오늘 추가).
   환경변수 바꾸면 반드시 `--force-recreate`.
3. **viewer/ 파일은 컨테이너에 read-only 마운트라 수정 즉시 반영** (재빌드 불필요).
   룸로그 `apps/web/src/` 수정은 `docker compose up -d --build web` 필요.
4. **⚠️ 브라우저가 `viewer/*.mjs`를 캐시해서 수정이 반영 안 됨.** 파일은 새 것인데 페이지는 옛
   모듈을 씀 (import URL에 쿼리가 없어 재검증 없이 디스크 캐시 사용). 5차 세션에서 이것 때문에
   "고쳤는데 렌더가 그대로"로 한참 헤맴. 검증할 땐 import URL에 `?v=<mtime>` 스탬프를 붙이거나
   (scratchpad `serve_viewer.py`가 이 방식), 페이지에서 `import("...?v="+Date.now())`로 확인할 것.
   `document.getElementById("scene")` 캡처 대신 **모듈 소스를 직접 확인**(`fn.toString()`)이 확실.

## 창문 톱니 정형화 (4차 세션, 확정)

3D 창 유리가 울퉁불퉁한 문제 — YOLO 세그먼트 외곽을 원본 그대로 스탬프해서 발생.
- `review_edits.py` `_apply_segmented_window_masks`: 스탬프 전 `cv2.approxPolyDP`로 외곽 단순화.
  `epsilon = 0.005 * arcLength` (**0.5% 확정**). 잔 톱니 제거 + 곡선 발코니 창의 곡률은 보존.
- ⚠️ 처음 `cv2.minAreaRect`로 정형화했다가 **곡선이 통짜 직선 슬래브가 되고 두꺼워져 되돌림**.
  곡선 창엔 절대 bounding rect 쓰지 말 것 — approxPolyDP(모양 보존)가 정답.
- epsilon 튜닝 참고(실제 곡선 도면 `APT_FP_STR_001681862` 곡선 창 최대 꼭짓점):
  0.3%→14, 0.5%→10, 1%→8, 1.5%→7. 낮을수록 곡선 촘촘/톱니 위험, 높을수록 매끈/각짐. 0.5%가 절충.
- 테스트: `test_compose_edits_smooths_jagged_window_segments` — 꼭짓점 개수 대신 **크기 보존**만 검증
  (mask_to_polygons가 래스터를 재추적해 계단 꼭짓점을 더해 개수 기반은 취약). 파이썬 104개 통과.

## 마루 바닥이 방에 안 깔리던 문제 (5차 세션 — 해결)

증상: 3D/가구배치 화면에서 건물 안 바닥이 대부분 회색(=외부 판정)이고 창가 몇 군데만 마루.
원인 **2개**였고 둘 다 수정 (실제 도면 `APT_FP_STR_001681862_p0.png` 기준 실내 6.5% → 35.4%,
오버레이로 건물 내부 전체가 채워짐을 확인):

1. **룸로그 프록시가 `compose-edits`에 `wall_polygon_mode=legacy`를 강제** → 응답의 door 폴리곤이
   **항상 0개**. legacy는 문 자리에 벽 픽셀을 되살리는 모드라 door 폴리곤이 안 나옴.
   뷰어의 실내 마스크는 door 폴리곤으로 문간 틈을 막고 flood fill을 하는데, 그게 없으니 바깥
   물이 문간마다 들어와 방 전체가 "외부"로 판정됨. → `mitunet-proxy.ts`의 강제 지정 제거
   (서버 기본값 exact 사용). 스펙도 갱신. **web 재빌드 필요했음.**
   ※ 로컬에서 compose를 직접 호출하면 door 9개가 나와서 서버는 멀쩡해 보임 — 프록시 경유로
   재현해야 보임. 디버깅 시 주의.
2. **기각된(rejected) 문이 문간을 안 막음** → 남은 방들이 계속 샘. 문간은 YOLO 문 채택 여부와
   무관하게 **벽 마스크에 실재하는 틈**이라, 기각돼도 flood 차단은 해줘야 함.
   → `floor-finishes.mjs`의 `buildInteriorMask(polygons, w, h, openings)` 4번째 인자 추가:
   `valid=false`인 문 중 **conf>=0.6**만 임시 flood 차단벽으로 사용 (서버 `MIN_DOOR_CONFIDENCE`와
   동일한 기준 — 0.51/0.55짜리 오탐이 열린 공간을 막아버리는 것 방지. 실측상 저신뢰 문은
   커버리지에 기여 0). 차단벽은 임시라 렌더 지오메트리/최종 마스크는 그대로.
   이 도면에서 기각된 문: `moved-too-far`×1, `no-nearby-wall`×2, `unsupported-ends`×1 (전부 conf 0.77~0.94).

**남은 한계**: 벽 마스크 자체에 없는 벽(욕실 칸막이 등)은 여전히 문제 — 기존 항목과 동일.
