# 원격 매물 3D 투어 — 기술 검토 & 설계 방향

집주인이 매물의 3D 공간을 만들어두면 임차인이 직접 가보지 않고 둘러보는 기능. 사진 vs 영상 캡처 비교, 기존 도면→3D 파이프라인(`apps/web/src/app/floor-plan-3d/`)과의 하이브리드 설계, 기술 챌린지 후보를 정리한다. (2026-07-03 논의 기준)

**전제**: 대상 매물은 원룸·오피스텔, 방 1~2개, 10평 이하. 웹 스택은 Next.js 16 + r3f 9 + three 0.185(설치됨), 백엔드 NestJS + Prisma/Postgres.

## 1. 결론 요약

- **캡처는 영상(1~2분 워크스루), 내부 처리는 사진.** 서버에서 키프레임 150~300장을 추출해 사진 파이프라인(SfM → 3D Gaussian Splatting)에 넣는다. 사진 직접 촬영은 비전문가(집주인)가 오버랩·루프 규칙을 지키지 못해 실패한다 — Luma·Polycam 등 소비자 서비스가 전부 영상으로 수렴한 이유.
- **재구성은 서버(또는 외부 API), 브라우저는 재생만.** 온디바이스 재구성은 비현실적. 몇 주 범위에선 재구성을 외부 API(Luma AI, KIRI Engine)에 맡기고, 검증·정합·렌더링을 자체 구현하는 게 정답.
- **뷰어는 [Spark](https://sparkjs.dev/)** (World Labs의 three.js용 3DGS 렌더러). r3f 씬에 `<primitive>`로 통합 가능, .ply/.spz/.sog 지원, 모바일 양호. 보수적 대안: [GaussianSplats3D](https://github.com/mkkellogg/GaussianSplats3D) (단, 자체 렌더 루프 성향이라 r3f 통합은 Spark가 깔끔).
- **"학습" 오해 주의**: 3DGS의 training은 업로드 1건을 3D로 바꾸는 장면별 최적화(GPU 몇 분)다. 데이터셋 수집·모델 학습이 아니며, 매물이 쌓여도 품질은 안 좋아진다 — 품질은 오직 그 영상의 캡처 품질이 결정.

## 2. 사진 vs 영상 (판단 근거)

| 기준 | 사진 여러 장 | 짧은 영상 워크스루 |
|---|---|---|
| 프레임 개별 품질 | 우위 (블러·압축 없음) | 모션블러·압축 열위 |
| 커버리지(실내 품질의 지배 요인) | 30장으론 구멍투성이 | 자연스럽게 70~80% 오버랩 확보 |
| 캡처 난이도(비전문가) | 규칙 못 지켜 거의 실패 | "천천히 걸으며 1~2분" 한 문장 |
| 처리 비용 | 낮음 | 높지만 키프레임 샘플링으로 통제 |
| SfM 정합 안정성 | 순서 힌트 없음 | 프레임 연속성이 인접성 힌트 |

이분법은 사실 가짜다 — 파이프라인은 영상을 받아도 프레임을 추출해 사진처럼 처리한다. 비교의 실체는 "캡처 UX를 뭘로 하느냐"이고, 캡처하는 사람이 집주인이라는 점에서 영상이 이긴다.

## 3. 아키텍처 (몇 주 범위)

```
집주인 폰 영상 업로드
  → [web] 업로드 UI + 캡처 가이드
  → [api] 잡 테이블(Prisma) + 사전검증 게이트(§6-②)
  → [재구성] 외부 API(Luma/KIRI) 또는 자체 GPU 워커(RunPod/Modal + COLMAP + gsplat)
  → splat 파일(.ply → .spz/.sog 압축, 방 하나 5~15MB)
  → [web] Spark 뷰어 + 도면 하이브리드 투어 UX
```

주차별 현실적 도달점:

1. **1주차**: 업로드 + 외부 API 연동 + splat 뷰어 라우트. 도면 3D에서 방 클릭 → splat 뷰 전환(정합 불필요).
2. **2주차**: 수동 2점 정합 UI + transform 저장 + 미니맵 위치 표시 + 클립 박스 floater 제거.
3. **3주차**: 시점 프리셋 투어 UX(drei `CameraControls`) + 가구 배치 결합.

범위 밖: 자체 GPU 재구성 파이프라인 안정화(데모 이후 과제), 멀티룸 자동 스티칭.

## 4. 도면×splat 하이브리드 (핵심 설계)

기존 파이프라인은 도면 이미지 → 벽 검출 → `convertWallsToWheretoputRoom3D()`로 **mm 실측 스케일의 3D 벽 메시**를 만들어 r3f Canvas에 렌더한다(`RoomlogFloorPlanEditor.tsx`). 이 실측 도면이 splat의 최대 약점 둘을 정확히 메꾼다:

1. **splat엔 절대 스케일이 없다** (단안 SfM은 스케일 불명) → 도면 정합으로 실측 스케일 획득.
2. **splat은 방 밖에 floater(유령 조각)가 뜬다** → 도면 벽 폴리곤 + 층고 = 공짜 클리핑 볼륨.

원룸 1~2개·10평 이하 전제가 리스크를 추가로 제거: 문턱 통과 정합 끊김 거의 없음, 파일 크기 소형, 캡처 1회.

### 정합 방법: 수동 2점 유사변환

풀 자유도는 4개(x, y, yaw, scale) — 바닥 수평(gravity alignment) 가정으로 3D 문제가 2D로 강등된다. 등록 시 1회, splat 뷰의 방 모서리 2곳 ↔ 도면의 같은 모서리 2곳을 클릭하면 유사변환이 닫힌 해로 풀린다. 결과는 `SplatAsset { propertyId, roomId, fileUrl, transform: Json }` 식으로 저장. 정밀도는 5~10cm 오차에 둔감한 소비자(미니맵·프리셋·클립 박스)라 2점이면 충분.

**자동 정합(ICP류)은 몇 주 범위의 함정** — 흰 벽 원룸은 splat 벽 지오메트리가 지저분해 자주 틀리고, 틀렸을 때 고치는 UI를 결국 만들게 된다. 수동으로 시작하고 사례가 쌓인 뒤 검토.

### 정합이 해금하는 기능

| 기능 | 구현 |
|---|---|
| 미니맵 내 위치 점 | splat 카메라 좌표 → 역변환 → 도면 위 점 (Matterport의 그 점) |
| 시점 프리셋 자동 생성 | 도면 방 중심·문 위치에서 계산 |
| floater 자동 제거 | 방 벽 폴리곤 + 층고 클립 박스 → 렌더 splat 수 감소 = FPS 상승 |
| 도면⟷실사 크로스페이드 | 같은 좌표계라 카메라 유지한 채 페이드 |
| 실측 가구 배치 | 기존 `useGLTF` 가구 시뮬레이터를 실사 splat 안에서 — "내 침대가 들어가나"에 답. Matterport에 없는 조합 |

## 5. 자산이 하나만 있을 때 (graceful degradation)

도면과 splat은 **각각 선택적 자산**으로 모델링한다. 하나만으로도 독립 동작하고, 둘 다 있으면 정합 transform이 붙으며 하이브리드 기능이 해금되는 구조. 데모 당일 한쪽이 실패해도 다른 쪽이 시연된다.

- **도면만**: 현행 기능 그대로(구조 3D + 실측 + 가구 배치).
- **영상만**: splat 투어는 무손실 동작. 실측 스케일은 "문 높이 ≈ 2m" 류 기준 1개 입력으로 근사 가능. 미니맵·클립 박스·자동 프리셋은 불가.

splat이 도면 대비 추가로 주는 것: **도면은 구조, splat은 상태.** 채광·마감·노후·창밖 뷰·옵션 실물 등 임차인의 계약 판단을 실제로 움직이는 정보는 도면에 없다. 부가 이점: (a) 3D 재구성은 광각 사진처럼 왜곡으로 속이기 어려워 허위매물 대응 신호가 됨, (b) 도면 없는 매물(구축)과 촬영 불가 매물(거주 중)을 서로 보완해 커버리지 확대.

## 6. 기술 챌린지 후보 (전부 지표 측정 가능)

| # | 챌린지 | 지표 | 성격 |
|---|---|---|---|
| ① | 키프레임 선택 — 3,600프레임 중 어떤 150장? 블러(라플라시안 분산) 제거 + 시차 기반 샘플링 vs 균등 샘플링 | 재구성 품질(PSNR)·처리시간 | 실험형 |
| ② | 업로드 사전검증 — 흰 벽·제자리 회전·저조도를 재구성 전 수 초에 반려. 특징점 밀도, 노출 히스토그램, 옵티컬 플로우 | 재구성 실패율 X%→Y% | 실서비스 가치 최대 |
| ③ | 도면–splat 정합 — 수동 2점 → 자동(바닥 평면 검출 → 중력 정렬 → 벽 방향 히스토그램 yaw → 도면 벽선 매칭) | 정합 오차 cm | 순수 기하, 도면 파이프라인 있어야 성립 |
| ④ | 웹 렌더링 최적화 — .spz/.sog 압축 비교, 프로그레시브 로딩, 클립 박스 프루닝 | 모바일 로딩 시간·FPS before/after | ②③과 연결 |
| ⑤ | GPU 잡 파이프라인 — serverless 콜드스타트 vs 상시 워커, 잡 큐 | 방당 처리 비용·대기시간 | 인프라(자체 호스팅 시) |
| ⑥ | 채광 시뮬레이션 — 도면 셸 + 창 검출(`detectOpeningCandidates`) + 방위 입력 + 태양 공식(SunCalc), 저층은 V-World 3D 건물로 차폐 | 직사광 시간대 예측 | 확장, §7 참고 |

**추천: ②(검증) + ③(정합) + ④(렌더링) 중 팀 역량에 맞는 2개.** ①⑤는 튜닝에 가깝고, ②③④는 설계를 처음부터 끝까지 설명·방어할 수 있는 문제. 특히 ③⑥은 기존 도면 파이프라인이 있어야만 성립해 차별화가 된다. "재구성은 외부 API로 사고 검증·정합·렌더링을 만든다"는 buy-vs-build 판단 자체도 발표 소재.

## 7. 채광: 기록 vs 시뮬레이션

- **splat은 채광을 계산하지 않고 기록한다.** 옆 건물 차폐, 실제 광량, 그림자가 촬영 시점에 픽셀로 찍혀 그대로 3D에 담긴다 — 추가 정보 필요 0. 한계: 촬영 시각 한 시점뿐, 집주인이 유리한 시간·조명 켠 상태로 찍는 바이어스. 즉 "증거"이지 "분석"이 아님. 촬영 시각 메타데이터 표기를 기본으로.
- **시간대별 시뮬레이션(⑥)**: 태양 위치는 공식으로 공짜(SunCalc → three.js 디렉셔널 라이트). 주변 건물은 V-World(국토부 3D 건물)·건축물대장으로 확보 가능. 진짜 어려운 건 **도면의 방위** — 이미지엔 북쪽 정보가 없어 집주인 나침반 입력 1회 또는 건축물대장 배치도 대조 필요. 역할 분담: 기록은 splat, 예측은 도면+태양.

## 8. 리스크 & 데모가 깨지는 지점

최대 리스크는 스택이 아니라 **사용자 캡처 품질의 분산**. 실내 실패 요인(텍스처 없는 흰 벽 → SfM 정합 실패, 거울·유리 반사 → 유령 지오메트리, 저조도 블러, 좁은 방 제자리 회전 → 시차 부족)이 한국 주거에 다 있다.

1. **업로드했는데 결과가 안 나옴** (SfM 실패) — 최악의 실패 모드. 완화: 인앱 캡처 가이드(조명 켜기·천천히 걷기·1~2분) + ② 사전검증 조기 반려.
2. **문턱 통과 정합 끊김** — 완화: 방당 splat 1개, 방 전환은 도면 미니맵으로.
3. **캡처 경로 밖 카메라** — splat은 촬영 시점 근처만 사진 같고 벗어나면 스파이크·번짐 노출. 완화: 자유 이동 금지, 시점 프리셋 + 제한 궤도 카메라.
4. **모바일 대용량 로딩** — 완화: .spz/.sog 압축(방당 5~15MB).
5. **라이브 데모는 반드시 사전 캡처·검증한 splat으로.**

## 9. 상품 티어 구조 (2026-07-04 확정)

각 티어는 아래 티어를 포함하며 쌓인다. 낮은 티어의 실패가 높은 티어를 막지 않는 계단식 리스크 격리.

| 티어 | 내용 | 상태 |
|---|---|---|
| 0 파노라마 | 360 사진 노드 (Matterport 기본 방식과 동일 체급). 집주인 진입장벽 최소화용 옵션 | 미착수(선택) |
| **기본** | splat 투어 = ②사전검증 + ③도면 정합 + ④최적화. 프리셋 카메라(캡처 경로 근처 고정) | 뷰어 완성, ②③ 진행 중 |
| **프리미엄** | + GPU 재구성 내재화 + ArtiFixer 도면 유도 보정 → **자유 이동 투어**. 배치 처리(등록 후 수십 분) 성격 → 프리미엄 매물 상품과 부합 | 스파이크 대기 |

- 프리미엄도 ③(정합)을 재사용한다 — 커버리지 갭 검출이 정합 위에서 동작. ③은 두 티어 공용 기반.
- 참고: Matterport의 실제 구조는 파노라마 노드 + 보조 mesh. 우리 프리셋 UX는 같은 교훈("촬영 지점에선 완벽, 사이는 절충")의 splat 일반화 — 프리셋 = sweep 지점, 제한 궤도 = 지점 근처 유지.

## 10. ArtiFixer 결합 — 도면 유도 보정 파이프라인 (프리미엄)

[ArtiFixer](https://research.nvidia.com/labs/sil/projects/artifixer/) (NVIDIA, SIGGRAPH 2026): 3DGS의 미관찰 영역 아티팩트를 14B 비디오 확산 모델로 수정. **Apache 2.0**(상업 사용 가능), 코드·가중치 공개([GitHub](https://github.com/nv-tlabs/artifixer), HF `nvidia/ArtiFixer`). 입력은 COLMAP 장면(원본 프레임+포즈) — **spz가 아니라 원본 영상이 필요**하다. 자체 파이프라인: COLMAP → 3DGRUT 재구성 → ArtiFixer 추론 → 3D 재증류.

```
영상 업로드 → 키프레임(①) → COLMAP → 3DGRUT 재구성
                                   ↓
                     도면 정합(③) ← 기존 도면→3D 파이프라인
                                   ↓
   도면의 역할: (a) 커버리지 갭 검출 — 도면 폴리곤 × 카메라 경로
              (b) 보정 궤적 생성 — 갭을 향한 가상 카메라 경로
              (c) 클립 박스 — 벽 밖 floater 제거로 보정 입력 정화
                                   ↓
        ArtiFixer 추론 → 재증류 → .spz → 뷰어 (자유 이동 투어)
```

(a)(b)(c)는 모델 수정 없는 오케스트레이션 레이어 — 도면 자산이 있어야만 가능한 차별화. ArtiFixer는 장면 구조를 모르므로 "어디를 어떤 시점으로 고칠지"를 도면이 제공한다는 것이 핵심 기여. 확장(연구급, 범위 밖): 도면 3D 셸 depth를 확산 조건으로 주입하는 fine-tune.

## 11. GPU 스파이크 런북 (G1 게이트)

**목표**: 예제 씬 1개 end-to-end 실측 → VRAM 피크·단계별 시간·비용 → 프리미엄 go/no-go.

- **인스턴스**: AWS `g6e.2xlarge` (L40S 48GB, 8 vCPU/64GB, 온디맨드 ~$2.2/h, us-east-1 권장). 근거: 14B bf16 가중치 ~28GB+activation → 24GB 탈락, 48GB가 단일 GPU 최소 안전선. L40S는 RT 코어 보유 → 3DGRUT의 OptiX 레이트레이싱 하드웨어 가속 (A100은 RT 코어 없음 — 상위 등급이 오히려 불리). AWS 사다리에서 "단일 GPU로 되는 가장 큰 것" = 중상급.
- **선행**: Service Quotas "Running On-Demand G and VT instances" ≥16 vCPU 증설 (기본 0, 승인 수시간~수일 — **가장 먼저 신청**).
- **셋업**: Deep Learning AMI(Ubuntu, Docker+NVIDIA 툴킷 포함) + EBS 200GB. `git clone --recurse-submodules https://github.com/nv-tlabs/ArtiFixer` → `docker build -f Dockerfile.cuda12` → HF에서 `artifixer-14b.pt` (~28GB).
- **실행**: 레포 예제 COLMAP 씬으로 `prepare_colmap_artifixer_inputs` → `run_inference`. `nvidia-smi` 로그로 VRAM 피크 기록.
- **기록 템플릿**: 단계별 시간(COLMAP/3DGRUT/추론/재증류), VRAM 피크, 인스턴스 시간×단가, 산출 splat 품질 스크린샷.
- **플랜 B**: 48GB OOM → RunPod/Lambda 단일 H100 80GB(~$3–4/h, 크레딧 밖이지만 10케이스 ~$100)로 동일 Docker 이동. 그래도 불가 → 프리미엄을 "SuperSplat 수동 보정 + 클립 박스"로 강등, 기본 티어 쇼케이스는 무손실.
- **예산**: 크레딧 $1000. 10케이스 × 4h × $2.2 ≈ $90, 실험 오버헤드 5배 잡아도 < $450.
- **케이스 수**: 쇼케이스는 3케이스(실스캔 확보 1 + 팀원 1–2)로 충분. 케이스마다 영상 워크스루(ArtiFixer용) + Scaniverse Splat 스캔(기본 티어용) **둘 다** 촬영.
