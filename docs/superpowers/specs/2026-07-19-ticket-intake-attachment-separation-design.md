# 신규 접수 첨부와 채팅 첨부 분리 설계

## 목표

관리자 민원/하자 상세 화면의 `임차인 입력·첨부`에는 신규 접수 시 제출된 사진만 표시한다. 접수 이후 세입자·관리자가 채팅으로 보낸 사진은 각 채팅 메시지에서만 표시한다.

## 원인

현재 web의 `managerTicketAttachmentUrls`가 티켓의 모든 메시지에서 `attachmentUrls`를 합친다. 이 때문에 최초 접수 사진과 이후 채팅 사진이 상세 첨부 영역에 함께 노출된다.

## 변경 설계

- `managerTicketAttachmentUrls`는 시간순 메시지 중 최초 접수 메시지의 첨부 URL만 반환한다.
- URL 앞뒤 공백 제거, 빈 값 제거, 중복 제거 동작은 유지한다.
- 최초 메시지 또는 첨부가 없으면 빈 배열을 반환한다.
- 채팅 메시지 데이터와 채팅 사진 렌더링은 변경하지 않는다.
- 한 번의 채팅 전송에서 사진을 최대 5장으로 제한하는 기존 규칙도 변경하지 않는다. 여러 메시지로 사진을 계속 보낼 수 있다.
- DB 스키마와 API 응답 계약은 변경하지 않는다. 기존 데이터에도 즉시 적용되도록 web 매핑 경계에서 분리한다.

## 검증

- 최초 메시지 사진과 후속 채팅 사진이 함께 있는 티켓에서 최초 사진만 상세 첨부로 반환되는 테스트를 추가한다.
- 최초 메시지에 첨부가 없고 후속 채팅에만 사진이 있으면 상세 첨부가 빈 배열인지 확인한다.
- 상세 첨부 갤러리와 채팅 사진 관련 기존 테스트를 함께 실행한다.
