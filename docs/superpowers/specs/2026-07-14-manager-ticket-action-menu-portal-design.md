# 관리자 티켓 작업 메뉴 Portal 설계

## 문제

민원/하자 관리 테이블은 가로 스크롤을 위해 `overflow-x: auto`를 사용한다. 브라우저에서 이 컨테이너의 세로 overflow도 `auto`로 계산되므로, 테이블 행 내부에 절대 위치로 렌더되는 점 3개 작업 메뉴가 컨테이너 경계를 넘을 때 잘린다. 마지막 세 행에 일괄 적용된 위쪽 열기 규칙은 상단 경계와의 충돌을 더 쉽게 만든다.

## 검토한 접근

1. **`overflow-y: visible` 추가**: 가로 스크롤과 함께 사용할 때 세로 overflow가 다시 `auto`로 계산될 수 있어 근본 해결이 아니다.
2. **테이블 내부에서 위/아래 방향만 동적으로 변경**: 가까운 경계는 피할 수 있지만 메뉴가 스크롤 컨테이너 안에 남아 있어 행 높이와 컨테이너 크기에 따라 다시 잘릴 수 있다.
3. **메뉴를 `document.body` Portal로 렌더**: 스크롤 컨테이너의 클리핑에서 분리되고 뷰포트 기준으로 방향과 좌우 경계를 계산할 수 있다.

3번을 채택한다.

## 컴포넌트 경계

- `TicketActionMenu`는 트리거 버튼, 열림 상태, 바깥 클릭과 Escape 닫기를 담당한다.
- 순수 함수 `placeTicketActionMenu`는 트리거 사각형, 메뉴 크기, 뷰포트 크기, 간격을 입력받아 `top`, `left`, `placement`를 반환한다.
- 메뉴 목록은 `createPortal(..., document.body)`로 렌더하고 `position: fixed` 좌표를 사용한다.
- 기존 티켓별 링크와 접근 가능한 이름은 유지한다.

## 위치 계산

- 기본 방향은 아래쪽이다.
- 아래쪽 공간이 부족하고 위쪽 공간이 더 충분하면 위쪽으로 뒤집는다.
- 메뉴의 좌우 위치는 뷰포트 여백 안으로 제한한다.
- 메뉴가 열린 동안 스크롤 또는 리사이즈가 발생하면 닫아 오래된 좌표가 남지 않게 한다.
- 한 행의 메뉴를 열면 기존에 열린 다른 `<details>` 메뉴에 의존하지 않고 해당 컴포넌트 상태만 사용한다.

## 접근성과 오류 처리

- 트리거는 `aria-haspopup="menu"`, `aria-expanded`, `aria-controls`를 제공한다.
- 메뉴는 `role="menu"`, 링크는 `role="menuitem"`을 유지한다.
- Escape와 바깥 클릭으로 닫힌다.
- Portal을 사용할 수 없는 서버 렌더 단계에는 메뉴 목록을 렌더하지 않는다.

## 테스트

- 위치 계산 순수 함수가 아래쪽 기본 배치, 위쪽 뒤집기, 좌우 경계 제한을 보장한다.
- 컴포넌트 계약 테스트가 Portal 사용, 접근성 속성, Escape/바깥 클릭/스크롤 닫기를 확인한다.
- 기존 관리자 티켓 대시보드 모델 및 화면 계약 테스트를 함께 실행한다.
- Docker 브라우저에서 첫 행, 중간 행, 마지막 행의 메뉴가 테이블 경계에서 잘리지 않는지 확인한다.

## 범위 제외

- 계약 관리 등 다른 테이블의 점 3개 메뉴 공통화
- 신규 UI 라이브러리 또는 Floating UI 의존성 추가
- 인프라 및 배포 설정 변경
