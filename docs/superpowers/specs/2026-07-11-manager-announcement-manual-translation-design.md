# 공지 작성 수동 번역 시작 설계

## 목표

`/manager/messaging/01`의 새 공지 작성 화면에서는 번역 결과를 미리 보여주지 않는다. 관리자가 각 언어의 번역 버튼을 눌렀을 때만 기존 번역 API를 호출하고 해당 언어의 결과를 표시한다.

## 범위

- 새 공지 작성 진입(`id` 검색 파라미터 없음)
- 저장된 공지 재진입(`?id=<draftId>`)
- English, 中文, Tiếng Việt 번역 카드
- 기존 번역 API와 검수 게이트

다른 관리자 화면, 번역 API 구현, 인프라 설정은 변경하지 않는다.

## 동작

### 새 공지

- 한국어 공지 제목과 본문은 기존 데모 초깃값을 유지한다.
- 번역 제목과 본문은 세 언어 모두 빈칸으로 표시한다.
- 번역문이 없으므로 각 언어의 검수 완료 체크박스는 비활성 상태로 시작한다.
- 언어별 번역 버튼을 누르면 해당 언어에 대해서만 기존 `translateAnnouncementAction`을 호출한다.
- 응답을 받은 언어의 제목과 본문만 채우고, 검수 상태는 미완료로 둔다.

### 저장된 공지

- `?id=<draftId>`로 연 초안은 저장된 번역 제목, 본문, 검수 상태를 그대로 표시한다.
- 화면 진입만으로 저장된 번역을 삭제하거나 다시 번역하지 않는다.

### 원문 수정

- 한국어 제목 또는 본문을 수정하면 현재 동작처럼 기존 번역문의 검수 상태만 해제한다.
- 기존 번역문은 유지하여 관리자가 내용을 비교하고 재번역하거나 직접 수정할 수 있게 한다.

## 구현 경계

- 페이지에서 `draftId` 유무로 새 작성과 저장된 초안을 구분한다.
- 새 작성일 때만 `AnnouncementComposer`에 전달하는 초기 초안의 `translations`를 빈 배열로 정규화한다.
- 저장된 초안과 API 응답 구조는 변경하지 않는다.
- 컴포넌트의 언어별 버튼 처리 로직은 기존 방식을 유지한다.

## 오류 처리

- 제목이나 본문이 비어 있을 때 번역 버튼을 누르면 기존 입력 오류 메시지를 표시한다.
- 번역 API가 실패하면 해당 언어의 기존 값은 변경하지 않고 오류 메시지를 표시한다.
- 다른 언어의 번역 결과와 검수 상태에는 영향을 주지 않는다.

## 테스트

- 새 작성 페이지가 데모 초안의 사전 번역을 전달하지 않는지 회귀 테스트로 고정한다.
- 저장된 초안 경로는 API에서 받은 번역을 그대로 전달하는지 확인한다.
- 언어별 버튼이 기존 번역 액션을 호출하는 계약과 긴급 공지 검수 게이트가 유지되는지 확인한다.
- web 단위 테스트, Next.js 빌드, Docker 브라우저에서 초기 빈칸과 버튼 클릭 후 단일 언어 번역을 검증한다.
