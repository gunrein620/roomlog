# Manager Announcement Saved Drafts Modal Design

## 목표

공지 작성 화면의 큰 임시 저장 목록을 제거하고, 상단 `공지 작성` 옆의 `임시 저장` 버튼을 통해 초안 목록을 모달로 확인하고 불러오게 한다.

## 사용자 흐름

1. 관리자가 `/manager/messaging/01` 공지 작성 화면을 연다.
2. 상단 하위 메뉴에서 `공지 작성` 옆의 `임시 저장`을 누른다.
3. 현재 화면 위에 임시 저장된 공지 목록 모달이 열린다.
4. 관리자가 초안의 `불러오기`를 누르면 모달이 닫히고 `/manager/messaging/01?id=<draftId>`에서 해당 내용을 작성 폼에 표시한다.
5. 닫기 버튼, 모달 바깥 영역 또는 `Esc`를 사용하면 작성 중인 초안을 유지한 채 모달만 닫힌다.

## 화면 구성

- `임시 저장` 진입 링크는 `/manager/messaging/01`에서만 상단 `공지 작성` 링크 오른쪽에 표시한다.
- 기존 `ScreenHeader` 아래의 `임시 저장된 공지` 카드는 제거한다.
- 모달 제목은 `임시 저장된 공지`로 표시한다.
- `status === "draft"`인 항목만 `updatedAt` 최신순으로 표시한다.
- 각 행에는 제목, 대상 라벨, 마지막 저장 시각, `불러오기`를 표시한다.
- 제목이 비어 있으면 `제목 없는 공지`를 사용한다.
- 초안이 없으면 `임시 저장된 공지가 없습니다.`를 표시한다.
- 작은 화면에서는 모달 폭을 뷰포트 안에 맞추고 목록 영역만 스크롤 가능하게 한다.

## 상태와 데이터 흐름

- 상단 링크는 현재 작성 중인 `id`를 보존하면서 `drafts=open` 쿼리를 추가한다.
- `apps/web/src/app/manager/messaging/01/page.tsx`는 기존 `listAnnouncementDrafts()` 결과를 모달에 전달한다.
- `drafts=open`이면 클라이언트 모달 컴포넌트가 네이티브 `<dialog>`의 `showModal()`을 호출한다.
- 모달 닫기는 `drafts` 쿼리만 제거하고 현재 `id`는 보존한다.
- `불러오기`는 기존 `?id=<draftId>` 경로, `getAnnouncementDraft(id)`, `prepareAnnouncementDraftForCompose()`를 그대로 사용한다.
- 별도 클라이언트 API 호출이나 전역 상태는 추가하지 않는다.
- 서버와 브라우저의 초안 저장 시각은 `Asia/Seoul`로 고정해 hydration 결과를 동일하게 유지한다.

## 접근성

- 네이티브 `<dialog>`와 `aria-labelledby`를 사용한다.
- 열기 링크에는 `aria-haspopup="dialog"`를 제공한다.
- 닫기 버튼에는 구체적인 접근성 이름을 제공한다.
- `Esc`, 닫기 버튼, 바깥 영역 클릭을 모두 지원한다.
- 네이티브 `cancel` 이벤트와 명시적 `Escape` keydown을 모두 처리한다.
- 키보드 포커스는 네이티브 모달 동작을 따른다.

## 테스트와 검증

- 상단 `임시 저장` 링크가 작성 화면에서만 구성되고 `id`를 보존하는지 순수 함수 단위 테스트로 고정한다.
- 공지 작성 페이지가 기존 카드 대신 모달을 렌더링하는지 계약 테스트로 검증한다.
- 모달이 `<dialog>`, 열림 동기화, 닫기 경로, 빈 상태, `불러오기` 링크를 포함하는지 계약 테스트로 검증한다.
- 기존 초안 필터·정렬·제목 폴백 단위 테스트를 유지한다.
- 날짜 포맷이 실행 환경의 시스템 타임존과 무관하게 한국 시각을 표시하는지 단위 테스트로 검증한다.
- 관련 web 테스트와 web 빌드 후 Docker web 이미지를 재빌드한다.
- 브라우저에서 버튼 노출, 열기, 닫기, 초안 불러오기, 콘솔 오류 부재를 확인한다.

## 비범위

- API, 데이터베이스, 공유 타입 변경
- 인프라 및 Docker 설정 변경
- 초안 삭제, 검색, 페이지네이션
- 발송 완료 공지 표시
- 소통 허브와 다른 관리자 화면에서의 `임시 저장` 버튼 노출
