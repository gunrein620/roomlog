# MitUNet 도면 진입 안정화 설계

## 배경

운영 도면 화면은 HTML이 먼저 표시된 뒤 `unpkg.com`의 Three.js와 Lucide, 여러 로컬 ES 모듈을 모두 읽어야 업로드 클릭 이벤트를 연결한다. 외부 모듈 응답이 느리면 화면은 보이지만 버튼이 반응하지 않는다. 뷰어 HTML과 모듈은 모두 `force-dynamic`과 `Cache-Control: no-store`로 제공되어 재방문 캐시도 사용할 수 없다.

도면 추론 서버는 단일 Uvicorn 워커의 `async` 엔드포인트 안에서 동기 GPU 추론을 실행한다. 동시 요청은 직렬로 밀리고 이벤트 루프를 막을 수 있다. RoomLog 프록시는 업스트림 타임아웃과 처리시간 로그가 없어 장애 시 요청이 무기한 대기하고 원인 구간도 남지 않는다.

## 목표

- 도면 화면이 보이는 즉시 파일 선택이 동작한다.
- Three.js와 아이콘 라이브러리 가용성이 파일 선택 자체를 막지 않는다.
- 변경되지 않은 뷰어 자산은 브라우저가 재사용한다.
- GPU 서버 지연·중단 시 제한된 시간 안에 명확한 오류를 반환한다.
- 동시 추론이 이벤트 루프 전체를 막지 않고, 요청별 처리시간을 운영 로그에서 확인할 수 있다.

## 비목표

- 도면 인식 모델이나 3D 렌더링 결과 변경
- 화면 디자인 전면 개편
- GPU 모델을 여러 프로세스에 복제하는 수평 확장
- CloudFront 신규 구축

## 검토한 접근

### A. 기존 구조에서 CDN 주소만 자체 호스팅 주소로 교체

수정량은 가장 작지만 정적 import가 완료되어야 업로드 이벤트가 연결되는 문제는 남는다. 자체 호스팅 장애나 큰 번들 다운로드가 느릴 때 같은 증상이 반복된다.

### B. 업로드 부트스트랩 분리 + 3D 의존성 자체 호스팅

작은 로컬 스크립트가 파일 선택과 대기 상태를 즉시 연결하고, Three.js 및 편집기는 뒤에서 로드한다. 선택한 파일은 준비 완료 후 한 번만 분석한다. 외부 런타임 CDN을 제거하고 기존 뷰어 동작을 유지할 수 있어 이 방식을 채택한다.

### C. 뷰어 전체를 Next/React 컴포넌트로 재작성

장기 구조는 깔끔하지만 현재 장애 범위보다 크고 렌더링 회귀 위험이 높다. 이번 작업에서는 제외한다.

## 설계

### 1. 즉시 동작하는 업로드 부트스트랩

`services/mitunet/viewer/upload-bootstrap.mjs`를 의존성 없는 선행 모듈로 추가한다. 이 모듈은 Three.js 정적 import가 있는 주 모듈보다 먼저 평가되며 DOM 요소를 찾아 다음 계약을 제공한다.

- 업로드 버튼 클릭 즉시 숨겨진 파일 입력을 연다.
- 파일 선택 시 `window`에 `mitunet-upload-selected` 커스텀 이벤트를 발생시킨다.
- 3D 모듈이 아직 준비되지 않았으면 마지막 선택 파일 하나만 보관한다.
- 3D 모듈이 준비되면 보관 파일을 정확히 한 번 전달한다.
- 외부 Three.js 로딩 여부와 무관하게 파일 선택 UI가 동작한다.

기존 인라인 모듈은 이 이벤트를 구독하고 준비 완료 신호를 보낸다. 기존 `pendingUploadFile` 로직은 부트스트랩 계약으로 대체한다.

### 2. 런타임 외부 CDN 제거

Three.js는 `apps/web`에 이미 설치된 `three@0.185.0`의 브라우저 배포 파일을 사용한다. 기존 뷰어의 `three@0.162.0`에서 버전이 바뀌므로 샘플 렌더링과 도면 편집 회귀 테스트를 반드시 통과해야 한다. RoomLog와 FastAPI 단독 뷰어가 같은 파일을 쓰도록 Three.js 핵심·필요 addon 번들·Draco·Lucide 브라우저 파일을 `services/mitunet/viewer/vendor`에 고정한다. 두 서버 모두 기존 `/viewer-assets` 정적 경로로 이를 제공하므로 외부 CDN이나 Node 패키지 런타임 탐색이 필요 없다.

import map과 Draco decoder 경로는 같은 오리진의 `/floor-plan-3d/mitunet-assets/vendor/`를 사용한다. Pretendard는 사이트 전체 폰트 정책을 따르되 폰트 실패가 기능 초기화를 막지 않으므로 이번 핵심 범위에서는 유지한다.

### 3. 캐시 정책

- 배포 SHA 쿼리가 있는 변환 없는 `.mjs`, `.js`, 이미지, vendor 자산: `public, max-age=31536000, immutable`
- 배포 SHA 쿼리가 없는 변환 없는 자산: `public, max-age=300, must-revalidate`
- 런타임 변환이 필요한 `review-editor.mjs`: `public, max-age=300, must-revalidate`
- HTML: `no-cache`로 재검증은 허용하되 `no-store`는 제거
- API 응답: 계속 `no-store`

배포마다 파일 내용이 바뀔 수 있으므로 `transformMitunetViewerHtml`과 모듈 변환이 `ROOMLOG_DEPLOY_SHA`를 내부 자산 URL 경로에 넣는다. 배포 워크플로가 현재 Git SHA를 Docker build/runtime에 전달한다. 값이 없으면 `dev` 경로를 사용하고 짧은 재검증 캐시만 허용한다. 모듈 안의 `/viewer-assets` import도 같은 버전 경로로 변환해 전체 모듈 그래프가 한 배포 버전에 고정된다.

### 4. GPU 요청 안정성

FastAPI에는 프로세스당 `asyncio.Semaphore(1)`을 두어 GPU 진입을 명시적으로 직렬화한다. 동기 추론은 `asyncio.to_thread`로 이벤트 루프 밖에서 실행한다. 대기와 실행 시간을 로그로 남기며, 헬스체크는 추론 중에도 응답할 수 있어야 한다. 여러 Uvicorn 워커는 GPU 메모리에 모델을 중복 적재하므로 사용하지 않는다.

RoomLog 프록시는 `AbortSignal.timeout`으로 업로드·조합 요청에 제한 시간을 적용하고, endpoint·status·elapsedMs를 구조화 로그로 기록한다. 타임아웃은 504 JSON 응답으로 변환한다.

### 5. 오류 처리

- 3D 모듈 로딩 실패: 파일 선택은 유지하고 화면 상태에 “3D 모듈을 불러오지 못했습니다. 다시 시도해 주세요.” 표시
- GPU 타임아웃: 504와 재시도 가능한 사용자 문구 반환
- GPU 대기 중: 기존 `aria-busy`와 상태 문구 유지
- 중복 파일 이벤트: 마지막 선택 하나만 처리하고 이미 전달한 파일은 재실행하지 않음

## 테스트

- 정적 테스트: HTML에 외부 `unpkg` Three.js/Lucide 주소가 없고 로컬 부트스트랩이 모듈보다 먼저 로드됨
- 부트스트랩 단위 테스트: 준비 전 파일 선택, 준비 후 한 번 전달, 재선택 시 마지막 파일 유지
- asset route 테스트: 파일 종류별 캐시 헤더 검증
- 프록시 단위 테스트: 성공 시간 로그와 업스트림 타임아웃 504 검증
- Python 테스트: 동시 추론이 세마포어를 따르면서 헬스체크 이벤트 루프가 살아 있는지 검증
- 회귀 검증: `pnpm test:web`, MitUNet JS/Python 테스트, `bash scripts/verify.sh`

## 성공 기준

- 네트워크에서 `unpkg.com`을 차단해도 업로드 파일 선택기가 즉시 열린다.
- 재방문 시 불변 뷰어 자산이 브라우저 캐시에서 제공된다.
- GPU 서버가 응답하지 않으면 설정된 시간 내 504가 반환된다.
- 3개 동시 요청에서 헬스체크가 추론 완료까지 차단되지 않는다.
- 기존 샘플, 도면 편집, 2D/3D 전환, RoomLog 저장 흐름 테스트가 모두 통과한다.
