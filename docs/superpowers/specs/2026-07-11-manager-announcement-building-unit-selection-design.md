# 관리자 공지 건물별 호실 선택 설계

## 목표

`/manager/messaging/01`에서 발송 범위를 `호실`로 선택했을 때 관리 중인 모든 호실을 한꺼번에 보여주지 않고, 건물을 먼저 선택한 뒤 해당 건물의 호실만 선택할 수 있게 한다.

## 범위

- 수정 화면: `/manager/messaging/01`
- 수정 대상: 공지 작성 컴포저와 공지 대상 상태 유틸리티 및 관련 web 테스트
- 제외 대상: `/manager/messaging/00`, `/02` 이후 화면, API 계약, 데이터베이스, Docker 및 배포 설정

## 사용자 흐름

1. 사용자가 발송 범위에서 `호실`을 선택한다.
2. 건물 드롭다운이 먼저 표시된다.
3. 선택한 건물에 속한 호실만 체크박스 목록으로 표시된다.
4. 같은 건물 내에서 하나 이상의 호실을 선택할 수 있다.
5. 선택 결과는 기존 대상 요약 규칙에 따라 단일 호실이면 건물명과 호실을, 복수 호실이면 첫 호실과 추가 세대 수를 표시한다.

## 상태 규칙

- 최초 선택 건물은 저장된 대상의 첫 호실이 속한 건물을 우선하고, 없으면 첫 관리 건물을 사용한다.
- 건물을 변경하면 기존 `selectedRoomIds`를 빈 배열로 초기화한다. 다른 건물의 선택이 화면에 숨은 채 발송 대상에 남지 않게 하기 위함이다.
- 호실 목록은 `room.buildingName === selectedBuilding` 조건으로 필터링한다.
- 선택한 건물에 호실이 없으면 체크박스 대신 `선택 가능한 호실이 없습니다.`를 표시한다.
- `전체`와 `건물` 범위의 기존 대상 계산 및 저장 방식은 변경하지 않는다.

## 컴포넌트 및 데이터 흐름

- `AnnouncementComposer`가 선택 건물과 선택 호실 ID 상태를 계속 소유한다.
- 건물 변경 핸들러가 `selectedBuilding`을 갱신하고 `selectedRoomIds`를 초기화한다.
- 순수 유틸리티가 전체 관리 호실에서 선택 건물의 호실만 반환한다. 이 유틸리티를 단위 테스트해 필터 조건과 빈 상태를 고정한다.
- `buildAnnouncementTarget`에는 필터된 목록이 아니라 전체 관리 호실과 선택 ID를 전달한다. 선택 ID 초기화로 다른 건물 대상이 남지 않으며 기존 저장 계약을 유지한다.

## 오류 및 빈 상태

- 건물 목록이 없으면 건물 선택 컨트롤은 빈 상태가 되고 선택 가능한 호실 안내를 표시한다.
- 호실을 선택하지 않은 상태에서는 기존 `호실 선택 필요` 대상 요약과 `발송 대상을 선택해 주세요.` 검증을 유지한다.
- 추가 API 호출이나 네트워크 오류 처리는 필요하지 않다. 관리 호실 데이터는 현재 페이지 입력값을 사용한다.

## 테스트

- 선택한 건물의 호실만 반환하는 순수 함수 단위 테스트
- 존재하지 않거나 빈 건물 선택 시 빈 배열을 반환하는 테스트
- `/01` 소스 계약 테스트에서 호실 범위에도 건물 선택과 필터된 목록을 사용하는지 확인
- web 단위 테스트와 프로덕션 빌드
- Docker 브라우저에서 `호실` 선택 → 건물 변경 → 호실 목록 필터링 → 기존 선택 초기화를 확인

## 완료 조건

- `호실` 범위에서 건물 선택 후 해당 건물 호실만 복수 선택할 수 있다.
- 건물 변경 시 이전 건물 호실이 발송 대상에 남지 않는다.
- `/manager/messaging/01` 외 제품 동작과 API·인프라 파일은 변경되지 않는다.
