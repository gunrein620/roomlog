# 계약 세입자 소통 스레드 생성 설계

## 목표

관리자가 소통 허브에서 기존 스레드 유무와 관계없이 관리 중인 계약 건물과 연결 세입자를 확인하고, 해당 세입자와 일반 대화를 시작할 수 있게 한다.

## 확인된 문제

- 공지 작성 화면은 로그인 응답의 `managedRooms`를 사용하므로 스레드가 없어도 관리 건물과 호실을 선택할 수 있다.
- 소통 허브는 메시지 스레드의 `buildingName`만 모아 건물 선택지를 만든다.
- 따라서 아직 대화가 없는 배포 계정은 계약 건물과 세입자가 연결되어 있어도 `전체 건물`만 보이며 최초 스레드를 만들 수 없다.
- 기존 메시지 생성 API는 관리 호실과 `TenantRoom` 관계를 검증하지만, 관리자가 선택할 수 있는 계약 세입자 목록 API와 시작 UI가 없다.

## 선택한 접근 방식

메시징 도메인에 관리자가 대화 가능한 계약 세입자 목록과 일반 대화 시작 동작을 추가한다.

대상자는 계약 문서의 검토 상태가 아니라 실제 입주 연결의 단일 진실인 `TenantRoom`과 `Room.landlordId`에서 파생한다. 이 방식은 미확정 계약이나 초대 대기 사용자가 대화 대상으로 잘못 노출되는 것을 막고 기존 메시지 권한 검사와 같은 기준을 사용한다.

고려했지만 선택하지 않은 방식:

- 계약 대시보드 API 재사용: 계약 검토용 응답이 크고 미확정·만료·미연결 계약까지 포함할 수 있어 메시지 수신 가능 대상과 의미가 다르다.
- `/auth/me`의 `managedRooms`에 세입자 포함: 모든 관리자 화면의 세션 응답을 키우고 인증 도메인에 메시징 전용 정보를 결합한다.

## 공유 타입과 API

`packages/types`에 다음 계약을 추가한다.

- `ManagerMessagingRecipient`
  - `roomId`
  - `buildingName`
  - `unitId`
  - `tenantId`
  - `tenantName`
  - `existingGeneralThreadId?`
- `StartManagerConversationInput`
  - `roomId`
  - `tenantId`
  - `body`

관리자 메시징 API:

- `GET /manager/messaging/recipients`
  - 로그인 관리자의 `Room.landlordId`에 속하면서 `TenantRoom`으로 연결된 세입자만 반환한다.
  - 같은 세입자의 일반 스레드가 있으면 `existingGeneralThreadId`를 포함한다.
  - 건물명, 호실, 세입자명 순으로 안정 정렬한다.
- `POST /manager/messaging/conversations`
  - `roomId`가 로그인 관리자 소유인지 검증한다.
  - `tenantId`가 해당 호실의 실제 `TenantRoom` 연결인지 검증한다.
  - 기존 일반 스레드가 있으면 새 스레드나 메시지를 만들지 않고 기존 스레드를 반환한다.
  - 기존 스레드가 없으면 입력한 첫 메시지로 `general` 스레드를 생성한다.

기존 범용 `POST /manager/messaging/threads` 동작은 다른 도메인 흐름과의 호환성을 위해 변경하지 않는다.

## 소통 허브 UI

- 상단에 `새 대화` 영역을 추가한다.
- 계약 세입자를 건물 → 호실·세입자 순으로 선택한다.
- 선택 대상에 기존 일반 스레드가 있으면 `기존 대화 열기`로 표시하고 해당 상세 화면으로 이동한다.
- 기존 일반 스레드가 없으면 첫 메시지를 입력하고 `대화 시작`을 누른다.
- 생성 성공 후 `/manager/messaging/04?id=<threadId>`로 이동한다.
- 계약 세입자가 없으면 `계약 연결된 세입자가 없습니다.`라고 표시하며 생성 입력을 비활성화한다.

건물 필터 선택지는 다음 합집합으로 구성한다.

- 기존 메시지 스레드에 포함된 건물
- 대화 가능한 계약 세입자가 있는 건물

따라서 대화가 아직 없는 계약 건물도 드롭다운에 표시된다. 해당 건물을 선택했지만 스레드가 없으면 빈 상태 문구는 `이 건물에는 아직 시작된 대화가 없습니다.`로 표시한다.

## 오류와 권한 처리

- 서버가 전달된 `roomId`, `tenantId`, 기존 스레드 여부를 다시 검증하며 클라이언트 값을 신뢰하지 않는다.
- 계약 연결이 변경되었거나 다른 관리자의 호실이면 403 또는 404로 거부한다.
- 메시지가 비어 있으면 생성하지 않고 입력 오류를 표시한다.
- API 네트워크 오류를 계약 대상 없음으로 위장하지 않고 오류 상태로 표시한다.

## 테스트

- API 도메인 테스트
  - 관리자는 자기 호실에 연결된 세입자만 목록에서 본다.
  - 다른 관리자의 세입자는 노출되지 않는다.
  - 기존 일반 스레드 식별자가 포함된다.
  - 최초 시작은 메시지와 스레드를 하나씩 생성한다.
  - 기존 일반 스레드 시작 요청은 중복 생성하지 않는다.
  - 잘못된 관리 호실·세입자 조합은 거부한다.
- Web 단위 테스트
  - 계약 세입자 건물이 기존 스레드가 없어도 필터 선택지에 포함된다.
  - 대상 선택 상태가 기존 대화 열기와 신규 대화 시작을 올바르게 구분한다.
  - API 경로와 요청 본문이 실제 관리자 메시징 엔드포인트를 사용한다.
- 통합 검증
  - `pnpm test:api`
  - `pnpm test:web`
  - `bash scripts/verify.sh`
  - Docker 재빌드 후 브라우저에서 계약 건물 노출, 신규 생성, 기존 대화 재사용을 확인한다.

## 범위 제외

- 실시간 웹소켓 구조 변경
- 계약 생성·수락 로직 변경
- 공지 작성 화면 변경
- 인프라·Docker·배포 설정 변경
- 기존 결제·퇴실·공지 맥락 스레드의 생성 규칙 변경
