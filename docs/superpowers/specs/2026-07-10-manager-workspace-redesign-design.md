# 관리자 워크스페이스 UI 개편 설계

## 상태

- 작성일: 2026-07-10
- 상태: 사용자 화면 설계 승인 완료
- 기준 화면: `관리자-와이어프레임.png`
- 대상: `/manager` 아래의 데스크톱 관리자 화면

## 배경

현재 관리자 기능은 자산 홈, 계약, 청구, 비용, 민원·하자, 소통, 퇴실, 업체, 리포트, AI 에이전트로 나뉘어 있지만 각 도메인이 서로 다른 `ManagerShell` 내비게이션을 사용한다. 사용자는 한 화면에서 전체 관리자 기능을 파악하거나 다른 업무로 이동하기 어렵다. `/manager/home/00`도 네 개 탭과 단순 목록 중심이라 와이어프레임의 통합 업무 화면 및 우측 AI 비서 구조를 반영하지 못한다.

이번 개편은 관리자 데스크톱 전체를 하나의 워크스페이스로 보이게 만들고, 통합 대시보드에서 실제 운영 데이터와 AI 진입점을 함께 제공한다.

## 목표

1. 모든 관리자 데스크톱 기능을 공통 왼쪽 사이드바에서 선택할 수 있게 한다.
2. 현재 경로의 상위 도메인과 세부 화면을 명확히 표시한다.
3. `/manager/home/00`을 KPI, 오늘의 업무, 매물·계약·민원 현황이 함께 보이는 통합 대시보드로 바꾼다.
4. 통합 대시보드에는 고정 우측 AI 비서 패널을 제공한다.
5. 다른 관리자 데스크톱 화면에는 작은 AI 버튼을 제공하고, 누르면 같은 비서 UI를 대화상자로 연다.
6. 기존 URL, 서버 데이터 원천, 도메인별 세부 화면과 모바일 전용 표면을 보존한다.
7. API 실패나 미구축 집계를 거짓 `0`으로 표현하지 않는다.
8. `/manager` 기본 진입은 매물 등록이 아니라 `/manager/home/00` 통합 대시보드로 연결한다.

## 비목표

- 새로운 관리자 API 또는 데이터베이스 테이블 추가
- AI가 계약 확정, 결제, 독촉, 발송을 자동 완료하는 기능
- `/manager/vox/*` 및 `/manager/ticket/call/*`의 `PhoneFrame` 재설계
- 로그인 화면 또는 임차인·업체 화면 개편
- 동적 상세 ID가 필요한 화면을 사이드바의 고정 링크로 만드는 작업
- 현재 기능과 무관한 전역 CSS 또는 레거시 화면의 전면 리팩터링

## 검토한 접근법

### 1. 앱 전용 `ManagerAppShell` 도입 — 선택

웹 앱에 경로를 아는 `ManagerAppShell`을 만들고, 공유 UI의 `ManagerShell`은 레이아웃 슬롯만 제공한다. 관리자 경로와 메뉴 구성은 웹 앱에서 한 번만 정의한다. 기존 도메인 래퍼는 로컬 메뉴를 `ManagerAppShell`에 넘긴다.

장점은 공유 UI와 제품 라우팅의 결합을 피하고, 전체 관리자 화면에서 동일한 크롬과 현재 경로 판정을 사용할 수 있다는 점이다. 변경 대상은 여러 도메인 래퍼에 걸치지만, 명시적으로 마이그레이션하므로 로그인·모바일 표면이 잘못 감싸지는 위험이 가장 낮다.

### 2. 공유 `ManagerShell`에 관리자 경로 하드코딩 — 제외

호출부 변경은 줄지만 `@roomlog/ui`가 Next.js 관리자 URL을 알아야 한다. 다른 제품 표면에서 재사용하기 어렵고 라우트 변경 시 UI 패키지가 함께 바뀌므로 제외한다.

### 3. 최상위 `/manager/layout.tsx`에서 전부 감싸기 — 제외

하나의 레이아웃만 수정할 수 있지만 로그인, 모바일 보이스, 모바일 통화까지 같은 셸에 들어온다. 기존 도메인 셸과도 중첩되므로 경로 예외가 늘어나고 구조가 불명확해져 제외한다.

## 정보 구조와 메뉴

사이드바는 상위 기능을 항상 보여주고, 현재 상위 기능의 세부 화면만 펼친다. 세부 메뉴에는 ID 없이 직접 진입해도 의미가 있는 목록·허브·신규 생성 화면만 포함한다. 오류 화면(`e0`)과 엔티티 ID가 필요한 상세 화면은 해당 목록이나 문맥 링크를 통해 들어가므로 고정 메뉴에서 제외한다.

| 그룹 | 상위 메뉴 | 시작 경로 | 펼쳐지는 세부 메뉴 |
| --- | --- | --- | --- |
| 홈 | 통합 대시보드 | `/manager/home/00` | 통합 대시보드, 오늘의 업무, 임대 현황(데모), 건물 관리(데모), 건물·호실 등록(데모) |
| 임대 운영 | 매물 관리 | `/sell` | 기존 매물 관리로 이동한다는 외부 표식 |
| 임대 운영 | 계약 관리 | `/manager/contract/00` | 검토 대시보드, 계약서 등록 |
| 임대 운영 | 청구·수납 | `/manager/billing` | 대시보드, 수금, 입금 매칭, 연체 |
| 임대 운영 | 비용 원장 | `/manager/cost/00` | 원장, 영수증 첨부, 공개 설정 |
| 운영 지원 | 민원·하자 | `/manager/ticket/dash/00` | 대시보드 |
| 운영 지원 | 소통·공지 | `/manager/messaging/00` | 허브, 공지 작성 |
| 운영 지원 | 퇴실·정산 | `/manager/moveout/00` | 대시보드 |
| 운영 지원 | 업체 관리 | `/manager/vendor-mgmt/00` | 주소록, 업체 추가 |
| 인사이트 | 운영 리포트 | `/manager/report/00` | 허브, 새 리포트 생성, 빠른 조회 |
| AI | AI 비서 | `/manager/agent/realtime` | 실시간 AI 운영 에이전트 |
| 계정 | 설정 | `/manager/home/06` | 설정 |

현재 경로와 일치하는 링크에는 `aria-current="page"`를 설정한다. 하위 경로에서는 상위 메뉴도 활성 상태로 표시하고 그 도메인의 세부 메뉴를 기본으로 펼친다. 매물 관리는 관리자 URL 밖으로 이동하므로 외부 이동 표식을 제공한다.

`/manager`는 인증된 임대인을 `/manager/home/00`으로 보낸다. 기존 `/sell` 매물 관리 기능은 사이드바의 `매물 관리` 항목으로 계속 접근한다.

다음 화면은 유효한 엔티티를 고른 뒤에만 문맥 링크로 표시하며 ID를 query에 보존한다.

- 계약: OCR 검토, 호실·타임라인, 임차인 초대, 보관·삭제
- 비용: OCR 분석, 비용 상세
- 민원·하자: 상세·검토, 사진 비교, 답변 초안, 업체·수리, 결제 승인
- 소통: 초안 검토, 발송 결과, 스레드 상세
- 퇴실·정산: 기록, 정산안, 이의
- 업체: 상세, 성과, 기존 업체 편집
- 리포트: 상세, 임대인 보고, 챗봇

현재 `/manager/home/02`부터 `/manager/home/05`에는 하드코딩된 프로토타입 데이터와 읽기 전용 동작이 있다. 이번 UI 범위에서는 메뉴와 화면 머리말에 `데모` 상태를 명시하고 실제 집계·저장이 완료된 것처럼 표현하지 않는다. 이 화면들의 실데이터화와 mutation은 별도 기능 범위다.

## 컴포넌트 구조

### 공유 UI 레이어

`packages/ui/src/components/ManagerShell.tsx`는 제품 경로를 모르는 데스크톱 레이아웃 컴포넌트로 유지한다. 다음 슬롯을 제공한다.

- 브랜드 및 전역 내비게이션을 위한 왼쪽 `nav`
- 화면 제목, 경로 문맥, 헤더 액션
- 도메인 내부 세부 이동을 위한 `subnav`
- 대시보드 전용 AI 영역을 위한 선택적 `rightRail`
- 본문 `children`

레이아웃은 왼쪽 사이드바, 가운데 작업 영역, 선택적 오른쪽 레일의 세 열로 구성한다. 공유 컴포넌트는 라우트, 사용자 데이터, AI 동작을 알지 않는다.

### 웹 앱 레이어

웹 앱에는 다음 책임을 분리한다.

- `manager-navigation`: 전체 메뉴 정의, 경로 일치, 활성 상위 메뉴 및 세부 메뉴 계산
- `ManagerSidebar`: 브랜드, 그룹 메뉴, 활성 상태, 사용자 영역 렌더링
- `ManagerAppShell`: 공유 `ManagerShell`에 사이드바와 AI 표면을 조합
- `ManagerAssistantPanel`: 대시보드 고정 패널과 대화상자에서 재사용하는 비서 내용
- `ManagerAssistantLauncher`: 대시보드 외 화면의 우측 하단 AI 버튼과 네이티브 대화상자
- `ManagerDashboard`: 통합 대시보드의 KPI, 업무 큐, 현황 섹션

도메인별 기존 하위 메뉴는 삭제하지 않고 `subnav`로 이동한다. billing, contract, cost, messaging, moveout, report, ticket dashboard, vendor management, agent 및 home 데스크톱 화면이 `ManagerAppShell`을 사용한다. `PhoneFrame`을 사용하는 call 및 vox 화면은 제외한다.

AI 전체 화면인 `/manager/agent/realtime`에서는 우측 하단 AI 실행기를 중복 표시하지 않는다.

## 화면 설계

### 공통 셸

- 왼쪽 사이드바는 넓은 화면에서 고정 폭으로 유지하고 문서가 길 때 자체 스크롤한다.
- 브랜드 아래에 그룹형 메뉴를 배치한다.
- 하단에는 사용자 역할, 설정, 홈 복귀 동선을 둔다.
- 가운데 상단 헤더에는 현재 화면명, 관리 문맥, 모바일 메뉴 버튼을 둔다.
- 도메인 세부 메뉴는 헤더 아래의 컴팩트한 서브 내비게이션으로 표시한다.
- 본문은 화면 폭을 유연하게 사용하며, 표가 전체 페이지 폭을 밀어내지 않도록 내부 스크롤 영역을 사용한다.

### 통합 대시보드

대시보드는 네 개 탭을 제거하고 다음 순서의 한 화면으로 통합한다.

1. 인사말과 관리 문맥
2. 주요 KPI 카드: 관리 중인 집, 미계약 매물, 계약 중인 집, 처리할 민원·하자, 수납 대기·연체
3. 오늘의 업무 큐: 긴급도와 원천이 명확한 항목 및 실제 처리 화면 링크
4. 운영 현황: 최근 매물, 계약, 민원·하자 목록을 한 화면에서 비교
5. 빈 상태에는 설명과 한 개의 다음 행동

오른쪽에는 항상 `ManagerAssistantPanel`을 표시한다. 패널은 인사말, 오늘의 브리핑, 추천 업무, 질문 입력, 실시간 AI 에이전트 진입을 제공한다.

현재 계약 목록의 기능은 보존한다. 계약 행을 선택하면 기존 계약 요약, 청구·민원 카드와 해당 계약 문의 스레드에 잠긴 세입자 채팅을 볼 수 있어야 한다. 개편 과정에서 이 상세 표면을 별도 컴포넌트로 추출할 수 있지만, 동작과 접근 경로를 제거하지 않는다.

### 다른 관리자 화면의 AI

대시보드가 아닌 관리자 데스크톱 화면에는 우측 하단 AI 버튼을 표시한다. 버튼은 네이티브 `<dialog>` 기반 대화상자를 열며, 닫기 버튼, Escape 닫기, 포커스 복귀를 제공한다. 대화상자 안에는 대시보드와 같은 비서 컴포넌트를 사용하되 현재 화면 이름을 문맥으로 제공한다.

질문을 제출하면 `/manager/agent/realtime?prompt=...`으로 이동하여 기존 실시간 에이전트 입력에 내용을 미리 채운다. 프롬프트는 앞뒤 공백을 제거하고 최대 1,000자로 제한하며 최초 렌더에서 한 번만 입력값으로 사용한다. 질문을 자동 실행하지 않으며 사용자가 전체 AI 화면에서 확인하고 전송한다.

## 데이터 흐름

통합 대시보드는 현재 사용 중인 서버 원천을 조합한다.

- 매물: `/trade/listings`에서 로그인한 소유자의 미계약 매물
- 계약: `/trade/contracts`에서 로그인한 임대인의 수락된 계약
- 민원·하자: `listManagerTickets()`의 미완료 티켓
- 청구: `/manager/bills/dashboard`의 실제 관리자 집계
- 관리 중인 집: 인증 세션의 `managedRooms`

각 원천은 독립된 성공·실패 상태를 유지한다. 한 원천이 실패해도 다른 섹션은 렌더링한다. 새 백엔드 DTO를 만들지 않고 기존 타입과 매핑을 사용한다. 구현 중 새로운 웹↔API 계약이 필요해지는 경우에만 `packages/types`에 도메인 접두어 타입을 먼저 추가한다.

현재 `listManagerTickets()`는 API 실패를 빈 배열로 바꾸므로 실데이터 `0건`과 장애를 구분할 수 없다. 대시보드용 티켓 조회에는 `live`, `demo`, `error`를 구분하는 결과 계약을 추가한다. 정상적인 빈 응답은 `live`와 빈 배열이고, API 미기동으로 데모 폴백을 사용하면 `demo`와 출처 배지를 표시하며, 인증·권한 또는 복구 불가능한 실패는 `error`와 재시도·원천 화면 이동을 표시한다. 기존 상세 화면의 데모 폴백 동작은 유지한다.

대시보드 KPI는 다음 규칙을 따른다.

- 목록이 정상 조회되면 실제 길이를 표시한다.
- 청구 집계가 실패하면 `0` 대신 `확인 필요`를 표시한다.
- 미구축 집계는 `집계 전` 또는 `연결 필요`로 표시한다.
- 카드의 클릭 대상은 해당 원천 화면과 일치해야 한다.

## AI 안전성과 false-agency 방지

- 패널은 AI가 작업을 완료했다고 주장하지 않는다.
- 계약 확정, 결제 승인, 독촉, 공지 발송은 각각의 원천 화면에서 사용자가 확인한다.
- AI 제안은 `추천`, 생성 문장은 `초안`으로 표시한다.
- 실제 AI 연결이 불가능하면 이유와 전체 AI 화면 이동만 제공하고 가짜 답변을 생성하지 않는다.
- 티켓과 수리, 초안과 발송, 집계 실패와 실제 `0`을 구분한다.

## 오류·로딩·빈 상태

- 서버 렌더 단계에서 원천별 오류를 포착하고 해당 섹션 가까이에 오류 문구와 실제 원천 화면 링크를 표시한다.
- 전체 페이지를 하나의 API 성공 여부에 종속시키지 않는다.
- 로딩 UI는 최종 카드 및 목록 구조와 같은 형태의 스켈레톤을 사용한다.
- 빈 상태는 제목, 이유, 명확한 다음 행동 하나만 제공한다.
- 사이드바 라우팅 오류는 존재하지 않는 링크를 숨기는 방식이 아니라 테스트에서 차단한다.

## 반응형 동작

- 큰 데스크톱: 왼쪽 사이드바 + 가운데 본문 + 대시보드 오른쪽 AI 레일
- 노트북 폭: 사이드바는 유지하고 AI 레일 폭과 본문 간격을 줄인다.
- 좁은 폭: 사이드바를 메뉴 버튼으로 여는 내비게이션 대화상자로 전환하고, 대시보드 AI 레일은 본문 아래로 이동한다.
- 고정 요소는 `safe-area-inset-*`를 반영하고 높이는 동적 뷰포트 단위를 사용한다.
- 모바일 전용 `PhoneFrame` 화면은 이 반응형 셸을 사용하지 않는다.

## 접근성

- 아이콘 전용 버튼에는 `aria-label`을 제공한다.
- 현재 메뉴에는 `aria-current="page"`를 제공한다.
- 내비게이션과 AI 대화상자는 키보드로 열고 닫을 수 있어야 한다.
- 대화상자는 이름, 닫기 동작, 초기 포커스와 닫은 뒤 포커스 복귀를 제공한다.
- 모든 상호작용 요소는 기존 터치 타깃 토큰 이상을 유지한다.
- 제목은 논리적인 heading 순서를 사용하고 KPI 수치는 tabular numeral 스타일을 사용한다.
- 별도 애니메이션은 추가하지 않는다.

## 시각 규칙

- `packages/ui/src/tokens.css`의 CSS 변수만 사용한다.
- 신규 TSX 및 CSS에 raw hex, 임의 RGB/RGBA 색상, 그라데이션을 추가하지 않는다.
- 한 화면의 강조색은 기존 `--primary` 하나로 제한한다.
- 공유 `Card`, `Badge`, `Button`, `Input`을 우선 재사용한다.
- 상태 표현에 필요한 의미 토큰이 없다면 먼저 `tokens.css`에 의미 기반 토큰을 추가한다.
- Lucide 아이콘은 웹 앱 레이어에서 사용하고 `@roomlog/ui`에 제품 의존성을 추가하지 않는다.

## 테스트 전략

구현은 테스트 우선으로 진행한다.

1. `manager-navigation` 순수 함수 테스트를 먼저 작성한다.
   - 모든 상위 관리자 도메인 시작 경로가 메뉴에 존재한다.
   - 하위 경로에서 올바른 상위 메뉴와 세부 메뉴가 활성화된다.
   - ID 없이 진입 가능한 정확한 세부 경로만 고정 메뉴에 존재한다.
   - 오류 및 동적 상세 경로는 고정 메뉴에 노출되지 않고 문맥 링크가 ID를 보존한다.
2. 셸 소스 계약 테스트를 작성한다.
   - 전역 사이드바, `aria-current`, 대시보드 오른쪽 레일, 비대시보드 AI 실행기 연결을 확인한다.
   - `PhoneFrame` 화면은 공통 데스크톱 셸에 포함되지 않음을 확인한다.
   - `/manager`가 통합 대시보드로 이동하고 AI 전체 화면에는 AI 실행기가 중복되지 않음을 확인한다.
3. 대시보드 데이터 매핑 테스트를 작성한다.
   - 정상 조회는 실제 개수를 표시한다.
   - 청구 실패를 `0`으로 바꾸지 않는다.
   - 티켓 실데이터 빈 목록, 명시적 데모 폴백, 오류 상태를 서로 구분한다.
   - API 미기동 상태에서 데모 출처가 표시된 채 화면이 렌더링된다.
   - 빈 상태가 다음 행동을 제공한다.
   - 계약 행의 계약 요약 및 해당 문의 스레드 세입자 채팅 동작이 유지된다.
4. AI 프롬프트 전달 테스트를 작성한다.
   - 질문은 공백 제거 및 1,000자 제한 후 인코딩된 query parameter로 전달된다.
   - 프롬프트는 초기 입력에 한 번만 반영되고 자동 실행되지 않는다.
5. 검증 명령을 실행한다.
   - 집중 테스트의 RED와 GREEN 확인
   - `pnpm test:web`
   - `pnpm --filter @roomlog/ui build`
   - `pnpm --filter web build`
   - `bash scripts/verify.sh`
6. Docker 표준 환경에서 브라우저 검증을 수행한다.
   - 1440px 데스크톱
   - 1024px 노트북
   - 좁은 폭의 접힌 메뉴
   - 주요 사이드바 이동, 현재 메뉴 표시, AI 패널 및 대화상자, 빈 상태와 실패 상태

## 완료 기준

- 모든 관리자 데스크톱 상위 기능을 왼쪽 사이드바에서 이동할 수 있다.
- `/manager` 진입 시 통합 대시보드로 이동한다.
- 현재 상위 기능과 세부 화면이 명확하게 표시된다.
- `/manager/home/00`에서 KPI, 오늘의 업무, 매물·계약·민원 현황과 우측 AI 비서가 함께 보인다.
- 다른 관리자 데스크톱 화면에서 AI 버튼으로 비서 대화상자를 열 수 있다.
- 계약 행에서 기존 계약 요약과 해당 문의 스레드 세입자 채팅을 계속 사용할 수 있다.
- 질문을 기존 실시간 AI 화면으로 전달하되 자동 실행하지 않는다.
- 모바일 통화·보이스 화면은 기존 `PhoneFrame`을 유지한다.
- API 실패, 미구축 집계, 실제 `0`이 구분된다.
- API 미기동 데모 폴백은 출처를 명확히 표시한다.
- ID가 필요한 상세 화면은 레코드를 선택한 문맥에서만 링크되고 해당 ID를 유지한다.
- 하드코딩된 홈 프로토타입 화면은 `데모`로 명확히 표시된다.
- 신규 스타일은 디자인 토큰만 사용한다.
- 키보드 포커스, 대화상자 닫기, `aria-current`가 동작한다.
- 웹 테스트, UI·web 빌드, 전체 검증 스크립트가 통과한다.
