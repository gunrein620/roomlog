# 거래 계약–청구 계약 연결 설계

## 목표

매물 문의에서 집주인이 제안한 계약을 세입자가 수락하면, 정확한 임대인·건물·호실에 청구용 계약 초안을 자동으로 연결한다. 관리자가 누락된 청구 정보를 확인하고 계약을 확정하면 해당 호실이 청구서 생성 화면에 즉시 나타나야 한다.

## 현재 문제

현재 거래 계약과 계약·청구 도메인은 서로 다른 레코드를 사용한다.

- 거래 계약 수락은 매물 상태와 `tenantRooms` 관계만 갱신한다.
- 계약 관리 화면은 Roomlog 계약 레코드만 조회한다.
- 청구서 생성은 Roomlog 계약 중 `lifecycle=active`, `review=confirmed`, `valueSource=confirmed`인 계약만 허용한다.
- 관리자 검토 확정은 현재 `review`와 `valueSource`만 변경하고 `lifecycle`을 `active`로 전환하지 않는다.
- 결과적으로 정상적인 거래 계약 수락 흐름만으로는 청구 가능한 계약이 만들어지지 않는다.

운영 화면에서도 세입자 측에는 체결 계약이 보이지만 관리자 계약 대시보드에는 계약이 0건이고, 청구서 생성 화면에는 생성 가능한 계약이 0호실로 표시된다.

## 검토한 접근법

### 1. 계약 초안 자동 생성 후 관리자 확정 — 채택

거래 계약 수락 시 청구용 계약 초안을 자동 생성한다. 거래 계약에 있는 월세와 보증금은 미리 채우고, 관리비·납부일·계약 기간은 관리자가 확인한다. 명시적으로 확정한 뒤에만 청구할 수 있다.

장점은 잘못된 기본값으로 청구하지 않으면서 기존 계약 검토 원칙을 유지한다는 점이다. 단점은 관리자가 한 번 확인해야 한다는 점이다.

### 2. 수락 즉시 활성 계약 생성

누락된 관리비·납부일·기간을 기본값으로 채워 즉시 청구 가능하게 한다. 빠르지만 근거 없는 금액과 날짜가 실제 청구에 사용될 수 있어 채택하지 않는다.

### 3. 수동 계약 등록 유지

거래 계약과 청구 계약을 계속 분리하고 관리자에게 계약서를 다시 등록하도록 안내한다. 구현은 단순하지만 이미 수락한 계약을 다시 입력해야 하며 현재 오류를 해결하지 못하므로 채택하지 않는다.

## 데이터 연결

### 신규 수락 계약

1. 세입자가 거래 계약을 수락한다.
2. 기존 `assignTenantRoomFromContract`가 임대인의 정확한 건물·호실을 찾거나 생성하고 `roomId`를 반환한다.
3. 반환된 동일 `roomId`를 사용해 청구용 계약 초안을 생성하거나 기존 초안을 재사용한다.
4. 계약 초안은 거래 계약 당사자의 `landlordId`, `tenantId`와 해당 `roomId`에 연결한다.
5. 월세와 보증금은 거래 계약 스냅샷에서 가져온다. 관리비·납부일·계약 시작일·종료일은 임의로 만들지 않는다.

초안 ID는 `ct_trade_<tradeContractId>` 형식의 결정적 ID를 사용한다. 같은 수락 요청이 재시도되거나 서버가 다시 기동돼도 같은 계약을 찾아 재사용하므로 중복 계약이 생기지 않는다. 별도의 데이터베이스 컬럼이나 마이그레이션은 추가하지 않는다.

같은 세입자가 이미 수락된 동일 계약에 다시 `accept=true`로 요청하면 오류로 끝내지 않고 기존 거래 계약을 반환한 뒤 청구용 계약 보정을 다시 실행한다. 거절 또는 다른 사용자의 재요청은 기존 권한·상태 검사를 유지한다.

### 기존 수락 계약 보정

API 기동 시 거래 저장소의 `accepted` 계약을 읽고 동일한 보정 함수를 실행한다. 이 과정은 결정적 계약 ID를 사용해 멱등성을 보장한다. 조회 API에 숨은 쓰기 동작은 추가하지 않는다.

기존 수락 계약의 임대인에게만 초안이 보인다. 같은 로그인 계정이 다른 집의 세입자이면서 별도 건물의 집주인이더라도, 세입자로 거주하는 집의 계약을 자신의 관리자 화면에서 청구할 수는 없다.

## 계약 초안 상태

자동 생성 직후 상태는 다음과 같다.

- `lifecycle`: `analyzing`
- `review`: `pending`
- `valueSource`: `unverified`
- 월세: 거래 계약값
- 보증금: 추출·확인 항목에 거래 계약값으로 기록
- 관리비, 납부일, 시작일, 종료일: 미확인

실제 계약서 파일이 업로드된 것처럼 보이게 하지 않는다. 문서가 없는 거래 계약 초안은 관리자 화면에서 출처를 `거래 계약`으로 표시하고 원본 파일이 없음을 명시한다.

## 관리자 확인 흐름

관리자 계약 대시보드에서 거래 계약 초안은 `확인 필요` 상태로 표시된다. 관리자는 해당 계약을 열어 다음 값을 확인한다.

- 계약 시작일
- 계약 종료일
- 월세
- 관리비 — 0원 허용
- 납부일

선택한 계약 ID는 검토·수동값 화면 사이에서 쿼리스트링으로 유지한다. 다른 계약이 대신 열리거나 첫 계약으로 되돌아가면 안 된다.

검토 확정 시 서버는 다음을 검증한다.

- 시작일과 종료일이 모두 존재한다.
- 종료일이 시작일보다 빠르지 않다.
- 월세와 관리비가 모두 입력됐고 0 이상의 정수다.
- 월세 또는 관리비가 있는 계약은 납부일이 1~31 사이여야 한다.
- 확인 필요 항목을 대조했다는 관리자의 명시적 체크가 있다.

검증을 통과하면 한 번의 확정 동작으로 다음 상태를 저장한다.

- `review=confirmed`
- `valueSource=confirmed`
- `lifecycle=active`

종료일이 이미 지난 계약은 활성화하지 않고 확정 요청을 거절한다.

## 청구서 생성

청구서 생성 옵션은 기존 관리자·건물·호실 권한 검사를 유지한다. 다음 조건을 모두 만족할 때만 호실을 생성 대상으로 표시한다.

- 현재 관리자가 관리하는 호실이다.
- 계약이 `active/confirmed/confirmed` 상태다.
- 월세와 관리비 합계가 0보다 크다.
- 납부일이 등록돼 있다.
- 같은 청구월의 취소되지 않은 청구서가 없다.

계약 확정은 청구서를 자동으로 만들거나 발송하지 않는다. 관리자가 청구서 생성 화면에서 호실을 선택하고 금액·납부기한·수납 계좌를 확인한 뒤 초안을 저장한다.

## 오류 처리와 안전장치

- 같은 거래 계약을 여러 번 처리해도 청구용 계약은 하나만 유지한다.
- 다른 임대인의 호실이나 계약은 관리자 계약·청구 화면에 노출하지 않는다.
- 같은 호실에 동일 임대인·임차인의 활성 계약이 있으면 기존 계약을 재사용한다.
- 같은 호실에 다른 임차인의 활성 계약이 있으면 대화형 수락 요청은 충돌 오류를 반환한다. 기동 시 보정은 해당 건을 기록하고 자동 덮어쓰기하지 않는다.
- 필수값이 빠진 계약은 확정하지 않고 누락 항목을 구체적으로 안내한다.
- 계약 초안 생성 실패 후 수락 요청이 재시도되면 멱등 보정으로 누락된 초안만 생성한다.
- 기존 청구서 중복 검사와 취소 청구서 예외는 유지한다.

## 테스트

구현 전 다음 실패 테스트를 먼저 추가한다.

1. 거래 계약 수락이 반환한 정확한 `roomId`에 청구용 계약 초안을 생성한다.
2. 월세와 보증금은 거래 계약값으로 채우고 관리비·납부일·기간은 만들지 않는다.
3. 같은 거래 계약을 재처리해도 계약 초안이 중복 생성되지 않는다.
4. 기존 `accepted` 거래 계약을 기동 시 보정한다.
5. 초안 계약은 청구서 생성 옵션에 포함되지 않는다.
6. 관리자 필수값 확인과 명시적 대조 확인 후 계약이 `active/confirmed/confirmed`로 전환된다.
7. 확정된 계약은 청구서 생성 옵션에 포함된다.
8. 금액 합계가 0이거나 납부일이 없는 계약은 청구서 생성 옵션에서 제외된다.
9. 다른 임대인의 계약은 관리자 계약·청구 범위에 포함되지 않는다.
10. 현재 선택한 계약 ID가 수동값 저장과 검토 확정 화면 사이에서 유지된다.

대상 테스트 통과 후 API 전체 테스트, web 전체 테스트, types·UI 타입체크, web·API 빌드를 실행한다. Docker와 운영 데이터 변경은 사용자가 별도로 요청할 때만 수행한다.

## 범위 밖

- 계약 확정과 동시에 청구서를 자동 생성하거나 발송하는 기능
- 임의의 관리비·납부일·계약 기간 기본값 생성
- 다른 임대인 소유 호실을 현재 관리자에게 이전하는 기능
- 결제 위젯 및 결제 승인 흐름 변경
- 기존 청구·수납 대시보드 디자인 변경

## 완료 기준

- 신규 및 기존 수락 거래 계약이 해당 임대인의 계약 대시보드에 확인 필요 초안으로 나타난다.
- 관리자가 누락값을 입력하고 명시적으로 확정할 수 있다.
- 확정 직후 같은 건물·호실이 청구서 생성 대상에 표시된다.
- 잘못된 관리자 범위, 중복 계약, 누락값, 과거 종료일은 서버에서 차단된다.
- 실제 계약서 파일이나 청구서가 자동으로 만들어졌다고 오인시키는 표시가 없다.
- 회귀 테스트와 빌드가 통과한다.
