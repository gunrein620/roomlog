# 민원·하자 채팅 사진 및 업체 배정 안내 설계

## 목표

- 세입자와 관리자가 같은 민원·하자 채팅에서 글과 사진을 양방향으로 주고받는다.
- 관리자가 업체를 배정하거나 재배정하면 업체명과 전화번호, 일정 협의 안내가 해당 채팅에 자동으로 남는다.
- 민원·하자 관리 목록의 단일 작업 메뉴를 `상세` 직접 이동 버튼으로 단순화한다.

## 범위

### 포함

- 세입자 민원·하자 이력 상세 채팅의 사진 선택, 미리보기, 삭제, 전송 및 렌더링
- 관리자 민원 대시보드 채팅의 사진 선택, 미리보기, 삭제 및 전송
- 관리자 답변 계약의 `attachmentUrls` 지원
- 업체 배정 성공 시 서버에서 자동 채팅 메시지 생성
- 목록의 `...` 메뉴를 기존 상세 처리 화면으로 이동하는 `상세` 버튼으로 교체
- 관련 공유 타입, API, 실시간 이벤트, 테스트와 스타일 수정

### 제외

- Gara 견적 및 크레딧 결제 흐름 변경
- 별도 채팅방 또는 업체 전용 채팅 신설
- 동영상·문서 첨부
- 채팅 이미지 편집, 압축 또는 AI 분석
- 기존 민원 접수 사진 업로드 구조 교체

## 채팅 사진 설계

기존 인증형 `POST /attachments` 이미지 업로드 API를 재사용한다. 브라우저는 각 사진을 먼저 업로드하고 반환된 `fileUrl` 배열을 기존 메시지 요청의 `attachmentUrls`로 전달한다. 새로운 바이너리 저장소나 multipart 메시지 API는 만들지 않는다.

- 한 메시지당 최대 5장
- 이미지 한 장당 최대 10MB
- 글만, 사진만, 글과 사진 동시 전송 허용
- 선택 직후 로컬 미리보기 표시 및 개별 제거 지원
- 모든 사진 업로드가 성공한 뒤 메시지를 한 번 저장
- 업로드나 메시지 저장이 실패하면 글과 선택 사진을 유지하고 오류를 표시
- 전송 성공 후 입력값과 미리보기 URL을 정리
- 닫기와 컴포넌트 해제 시 로컬 object URL 해제

세입자 메시지 API는 이미 `attachmentUrls`와 사진-only 메시지를 지원한다. 관리자 `ManagerTicketReplyInput`과 서버 저장 로직을 같은 계약으로 확장하고, 본문과 첨부가 모두 비었을 때만 거부한다. 사진-only 메시지는 `사진을 첨부했습니다.`라는 기본 본문으로 저장한다.

관리자 채팅은 기존 첨부 썸네일 렌더링을 유지한다. 세입자 채팅에도 같은 방식으로 첨부 썸네일과 원본 열기, 이미지 로드 실패 시 파일 링크 폴백을 추가한다. 저장된 메시지는 현재 `roomlog:ticket-message` 이벤트를 그대로 사용해 상대 화면에 반영한다.

## 업체 배정 자동 안내 설계

빠른 배정과 상세 화면 배정이 모두 사용하는 서버의 업체 배정 성공 지점에서 메시지를 생성한다. 클라이언트별 후처리로 만들지 않아 어떤 관리자 화면에서 배정해도 동일하게 동작한다.

기본 문구는 다음과 같다.

> `배정 업체: {업체명}. 연락처는 {전화번호}입니다. 해당 업체에 전화하여 방문 일정을 상의해 주세요.`

- 전화번호가 없으면 연락처 문장을 생략하고 일정 협의 안내는 유지한다.
- 업체명이 없으면 `수리 업체`로 표시한다.
- 최초 배정과 다른 업체로의 재배정은 각각 새 안내를 남긴다.
- 동일 업체에 대한 멱등 재호출은 새 안내를 만들지 않는다.
- 업체 배정과 자동 메시지는 같은 서버 트랜잭션 안에서 저장해 일부 성공 상태를 만들지 않는다.
- 배정 응답 이후 `roomlog:ticket-message` 이벤트를 발행해 열린 세입자·관리자 채팅에 즉시 표시한다.

## 목록 상세 버튼 설계

`TicketActionMenu`의 `...` 트리거와 포털 메뉴를 제거한다. 작업 열에는 `상세` 링크를 직접 렌더링하며 기존 목적지인 `/manager/ticket/dash/01?id={ticketId}`를 사용한다. 메뉴 위치 계산 모듈이 다른 곳에서 사용되지 않으면 함께 삭제한다.

## 오류 및 권한

- 기존 인증 쿠키와 `/attachments` 권한 검사를 그대로 사용한다.
- 세입자는 자신이 접수한 민원에만 메시지를 보낼 수 있고, 관리자는 자신이 관리하는 호실의 티켓에만 답변할 수 있다.
- 서버는 이미지 MIME과 10MB 제한을 최종 강제한다.
- 클라이언트는 5장 제한과 이미지 파일 여부를 전송 전에 안내한다.
- 업로드된 파일 중 메시지 저장 전에 실패한 고아 파일 정리는 이번 범위에 포함하지 않는다. 기존 첨부 업로드도 같은 수명주기를 사용한다.

## 테스트 및 완료 조건

- 세입자와 관리자 모두 사진-only 및 글+사진 메시지를 보낼 수 있다.
- 양쪽 화면에서 상대가 보낸 사진이 썸네일로 표시된다.
- 5장 초과, 이미지가 아닌 파일, 10MB 초과 파일은 명확한 오류로 차단된다.
- 업로드 또는 메시지 저장 실패 시 작성 중인 내용이 유지된다.
- 업체 배정 시 업체명·전화번호·일정 협의 문구가 채팅에 한 번 저장된다.
- 재배정은 새 업체 안내를 생성하고 동일 업체 멱등 호출은 중복 안내를 만들지 않는다.
- 목록의 `상세` 버튼이 기존 상세 처리 화면으로 직접 이동하며 `...` 메뉴가 남지 않는다.
- 관련 단위 테스트, `@roomlog/types` 타입검사, web/api 빌드가 통과한다.

## 배포 및 Git 제약

- 작업 브랜치는 `codex/ticket-chat-photo-vendor-notice`를 사용한다.
- 기능 구현 커밋은 로컬 브랜치에만 만들고 사용자의 별도 요청 전에는 추가 push하지 않는다.
