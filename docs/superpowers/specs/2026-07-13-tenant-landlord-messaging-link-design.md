# 세입자 임대인 문의와 관리자 소통 연동 설계

## 목표

세입자 `마이 > 입주 정보`의 `임대인에게 문의하기`를 관리자 `소통`과 동일한 룸로그 메시징 스레드에 연결한다. 세입자가 보낸 첫 메시지와 이후 답변은 관리자 소통 허브에 표시되고, 관리자의 답변도 세입자 대화 상세에 같은 순서로 표시되어야 한다.

## 확인된 문제

- 현재 `TenantMyPage`의 문의 버튼은 계약 응답의 거래 스레드 ID를 사용해 `TradeChatCenter`를 연다.
- 관리자 소통 허브는 `MessagingThread`와 `/manager/messaging/*` API를 사용한다.
- 거래 채팅과 룸로그 메시징은 식별자, 저장소, API가 서로 다르므로 같은 계약 당사자여도 메시지가 공유되지 않는다.
- 룸로그 메시징 도메인은 이미 `TenantRoom`과 `Room.landlordId`를 기준으로 세입자-관리자 권한을 검증하고 양방향 메시지를 제공한다.

## 선택한 접근 방식

`임대인에게 문의하기`를 거래 채팅 모달에서 분리하고 룸로그 메시징의 일반 문의 진입점으로 바꾼다.

- 계약 및 입주 연결이 있는 세입자가 버튼을 누르면 본인의 일반 문의 스레드를 조회한다.
- `context=general`, `contextRef 없음`, 동일 `tenantId + roomId`인 스레드가 있으면 그 상세 화면으로 바로 이동한다.
- 기존 스레드가 없으면 첫 메시지 작성 패널을 열고, 전송 시 일반 문의 스레드를 생성한 뒤 상세 화면으로 이동한다.
- 관리자 소통에서 먼저 만든 일반 문의도 같은 조건으로 재사용한다.
- 중복 생성 경쟁을 막기 위해 서버가 일반 문의 생성 시 기존 스레드를 다시 확인한다.

고려했지만 선택하지 않은 방식:

- 기존 `TradeChatCenter`를 관리자 메시징 API에 맞춰 개조: 거래 전 문의 UI와 입주 후 관리 소통의 책임이 섞이고 메시징 화면이 이중화된다.
- 버튼 클릭만으로 빈 스레드 생성: 관리자 목록에 실제 메시지가 없는 대화가 생기고 `lastMessage` 계약을 훼손한다.
- 세입자 메시지 목록으로만 이동: 최초 스레드가 없는 사용자는 대화를 시작할 수 없어 요구사항을 충족하지 못한다.

## 공유 계약과 API

세입자가 대화 진입 상태를 확인할 수 있는 응답 계약을 추가한다.

- `TenantLandlordConversation`
  - `threadId?: string`
  - `buildingName`
  - `unitId`
  - `landlordName`

세입자 메시징 API:

- `GET /tenant/messaging/landlord-conversation`
  - 인증된 세입자의 `TenantRoom` 연결과 해당 방의 `landlordId`를 검증한다.
  - 기존 일반 문의 스레드가 있으면 `threadId`를 반환한다.
  - 없으면 연결 정보만 반환하며 스레드를 만들지 않는다.
- 기존 `POST /tenant/messaging/threads`
  - 일반 문의이면서 `contextRef`가 없을 때 동일 세입자·호실의 기존 일반 스레드가 있으면 새 스레드를 만들지 않는다.
  - 기존 스레드가 발견되면 제출한 첫 메시지를 그 스레드에 추가하고 해당 스레드를 반환한다.
  - 신규 스레드 생성과 기존 스레드 메시지 추가 모두 `roomlog:activity`를 발생시켜 관리자 목록을 갱신한다.

서버는 클라이언트가 보낸 임대인·호실 식별자를 신뢰하지 않고 인증된 세입자 관계에서 대상을 결정한다.

## 세입자 UI 흐름

`TenantMyPage`의 버튼 동작:

1. 연결된 입주 정보가 없으면 기존 안내 토스트를 유지한다.
2. 연결된 집이 있으면 `GET /api/tenant/messaging/landlord-conversation`을 호출한다.
3. 기존 `threadId`가 있으면 `/tenant/messaging/01?id=<threadId>`로 이동한다.
4. 기존 스레드가 없으면 첫 메시지 작성 패널을 연다.
5. 메시지를 전송하면 `POST /api/tenant/messaging/threads`로 일반 문의를 생성하고 반환된 상세 화면으로 이동한다.

기존 집주인 채팅 패널의 `TradeChatCenter`는 제거한다. 새 작성 패널은 현재 입주 건물·호실과 임대인 이름을 명시하고, 전송 중 중복 클릭을 막으며 오류를 패널 안에 표시한다.

## 관리자 UI와 실시간 반영

- 신규 세입자 메시지는 기존 관리자 소통 목록 API에 동일한 `MessagingThread`로 노출된다.
- 기존 컨트롤러의 `roomlog:activity` 브로드캐스트를 유지한다.
- 관리자 소통 허브는 기존 실시간 갱신 로직으로 신규 스레드와 마지막 메시지를 반영한다.
- 세입자 상세 화면은 기존 `MessageAutoRefresh`를 사용해 관리자의 답변을 반영한다.
- 이번 작업에서 웹소켓 프로토콜이나 인프라 설정은 변경하지 않는다.

## 오류와 경계 조건

- 로그인하지 않았거나 TENANT 역할이 아니면 기존 인증 정책대로 401/403을 반환한다.
- 세입자에게 연결된 방 또는 관리자가 없으면 대화를 만들지 않고 명시적인 안내를 반환한다.
- 빈 메시지는 클라이언트와 서버 양쪽에서 거부한다.
- 기존 일반 문의가 여러 개 남아 있는 이전 데이터에서는 가장 최근 갱신된 스레드를 선택한다.
- 생성 요청 직전에 관리자가 일반 스레드를 만든 경우에도 서버 재검사로 중복 스레드를 만들지 않고 제출 메시지를 기존 스레드에 추가한다.
- API 실패를 연결 없음으로 위장하지 않고 재시도 가능한 오류로 표시한다.

## 테스트

- API 도메인 테스트
  - 연결된 세입자에게 기존 일반 문의가 없으면 연결 정보만 반환한다.
  - 관리자 또는 세입자가 먼저 만든 기존 일반 문의를 조회한다.
  - 기존 일반 문의에 첫 메시지 요청을 보내면 스레드 수는 늘지 않고 메시지만 추가된다.
  - 신규 일반 문의는 관리자 목록에 같은 ID로 노출된다.
  - 다른 세입자·관리자의 스레드는 재사용하지 않는다.
  - 방 또는 관리자 연결이 없으면 거부한다.
- Web 단위 테스트
  - 세입자 대화 API 경로가 올바르다.
  - 기존 스레드가 있으면 상세 URL을 만든다.
  - 최초 작성 요청은 `general`과 첫 메시지를 전송한다.
  - 빈 메시지 및 중복 제출을 막는다.
- 통합 검증
  - `pnpm --filter @roomlog/types typecheck`
  - 관련 API/Web 테스트
  - `pnpm test:api`
  - `pnpm test:web`
  - `bash scripts/verify.sh`
  - Docker에서 세입자 첫 문의 → 관리자 소통 목록/상세 노출 → 관리자 답변 → 세입자 상세 반영을 브라우저로 확인한다.

## 범위 제외

- 거래 전 매물 문의 채팅의 데이터 마이그레이션
- 과거 `TradeChatCenter` 메시지를 룸로그 메시징으로 복사
- 웹소켓·Docker·배포·AWS 설정 변경
- 계약 생성·수락 또는 입주 연결 정책 변경
- 파일 첨부와 푸시 알림 신규 구현
