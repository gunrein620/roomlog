# Manager Listing Media Edit Design

## Goal

관리인이 매물 상세 팝업의 수정 모드에서 기본정보뿐 아니라 기존 사진과 3D 도면을 함께 변경할 수 있게 한다. 기존 미디어는 사용자가 명시적으로 삭제하거나 교체하기 전까지 유지한다.

## User Experience

### 사진

- 수정 화면에 기존 사진과 새로 선택한 사진을 하나의 미리보기 목록으로 표시한다.
- 사진마다 개별 삭제 버튼을 제공한다.
- 새 이미지 파일을 추가할 수 있으며 전체 사진은 최대 10장으로 제한한다.
- 목록의 첫 번째 사진을 대표 사진으로 사용한다.
- 이번 범위에서는 드래그 정렬을 추가하지 않는다. 기존 사진 순서를 유지하고 새 사진은 뒤에 추가한다.
- 저장 성공 후 카드와 상세 팝업의 대표 사진 및 사진 수를 즉시 갱신한다.

### 3D 도면

- 현재 연결 상태를 수정 화면에 표시한다.
- `3D 도면 만들기/다시 열기`는 `/floor-plan-3d`를 새 탭으로 열어 현재 입력과 선택 파일을 보존한다.
- 에디터가 로컬 저장소에 기록한 `roomlogListingFloorPlan3D` 스냅샷을 수정 화면 복귀 시 읽어 교체 후보로 반영한다.
- `도면 JSON 업로드`는 `walls3D` 배열 또는 기존 등록 화면이 지원하는 호환 구조를 정규화해 교체 후보로 반영한다.
- `3D 연결 해제`는 저장 시 `floorPlan: null`을 전송한다.
- 유효한 벽 데이터가 없는 JSON은 거부하고 기존 연결을 유지한다.

## Architecture

### Domain model

`ManagerListingRow`가 카드 표시용 요약뿐 아니라 편집에 필요한 `images`와 `floorPlan` 원본을 보유한다. API 응답을 행으로 변환할 때 배열과 도면 참조를 보존한다.

### Media helpers

사진 선택 병합, 최대 장수 검증, 3D JSON 정규화, 로컬 스냅샷 읽기는 별도 순수 유틸리티로 분리한다. 파일 업로드와 브라우저 저장소 접근은 컴포넌트 경계에서 수행하고 정규화 함수는 단위 테스트한다.

### Update API

수정 입력 계약에 `images`와 `floorPlan`을 추가한다.

- 사진을 수정하지 않으면 기존 `images` 배열을 그대로 전송한다.
- 사진을 삭제하거나 추가하면 최종 URL 배열을 전송해 서버에서 통째로 교체한다.
- 3D를 수정하지 않으면 기존 도면 객체를 그대로 전송한다.
- 새 도면을 연결하면 정규화된 도면 객체를 전송한다.
- 연결 해제는 `floorPlan: null`을 전송한다.

새 사진이 있으면 먼저 `/api/trade/uploads`로 업로드하고, 업로드가 모두 성공한 뒤 기본정보·최종 사진 URL·3D 도면을 하나의 PATCH 요청으로 저장한다.

## Error Handling

- 이미지가 아니거나 10장을 초과하면 업로드 요청 전에 오류를 표시한다.
- 사진 업로드가 실패하면 PATCH를 실행하지 않고 기존 서버 데이터를 유지한다.
- 3D JSON 파싱 또는 구조 검증이 실패하면 교체 후보로 반영하지 않는다.
- PATCH가 실패하면 팝업을 수정 모드로 유지하고 기존 목록 상태를 바꾸지 않는다.
- 저장 성공 후에만 로컬 목록 행을 서버 응답으로 교체한다.

## Accessibility

- 파일 입력은 명확한 접근성 이름을 가진다.
- 각 사진 삭제 버튼은 사진 순서를 포함한 이름을 가진다.
- 사진/3D 상태와 오류는 텍스트로 전달하며 색상만으로 구분하지 않는다.
- 저장 중에는 중복 제출과 미디어 변경 버튼을 비활성화한다.

## Testing

- 사진 병합과 10장 제한을 단위 테스트한다.
- 기존 사진 유지, 개별 삭제, 새 사진 업로드 후 최종 URL 배열 생성 흐름을 테스트한다.
- 유효/무효 3D JSON 정규화와 연결 해제 payload를 테스트한다.
- 수정 API가 `images`와 `floorPlan`을 포함하고 업로드 실패 시 PATCH하지 않는지 테스트한다.
- 수정 화면에 사진 추가·개별 삭제·3D 다시 열기·JSON 업로드·연결 해제 컨트롤이 노출되는지 테스트한다.
- 전체 웹 테스트와 `bash scripts/verify.sh`를 실행한다.

## Constraints

- `packages/ui/src/tokens.css`의 토큰만 사용하고 raw hex 색상은 추가하지 않는다.
- 인프라 파일은 수정하지 않는다.
- 서버의 기존 PATCH 계약을 사용하며 API 인프라나 데이터베이스 스키마를 변경하지 않는다.
- 기존 미추적 문서와 사용자 변경사항을 커밋에 포함하지 않는다.
