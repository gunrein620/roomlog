# 관리자 민원/하자 로컬 더미 데이터 설계

## 목표

관리자 `민원 대시보드`와 `민원/하자 관리`에서 로컬 접속일 때만 실제 API 데이터 뒤에 화면 확인용 더미 데이터 10개를 추가한다. 배포 도메인에서는 더미 데이터를 목록과 통계에 포함하지 않는다.

## 로컬 판별

- Next 서버가 받은 요청의 `Host` 헤더를 사용한다.
- 호스트 이름이 `localhost`, `127.0.0.1`, `[::1]`인 경우만 로컬로 판정한다.
- 포트 유무와 관계없이 위 세 호스트만 허용한다.
- 헤더가 없거나 형식이 잘못됐거나 배포 도메인이면 로컬로 판정하지 않는다.
- 쿼리 파라미터나 클라이언트 입력으로 로컬 판정을 켤 수 없게 한다.

## 데이터 구성

- 기존 `MANAGER_DEFECT_DASHBOARD_DEMO_RECORDS`에서 앞의 10개 레코드를 재사용한다.
- 새로운 더미 데이터 집합을 중복 정의하지 않는다.
- 각 레코드는 기존 `ticket`, `repair`, `buildingName`을 관리자 대시보드 행 형식으로 변환한다.
- 실제 API 행을 먼저 유지하고 더미 10개를 그 뒤에 추가한다.
- 로컬 API 호출이 실패해 실제 행이 비어도 더미 10개는 표시한다.
- 배포 호스트에서는 실제 API 행만 반환한다.

## 적용 범위

- `/manager/ticket/dash/00`: 기존 민원 통계 화면이 로컬 더미 데이터를 포함한다.
- `/manager/ticket/dash/00?view=management`: 통합 관리 테이블이 로컬 더미 데이터를 포함한다.
- 기존 유형 필터와 상세 화면의 더미 ID 조회 기능은 유지한다.

API, 공유 타입, 데이터베이스, Docker Compose, Dockerfile, workflow 및 배포 환경 변수는 변경하지 않는다.

## 테스트 및 검증

1. 로컬 호스트 세 종류와 포트 포함 주소가 로컬로 판정되는지 테스트한다.
2. 배포 도메인, 유사 도메인, 빈 헤더가 로컬로 판정되지 않는지 테스트한다.
3. 실데이터 뒤에 정확히 더미 10개가 추가되고 원본 배열은 변경되지 않는지 테스트한다.
4. 배포 호스트에서는 실데이터만 유지되는지 테스트한다.
5. 관련 테스트, web 전체 테스트, `bash scripts/verify.sh`를 실행한다.
6. 검증 통과 후 기능 커밋을 `kms-complaint1`에 푸시한다.
