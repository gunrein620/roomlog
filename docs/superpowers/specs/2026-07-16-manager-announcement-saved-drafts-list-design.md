# Manager Announcement Saved Drafts List Design

## 목표

관리자가 공지 작성 화면에서 임시 저장한 공지를 다시 찾고 불러와 수정할 수 있게 한다.

## 사용자 흐름

1. 관리자가 `/manager/messaging/01` 공지 작성 화면을 연다.
2. 화면 상단의 `임시 저장된 공지` 카드에서 아직 발송하지 않은 초안을 확인한다.
3. 초안의 `불러오기`를 누르면 `/manager/messaging/01?id=<draftId>`로 이동한다.
4. 기존 작성 폼이 초안의 대상, 제목, 본문, 번역 내용을 불러오고 이후 임시 저장은 같은 초안을 갱신한다.

## 화면 구성

- `ScreenHeader` 아래, `AnnouncementComposer` 위에 초안 목록 카드를 배치한다.
- 목록은 `updatedAt` 최신순으로 표시한다.
- 각 행에는 제목, 대상 라벨, 마지막 저장 시각, `불러오기` 링크를 표시한다.
- 제목이 비어 있으면 `제목 없는 공지`로 표시한다.
- `status === "draft"`인 항목만 표시하고 발송 완료 공지는 제외한다.
- 초안이 없으면 같은 카드 안에 `임시 저장된 공지가 없습니다.`를 표시한다.
- 삭제, 검색, 페이지네이션, 발송 완료 내역은 이번 범위에 포함하지 않는다.

## 구조와 데이터 흐름

- `apps/web/src/app/manager/messaging/01/page.tsx`가 기존 `listAnnouncementDrafts()`를 호출한다.
- 서버 페이지에서 `status === "draft"` 필터와 최신순 정렬을 적용한다.
- 초안 목록 표시는 작성 폼의 클라이언트 상태와 분리된 전용 표시 컴포넌트가 담당한다.
- `불러오기`는 기존 `getAnnouncementDraft(id)`와 `prepareAnnouncementDraftForCompose()` 경로를 그대로 사용한다.
- 임시 저장 API, 초안 타입, 백엔드 저장 방식은 변경하지 않는다.

## 오류와 폴백

- 기존 `listAnnouncementDrafts()`의 데모 폴백 정책을 유지한다.
- 상세 초안 조회 중 인증 오류가 발생할 때의 기존 로그인 이동 정책은 변경하지 않는다.
- 목록을 불러오기 위한 별도 클라이언트 로딩 상태나 재시도 UI는 추가하지 않는다.

## 테스트와 검증

- 공지 작성 페이지가 `listAnnouncementDrafts()`를 호출하는지 계약 테스트로 고정한다.
- 목록이 `draft` 상태만 필터링하고 최신순으로 정렬하는지 순수 함수 단위 테스트로 검증한다.
- 목록 컴포넌트가 제목, 대상, 저장 시각, `불러오기` 링크와 빈 상태를 렌더링하는지 검증한다.
- 기존 `?id=` 초안 불러오기 및 같은 초안 갱신 계약은 유지한다.
- 관련 web 테스트, 프로덕션 빌드, Docker 재빌드와 실제 브라우저 화면을 검증한다.
