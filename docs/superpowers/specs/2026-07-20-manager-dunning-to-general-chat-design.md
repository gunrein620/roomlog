# 관리인 AI 독촉 메시지의 계약 채팅 전달

## 목표

관리인이 AI 비서에 `103호 월세 독촉 보내`처럼 요청하면, 실제 미납 청구를 서버가 결정하고 확인 절차를 거친 뒤 해당 세입자와의 소통 허브 일반 채팅으로 납부 안내를 보낸다.

## 문제 원인

- AI 도구에 내부 `billId` 입력이 노출돼 있다. 모델이 호실 번호 `103`을 청구서 ID처럼 전달하면 서버가 자연어 호실 검색보다 이 값을 우선해 `청구서를 찾을 수 없습니다`로 종료한다.
- 독촉 발송 성공 시 기존 계약 일반 채팅이 아니라 `context=payment`, `contextRef=billId`인 별도 스레드를 직접 생성한다. 이 때문에 사용자가 기대하는 계약 관계의 단순 채팅 흐름과 분리된다.
- 모델이 첫 요청에서 서버 확인 카드를 만들지 않고 대화 문장으로 한 번 더 확인하면, 다음 `보내봐` 턴에는 대상 호실 정보가 사라질 수 있다.

## 설계

### 대상 결정과 확인

1. 채팅의 마지막 사용자 발화가 독촉 발송 의도를 명확히 포함하면 모델 응답에 의존하지 않고 서버가 `billing.send_dunning` 보류 작업을 준비한다.
2. 발송 의도는 독촉·미납·연체·월세·납부 문맥과 보내기·발송 요청이 함께 있는 경우로 제한한다. 단순 조회나 문구 초안 요청은 포함하지 않는다.
3. 일반 AI 채팅 도구 스키마에서는 내부 `billId` 입력을 제거한다. 청구 화면에서 전달하는 검증된 `intent.billId` 경로는 유지한다.
4. 자연어 요청은 호실, 임차인, 청구월 기준으로 관리인 소유의 현재 미납 청구를 서버에서 검색한다.
5. 확인 시점에 미납 잔액, 공개 상태, 납부 신고, 미연결 입금 가드를 다시 검사한다.

### 메시지 전달

1. 청구에 연결된 호실과 세입자를 서버에서 찾는다.
2. 같은 `roomId + tenantId`의 `context=general`이며 `contextRef`가 없는 최신 계약 일반 채팅을 재사용한다.
3. 기존 일반 채팅이 없으면 같은 계약 관계로 일반 채팅을 자동 생성한다.
4. 확인된 납부 안내를 관리자 발신 메시지로 추가하고 세입자 미확인 수를 증가시킨다.
5. 별도의 payment 스레드는 만들지 않는다.

### 응답과 이동

- 성공 문구는 `103호 소통 채팅으로 월세 독촉 메시지를 발송했습니다.`처럼 전달 위치를 명시한다.
- 완료 후 이동 경로는 생성 또는 재사용한 `/manager/messaging/04?id=<threadId>`이다.
- 계약 세입자나 호실을 찾지 못하면 발송 성공으로 응답하지 않고 원인을 표시한다.

## 테스트

- 모델이 내부 ID를 만들지 못하도록 도구 스키마에 `billId`가 없음을 검증한다.
- `103호 월세 독촉 보내`가 첫 턴에 보류 작업을 생성하는지 검증한다.
- 발송 확인 시 기존 일반 채팅을 재사용하고 메시지가 보이는지 검증한다.
- 일반 채팅이 없으면 자동 생성하는지 검증한다.
- payment 스레드가 생성되지 않는지 검증한다.
- 납부 신고 또는 미연결 입금이 있으면 채팅 생성과 발송을 모두 차단하는 기존 가드를 유지한다.

## 범위 제외

- SMS나 외부 문자 발송 연동
- 공지 발송 정책 변경
- 미납 가드 완화
- 소통 허브 화면 디자인 변경
