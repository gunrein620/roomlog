# 관리 비서 음성 대상 해석·확인 흐름 개선 설계

## 목표

관리인이 음성 또는 텍스트로 특정 호실의 독촉·공지를 요청할 때, 건물명 띄어쓰기나 경미한 음성 전사 오류 때문에 같은 정보를 반복해서 묻지 않도록 한다. 관리 범위 안에서 후보가 하나면 서버가 자동 선택하고, 실제로 여러 후보가 있을 때만 짧게 되묻는다.

발송 정책 자체는 유지한다. 독촉과 공지는 초안을 한 번 보여준 뒤 관리인이 `승인`, `진행해`, `보내` 중 하나로 확인해야 실행한다. 다만 확인 상태는 API 재시작과 브라우저 재접속 뒤에도 살아 있어야 한다.

## 확인된 원인

1. 공지 대상 해석은 입력에 실제 `buildingName`이 그대로 포함되는지만 확인한다. `플로우테스트`가 `플로어 테스트`로 전사되면 같은 건물로 보지 못한다.
2. 호실은 `/숫자+호/`만 인식한다. 음성 전사가 숫자를 깨뜨리면 후보 축소가 불가능하다.
3. 독촉 요청에서 모델이 `messaging.send_reply`를 선택하면 서버 정책 차단으로 끝나며, 사용자의 독촉 의도를 올바른 도구로 교정하지 않는다.
4. 기존 코파일럿 확인 상태는 프로세스 메모리 `Map`에 저장하지만 브라우저 대화는 `sessionStorage`에 남는다. API 재시작·배포 뒤 브라우저가 사라진 확인 ID를 보내면 `확인할 수 있는 보류 액션이 아닙니다`가 발생한다.
5. 저장소에는 이미 PostgreSQL 기반 `AgentToolAction` 확인 저장소와 재시작 안전한 확인 API가 있지만, 현재 관리 비서 독촉 흐름이 이를 우회한다.

## 검토한 접근

### A. 프롬프트만 느슨하게 수정

모델에게 비슷한 건물명을 알아서 추정하도록 지시한다. 수정량은 가장 작지만 모델이 실제 관리 범위 밖 대상을 만들 수 있고, 확인 상태 유실은 해결하지 못한다.

### B. 서버 후보 해석 완화 + 기존 DB 확인 게이트 재사용

서버가 관리인의 실제 방·계약·청구 후보 안에서만 대상을 정규화하고, 후보가 하나면 자동 선택한다. 독촉·공지의 확인 상태는 기존 PostgreSQL `AgentToolAction`으로 통합한다.

이 방식을 채택한다. 사용자 편의는 높이면서 대상과 발송 정책을 서버가 계속 통제할 수 있다.

### C. 발송 확인 완전 제거

첫 요청에서 바로 발송한다. 가장 빠르지만 음성 오인식이 실제 외부 발송으로 이어질 수 있어 채택하지 않는다.

## 대상 해석

### 공통 정규화

대상 문자열과 실제 건물명에 다음 정규화를 적용한다.

- Unicode NFKC 정규화
- 영문 소문자화
- 공백, 하이픈, 일반 문장부호 제거
- `건물`, `빌라`, `호실` 같은 일반 접미어는 비교 점수에서 낮은 비중으로 처리
- 호실 숫자는 건물명 비교 전에 분리

### 후보 선택 순서

1. 명시된 내부 참조 ID 또는 청구 ID가 있으면 그대로 사용한다.
2. 숫자 호실과 정확한 건물명이 모두 있으면 정확 일치한다.
3. 숫자 호실이 있으면 관리 범위 내 같은 호실 후보만 남긴 뒤 건물명 유사도로 정렬한다.
4. 건물명만 인식되면 유사도 기준으로 건물을 찾는다.
5. 필터링 결과가 하나면 자동 선택한다.
6. 결과가 여러 개면 최대 3개의 실제 후보 이름만 보여주고 선택을 요청한다.
7. 결과가 없으면 추측 발송하지 않고 관리 중인 실제 후보를 안내한다.

건물명 유사도는 정규화 문자열의 포함 관계를 먼저 보고, 그다음 편집 거리 기반 점수를 사용한다. 최고 점수가 기준 이상이고 2위와 충분한 차이가 있을 때만 자동 선택한다. 점수 기준은 테스트 사례로 고정하고 운영 데이터 밖의 이름을 새로 만들지 않는다.

### 독촉 특례

독촉은 실제 `미납 잔액 > 0`이고 기존 입금 가드를 통과한 청구만 후보가 된다.

- 연체 후보가 하나면 건물명이나 호실 전사가 일부 깨져도 그 청구를 초안 대상으로 자동 선택한다.
- 여러 후보면 인식된 호실, 세입자명, 청구월 순으로 좁힌다.
- 모델이 일반 답장 도구를 선택했더라도 사용자 문장에 독촉·미납·연체·납부 요청이 있으면 서버가 `billing.send_dunning` 준비 경로로 교정한다.
- 실제 발송 직전에는 기존 `sendManagerDunning` 가드를 다시 실행한다.

## 확인 상태와 발송

기존 `pendingCopilotActions` 메모리 저장소를 독촉·공지 경로에서 제거하고 `AgentToolGateService`를 사용한다.

- `billing.send_dunning`: 기존 `PREPARE` 정책과 PostgreSQL 저장소를 그대로 사용한다.
- `messaging.send_announcement`: 관리인 도구 목록에 `PREPARE` 정책으로 추가한다.
- 준비 시 서버가 확정한 대상 ID, 제목, 본문, 청구 ID를 command payload에 저장한다.
- 패널이 열리거나 음성 연결이 시작될 때 `manager/agent-confirmations/current`를 조회해 현재 확인 건을 복원한다.
- `승인`, `진행해`, `보내`는 공백과 문장부호를 제거한 뒤 판별한다.
- 확인 ID가 만료되었거나 이미 실행된 경우 서버의 현재 확인 건을 다시 조회한다. 현재 건이 없으면 같은 사용자 요청으로 새 초안을 한 번만 만들고 이유를 짧게 안내한다.
- 확인 후 실행은 confirmation ID를 멱등 키로 사용해 중복 발송을 막는다.

## 대화 UX

- 성공: `102호 일반 소통 채팅으로 독촉을 발송했습니다.`처럼 결과와 전달 위치를 한 문장으로 말한다.
- 후보 하나: 추가 질문 없이 초안을 보여주고 한 번만 승인받는다.
- 후보 여러 개: `102호가 두 곳입니다. 플로우테스트1과 플로우테스트2 중 어디인가요?`처럼 실제 후보만 묻는다.
- 전사 불명확: 사용자가 말한 깨진 문자열을 반복하지 않고, 서버가 찾은 실제 후보명을 제시한다.
- 차단: “이 경로에서는 안 된다”는 일반 문구 대신 입금 확인 중, 미납 없음, 계약 세입자 없음 등 실제 서버 가드 사유를 말한다.
- 공지 수신자가 없는 호실은 발송하지 않되, 계약/입주 연결이 없는 실제 이유를 안내한다.

## 오류 처리

- 유사도 동률 또는 기준 미달이면 자동 선택하지 않는다.
- 관리 범위 밖 방은 후보에 포함하지 않는다.
- DB 확인 저장소가 일시적으로 실패하면 발송하지 않고 재시도 가능한 안내를 남긴다.
- 음성 전사와 도구 호출 순서가 바뀌어도 서버의 현재 확인 건을 기준으로 처리한다.
- 실행 응답을 받지 못한 재시도는 같은 confirmation ID로 조회하여 중복 발송을 방지한다.

## 테스트

### 대상 해석

- `관리자 세입자 플로어 테스트 2 102호`가 실제 `관리자-세입자 플로우테스트2 102호` 하나로 해석된다.
- 같은 `102호`가 두 건물에 있고 건물 단서가 없으면 두 후보를 반환한다.
- 유사한 건물명이 둘이고 점수 차이가 작으면 자동 선택하지 않는다.
- 관리 범위 밖 건물명은 선택되지 않는다.

### 독촉

- 연체 후보가 하나면 불완전한 음성 대상에서도 해당 청구 초안이 생성된다.
- 일반 답장으로 잘못 분류된 독촉 문장은 독촉 준비 경로로 교정된다.
- 승인 전에는 발송되지 않고, 승인 한 번 후 일반 소통 채팅에 정확히 한 번 기록된다.
- 입금 확인 중이거나 미납이 없으면 기존 가드가 차단한다.

### 확인 상태

- API 서비스 인스턴스를 다시 만든 뒤에도 DB에 저장된 확인 건을 조회하고 승인할 수 있다.
- 브라우저의 오래된 확인 ID는 현재 서버 확인 건으로 복구된다.
- `진행해.`, `승인!`, `보내`를 확인으로 인식한다.
- 같은 확인 요청을 재시도해도 중복 발송되지 않는다.

### 공지

- 유사 건물명과 정확한 호실로 공지 수신자를 찾는다.
- 수신 계약이 없는 호실은 발송하지 않고 구체적인 이유를 반환한다.
- 공지도 재시작 뒤 확인 상태가 유지되고 한 번만 발송된다.

## 제외 범위

- 범용 한국어 음성 교정 사전 구축
- 관리 범위 밖 주소 검색
- 승인 없는 즉시 외부 발송
- UI 확인 버튼 복원
