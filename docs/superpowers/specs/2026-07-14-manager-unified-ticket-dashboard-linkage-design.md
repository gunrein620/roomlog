# 관리자 민원/하자 통합 대시보드 데이터 연동 설계

## 목표

`/manager/ticket/dash/00` 기본 대시보드가 관리자에게 배정된 민원과 하자 티켓을 모두 집계하도록 변경한다. 현재 API에서 정상 조회되는 하자 티켓이 `ticket.type === "complaint"` 필터로 제외되어 0건처럼 보이는 문제를 해결하고, API 오류와 실제 빈 상태를 구분하며 신규 접수도 자동 반영한다.

## 사용자 경험

- 기본 화면 제목은 `민원/하자 대시보드`로 표시한다.
- 요약 카드, 최근 6개월 추이, 유형별 비율, 최근 접수 내역, CSV 보고서는 선택 월의 민원과 하자 티켓을 모두 사용한다.
- 기존 `?type=complaint`, `?type=defect` 목록 링크와 필터는 호환성을 유지한다.
- API가 빈 배열을 정상 반환하면 0건 빈 상태를 표시한다.
- 인증, 네트워크 또는 서버 오류가 발생하면 0건으로 위장하지 않고 재시도 가능한 오류 화면을 표시한다.
- 기본 대시보드도 티켓 변경 이벤트와 주기적 갱신을 받아 신규 접수를 수동 새로고침 없이 반영한다.

## 데이터 계약

### 공유 타입

`packages/types`의 티켓 모델에 도메인 구분값을 단일 소스로 둔다.

```ts
type TicketKind = "defect" | "complaint";
```

웹은 한글 카테고리 문자열을 직접 해석해 티켓 종류를 추론하지 않는다. API 응답의 명시적인 구분값을 우선 사용한다. 과거 데이터나 구버전 API 응답을 위한 호환 분류는 매퍼 내부의 제한된 fallback으로만 남기고, 알 수 없는 값은 자동으로 일반 민원으로 부풀리지 않는다.

### API

- 티켓 생성 시 분석 카테고리에서 `TicketKind`를 한 번 결정해 티켓 원천 데이터에 저장한다.
- `GET /manager/tickets`는 각 티켓의 명시적인 구분값을 반환한다.
- 관리인 권한 범위인 `landlordId + roomId` 필터는 기존대로 서버에서 강제한다.
- 기존 티켓은 저장된 카테고리를 기반으로 호환 구분값을 산출하여 응답한다.

### 웹 매핑

- `toManagerTicket`은 API의 명시적인 구분값을 `@roomlog/types`의 `Ticket.type`으로 전달한다.
- 한글 카테고리 whitelist는 구버전 응답 fallback으로 격리한다.
- 통합 대시보드 모델은 선택 월의 전체 행을 입력으로 사용하고 `Ticket.type`은 유형별 비율에만 사용한다.

## 대시보드 계산

- 전체 접수: 선택 월의 민원과 하자 티켓 수
- 처리 중, 대기, 완료: 기존 `ticketStatusGroup` 기준 유지
- 지난달 대비: 이전 달의 전체 티켓 수와 비교
- 최근 6개월: 각 월의 전체 티켓 수
- 유형별 비율: `하자`와 `일반 민원`을 1차 구분하고, 화면의 기존 세부 분류가 필요하면 `수리 요청`, `소음 민원`, `결제 문의`, `기타`를 보조 표시한다.
- 최근 접수: 선택 월의 전체 티켓을 최신순으로 최대 5건 표시
- CSV: 화면과 동일한 전체 데이터 범위를 사용하며 유형 열에 `하자` 또는 일반 민원 세부 유형을 기록

집계 함수 이름은 일반 민원 전용 의미를 제거해 통합 대시보드 역할이 드러나도록 변경한다. 화면 컴포넌트도 동일하게 통합 명칭을 사용한다.

## 오류 처리와 갱신

`listManagerTicketRows`는 API 오류를 `[]`로 변환하지 않는다. 호출 실패는 상위 route error boundary로 전달한다. 티켓 대시보드 경로에 전용 `error.tsx`를 두어 오류 안내와 `reset()` 재시도를 제공한다.

`TicketDashboardAutoRefresh`는 관리 목록뿐 아니라 기본 통합 대시보드에도 마운트한다. 기존 티켓 활동 이벤트 기반 갱신과 주기적 fallback을 재사용하며 별도 실시간 채널은 추가하지 않는다.

## 구현 슬라이스

### 슬라이스 1: 통합 집계와 화면

- 통합 모델 테스트에서 같은 달의 `defect`와 `complaint`가 모두 합산되는 실패 사례를 먼저 추가한다.
- 모델, 제목, 카드, 범례, 최근 내역, CSV를 통합 범위로 변경한다.
- 기본 대시보드에 기존 자동 갱신 컴포넌트를 연결한다.
- targeted web 테스트와 관련 UI 계약 테스트를 통과시킨 뒤 커밋·푸시한다.

### 슬라이스 2: 명시적인 티켓 종류 계약

- 공유 타입 및 API 테스트에 명시적인 `TicketKind` 전달 실패 사례를 먼저 추가한다.
- API 생성·응답과 웹 매퍼를 단일 계약으로 연결한다.
- `packages/types`를 먼저 빌드하고 API·web 관련 테스트를 통과시킨 뒤 커밋·푸시한다.

### 슬라이스 3: 오류 상태 분리

- API 오류가 빈 목록으로 변환되지 않는 실패 테스트를 먼저 추가한다.
- 오류 전파와 티켓 대시보드 전용 재시도 화면을 구현한다.
- 관련 테스트와 전체 검증을 통과시킨 뒤 커밋·푸시한다.

## 검증 기준

- 같은 월에 하자 1건과 일반 민원 1건이 있으면 전체 접수는 2건이다.
- 하자 티켓도 월별 추이, 최근 접수, CSV에 포함된다.
- `?type=complaint`와 `?type=defect` 관리 목록은 기존대로 필터링된다.
- API의 명시적인 티켓 종류가 웹까지 손실 없이 전달된다.
- API 오류는 0건 화면이 아니라 오류 화면으로 나타난다.
- 신규 티켓 이벤트 또는 주기 갱신 시 기본 대시보드가 다시 조회된다.
- 각 슬라이스에서 targeted 테스트를 먼저 통과시키고 마지막에 `bash scripts/verify.sh`를 실행한다.

## 제외 범위

- 신규 대시보드 전용 백엔드 집계 API
- 데이터베이스 마이그레이션이 필요한 별도 통계 테이블
- 기존 티켓 상태 체계 또는 권한 스코프 변경
- Docker Compose, 배포 워크플로, AWS 인프라 변경
