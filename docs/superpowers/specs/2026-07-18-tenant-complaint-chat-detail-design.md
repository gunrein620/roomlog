# 세입자 민원/하자 이력 상세 채팅 중심화 설계

## 목적

세입자가 `민원/하자 이력`에서 접수 건을 열었을 때, 접수 이후의 모든 진행을 한 티켓 채팅에서 확인하고 조율하게 한다. 신규 요청 화면은 변경하지 않는다.

## 확정 화면

상세 모달에는 다음 요소만 남긴다.

1. 접수 제목과 닫기 버튼
2. 현재 처리 상태와 `긴급도 N`을 나란히 표시하는 요약 카드
3. 해당 티켓의 진행 메시지 목록과 메시지 입력창

현재 상세에 표시되는 요청 유형, 발생일시, 본문, 발생 위치, 첨부 이미지, 책임 추정, 관리자 책임 확정, 직접 처리 안내, 완료 확인, 업체 연결, 책임 이의제기는 제거한다. 접수 원문과 관련 데이터는 서버에서 삭제하지 않으며 이 화면에서만 숨긴다. 후속 설명과 조율은 진행 메시지 스레드에서 수행한다.

## 상태와 예외

- 상세 API를 불러오는 동안에는 목록에서 이미 가진 제목을 유지하고, 기존 로딩 상태를 사용한다.
- 상세 조회 실패는 기존 오류 안내를 표시하되 접수 제목은 유지한다. 티켓 상세가 없으면 상태 카드와 채팅 입력은 표시하지 않는다.
- 완료 또는 취소된 티켓은 메시지 이력을 읽을 수 있지만 새 메시지 입력창은 표시하지 않는다.
- 신규 요청 모달과 접수 생성 데이터는 변경하지 않는다.

## 구현 경계

- `TenantMyPage.tsx`의 이력 상세 렌더링과 상세 전용 상태·핸들러만 정리한다.
- 제거된 상세 블록에만 쓰이던 스타일을 삭제하고, 상태 카드 아래 채팅이 자연스럽게 이어지도록 기존 토큰 기반 스타일을 유지한다.
- API, 공유 타입, 관리자 및 업체 화면은 변경하지 않는다.

## 검증

- 소스 계약 테스트로 접수 제목, 상태·긴급도, 진행 메시지와 입력이 남아 있는지 확인한다.
- 같은 테스트로 상세 모달에서 요청 유형, 발생일시, 본문, 위치, 첨부, 책임/이의제기, 완료 확인, 업체 연결 UI가 제거됐는지 확인한다.
- 신규 요청 폼의 유형, 발생일시, 본문, 긴급도 입력은 그대로 존재하는지 별도로 확인한다.
- 웹 단위 테스트와 기본 저장소 검증을 실행한다.
