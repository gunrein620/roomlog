# 공지 수신자 없음 처리 설계

## 목표

관리자 공지의 대상 호실에 계약 세입자가 연결되어 있지 않을 때 발송을 시도하지 못하게 하고, 발송 시점에 API가 거절하더라도 Next.js 전체 화면 서버 오류 대신 검토 화면 안에서 원인을 안내한다.

## 범위

- `TenantRoom`으로 연결된 계약 세입자만 실제 공지 수신자로 유지한다.
- 수신자 명단이 비어 있으면 검토 화면에 명시적인 빈 상태를 표시한다.
- 수신자가 없으면 발송 버튼 문구를 `수신자 없음`으로 바꾸고, 회색 배경과 회색 글자로 비활성 상태를 명확히 표시한다.
- 화면을 연 뒤 수신자 연결이 사라지는 경우처럼 발송 시점에 API가 `400`을 반환해도 검토 화면 안에서 오류를 표시한다.
- 인증 오류 `401`, `403`의 기존 로그인 이동은 유지한다.
- 발송 성공 시 기존 발송 결과 화면 `/manager/messaging/03`으로 이동한다.

## 제외 범위

- 빈 호실에 테스트 세입자를 자동 생성하거나 `TenantRoom` 관계를 임의로 추가하지 않는다.
- 계약·초대·세입자 연동 도메인의 생성 규칙을 변경하지 않는다.
- Docker Compose, 배포 환경, AWS, CI/CD 설정을 변경하지 않는다.
- API의 `공지 수신 세대가 없습니다.` 검증을 제거하거나 우회하지 않는다.

## 설계

### 1. 검토 화면의 빈 수신자 상태

검토 화면은 `listAnnouncementRecipients(draft.id)` 결과를 기준으로 `hasRecipients`를 계산한다. 수신자가 한 명 이상이면 기존 명단과 발송 버튼을 유지한다. 수신자가 0명이면 명단 영역에 `연결된 계약 세입자가 없습니다.`와 계약 세입자 연결이 필요하다는 설명을 표시한다. 발송 버튼은 실제 `disabled` 속성을 유지하면서 문구를 `수신자 없음`으로 바꾸고, 공유 디자인 토큰의 회색 배경·글자와 `not-allowed` 커서를 적용한다. 공용 `Button`은 변경하지 않아 다른 화면에 영향을 주지 않는다.

대상 라벨의 `10세대`는 관리 중인 호실 수이며 실제 발송 수신자 수가 아니다. 화면에서는 수신자 명단 상태를 별도로 보여 주어 두 개념을 혼동하지 않게 한다.

### 2. 발송 오류의 화면 내 처리

발송 서버 액션은 성공 결과만 발송 결과 화면으로 리다이렉트한다. `401`, `403`은 기존처럼 로그인 화면으로 이동한다. 그 밖의 `ApiError`는 검토 화면으로 돌아오면서 오류 상태를 전달하고, 화면은 해당 메시지를 경고 영역에 표시한다.

오류 상태는 사용자가 수정하거나 재시도할 수 있도록 현재 초안 ID를 보존한다. 알 수 없는 오류에는 내부 오류 세부사항 대신 일반적인 발송 실패 안내를 사용한다.

### 3. API 불변식

API는 초안의 대상 호실과 `TenantRoom` 관계를 교차해 수신자를 계산한다. 결과가 0명이면 현재와 동일하게 `400 Bad Request`를 반환한다. 이 검증은 화면 사전 차단을 우회하거나, 화면을 연 뒤 데이터가 바뀐 경우를 막는 최종 안전장치다.

## 데이터 흐름

1. 관리자가 공지 검토 화면에 진입한다.
2. 웹 서버가 초안과 실제 계약 수신자 명단을 조회한다.
3. 수신자가 0명이면 빈 상태를 표시하고 발송 버튼을 비활성화한다.
4. 수신자가 있으면 관리자가 발송을 승인한다.
5. API가 발송 직전 수신자를 다시 계산한다.
6. 수신자가 있으면 공지와 배송 내역을 생성하고 결과 화면으로 이동한다.
7. 수신자가 사라졌거나 다른 400 오류가 발생하면 검토 화면에서 오류를 표시한다.

## 테스트 전략

### 기능 1: 빈 수신자 안내

- 수신자 배열이 비어 있을 때 빈 상태 문구가 노출되는지 검증한다.
- 수신자가 없을 때 발송 버튼이 비활성화되는지 검증한다.
- 수신자가 없을 때 버튼 문구가 `수신자 없음`이고 회색 토큰 스타일이 적용되는지 검증한다.
- 수신자가 있을 때 기존 명단과 활성 발송 버튼이 유지되는지 검증한다.
- 대상 웹 테스트와 웹 빌드 통과 후 기능 단위로 커밋·푸시한다.

### 기능 2: 발송 오류 처리

- 발송 API의 `400` 오류가 전체 화면 오류로 전파되지 않는지 검증한다.
- 검토 화면에 오류 문구와 동일한 초안 ID가 유지되는지 검증한다.
- `401`, `403` 로그인 이동과 성공 시 결과 화면 이동이 유지되는지 검증한다.
- 대상 웹 테스트와 웹 전체 테스트 통과 후 기능 단위로 커밋·푸시한다.

### 최종 검증

- `pnpm test:web`
- `pnpm test:api`
- `bash scripts/verify.sh`
- Docker Compose 웹·API 재빌드 후 브라우저에서 수신자 없는 공지 검토 및 발송 차단 확인

## 인프라 경계

이 변경은 웹 화면과 서버 액션의 오류 표현에 한정한다. 인프라 파일 수정 필요성이 발견되면 `.local-agents/local-infra-guard.prompt.md`의 보고 형식으로 먼저 사용자에게 알리고 승인 전에는 변경하지 않는다.
