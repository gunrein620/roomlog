# 관리자 크레딧 Toss 결제창 중첩 해소 설계

## 배경

관리자 크레딧 충전은 네이티브 `<dialog>` 안에서 Toss 결제위젯을 렌더링한다. 구매자가 결제를 요청하면 Toss가 데스크톱용 결제 iframe과 자체 오버레이를 추가하지만 기존 충전 다이얼로그와 backdrop도 열린 채 남아 이중으로 어두워지고, 비활성화된 `결제 요청 중` 버튼까지 비쳐 보인다.

## 목표

- Toss 결제 iframe이 열린 동안 기존 충전 다이얼로그가 뒤에 보이지 않게 한다.
- 결제 주문, 금액, 성공·실패 콜백과 서버 승인 흐름은 변경하지 않는다.
- Toss 요청 취소나 SDK 오류가 발생하면 관리자가 오류를 확인하고 다시 시도할 수 있게 한다.

## 비목표

- Toss iframe 내부 디자인 변경
- 결제 API, 크레딧 원장 또는 주문 상태 모델 변경
- 별도 결제 페이지 추가
- 다른 관리자·세입자 결제창 개편

## 동작 설계

1. 관리자가 준비된 결제위젯에서 결제를 요청한다.
2. 현재 다이얼로그와 위젯이 열린 상태에서 `requestPayment()`를 먼저 호출한다. 이를 통해 Toss가 선택된 결제수단과 주문 정보를 인계받는다.
3. SDK 호출이 시작된 직후 기존 크레딧 충전 다이얼로그만 `close()`한다. React 상태, 위젯 인스턴스와 준비 주문은 유지하며 취소 API는 호출하지 않는다.
4. 결제 성공은 기존 `successUrl`로 이동해 서버 승인과 크레딧 반영을 그대로 수행한다.
5. 결제 실패는 기존 `failUrl` 흐름을 그대로 사용한다.
6. 사용자가 Toss 결제창을 닫거나 SDK가 동기·비동기 오류를 반환하면 기존 정리 절차로 READY 주문을 취소하고 충전 다이얼로그를 다시 연 뒤 오류 메시지를 표시한다.

API 개별 연동 키를 사용하는 직접 결제창 경로와 결제위젯 키 경로 모두 같은 호출 순서를 사용한다.

## 안전 조건

- Toss 호출 전에 다이얼로그를 닫지 않는다.
- 다이얼로그를 닫는 행위만으로 준비 주문을 취소하지 않는다.
- 결제 요청 중 컴포넌트와 위젯 DOM을 언마운트하지 않는다.
- Toss 호출이 거부되면 빈 화면으로 남기지 않고 다이얼로그를 복구한다.
- 중복 클릭 방지는 기존 `submitting` 상태를 유지한다.

## 테스트

- 결제위젯 경로에서 SDK 요청이 다이얼로그 닫기보다 먼저 실행되는지 검증한다.
- 직접 결제창 경로도 같은 순서를 지키는지 검증한다.
- SDK 요청 거부 시 준비 주문 정리 후 다이얼로그가 다시 열리고 오류가 노출되는지 검증한다.
- 결제 요청 중에는 중복 주문이나 중복 SDK 요청이 발생하지 않는 기존 회귀 테스트를 유지한다.
- 웹 단위 테스트, 프로덕션 빌드, 기본 검증 스크립트와 Docker 재빌드를 실행한다.
