# 관리자 하자 관리 대시보드 설계

## 목표

`/manager/ticket/dash/00` 본문을 첨부된 Stitch 디자인의 정보 구조와 시각 위계에 맞게 재구성한다. 기존 티켓·수리 API 데이터를 계속 사용하고, 티켓 상세 이동을 유지한다.

## 범위

- 변경 대상은 `apps/web/src/app/manager/ticket/dash/00`과 그 화면 전용 컴포넌트·스타일·테스트다.
- 제목, 상태 필터, 선택 필터, 관리 표, 페이지 정보를 첨부 디자인과 같은 순서로 배치한다.
- `ManagerSidebar`, `manager-navigation.ts`, 다른 사이드바 항목과 다른 관리자 화면은 수정하지 않는다.
- Dockerfile, Compose, workflow, 배포 환경 파일은 수정하지 않는다.

## 화면 구조

1. `하자 관리` 제목
2. 상태 칩: 전체, 대기, 진행중, 완료, 취소, 정기점검
3. 선택 필터: 담당자, 건물, 템플릿
4. 관리 표: 유형, 작업명, 건물, 호실, 작업자, 예정일시, 청구 금액, 상태, 작업
5. 결과 건수와 간단한 페이지 이동

첨부 시안의 간격, 둥근 컨테이너, 상태 배지, 파란색 주요 액션을 재현하되 모든 색상과 크기는 기존 Roomlog CSS 토큰을 사용한다. 작은 화면에서는 표를 찌그러뜨리지 않고 가로 스크롤을 제공한다.

## 데이터와 상호작용

- 서버 페이지가 `listManagerTickets()`와 `getManagerRepair()`로 실데이터를 조회한다.
- 화면 전용 클라이언트 컴포넌트가 조회 결과를 받아 상태 칩과 선택 필터를 적용한다.
- 대기에는 `received`, `reviewing`, `info_requested`, `reopened`를 포함하고, 진행중은 `processing`, 완료는 `resolved`, 취소는 `cancelled`로 계산한다.
- 현재 도메인에 정기점검 타입이 없으므로 정기점검 필터는 0건의 빈 결과를 명확히 보여준다. 샘플 데이터를 만들어 채우지 않는다.
- 건물명과 담당자 연락처는 현재 공유 `Ticket` 계약에 없으므로 값을 추측하지 않는다. 건물은 `—`, 작업자는 연결된 수리업체명 또는 `미배정`으로 표시한다.
- 예정일시와 청구 금액은 연결된 `RepairJob` 값이 있을 때만 표시하며 없으면 `—`로 표시한다.
- 행의 작업 버튼은 기존 `ticketDashHref("01", ticket.id)` 상세 경로로 이동한다.
- API 오류나 빈 목록은 기존 빈 배열 계약을 존중하고 표 안에 빈 상태를 표시한다.

## 컴포넌트 경계

- `page.tsx`: 서버 데이터 조회와 티켓·수리 결합만 담당한다.
- 화면 전용 클라이언트 컴포넌트: 필터 상태, 집계, 표 렌더링만 담당한다.
- 기존 공용 `ManagerAppShell`과 사이드바는 그대로 사용한다.

## 검증

- 회귀 테스트에서 사이드바·전역 내비게이션 파일이 변경 대상이 아닌지 확인한다.
- 새 화면 테스트로 제목, 여섯 상태 칩, 세 선택 필터, 아홉 표 열, 상세 링크가 존재하는지 검증한다.
- 필터 함수는 상태별 분류와 정기점검 빈 결과를 실제 데이터로 검증한다.
- `pnpm test:web`과 관련 빌드를 통과시킨다.
- Docker web 이미지를 재빌드한 뒤 `/manager/ticket/dash/00`을 데스크톱 뷰포트에서 확인한다.
