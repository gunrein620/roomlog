# 관리인 공지 작성 화면 및 언어별 자동 번역 설계

## 목적

관리인 공지 작성 화면(M-MSG-01)을 제공된 Stitch 시안의 카드형 레이아웃으로 개편하고, 실제 관리 건물·호실 선택, 같은 초안의 임시 저장·수정, 언어별 OpenAI 자동 번역, 긴급 공지 검수 게이트까지 정상 동작하도록 만든다.

기존 `ManagerShell`, `@roomlog/ui` 디자인 토큰, 공지 작성→검토→발송 흐름과 D20/D21 원칙은 유지한다. 첨부 HTML의 Azure Horizon 브랜드, 전역 셸, Tailwind CDN, 외부 이미지, raw hex 스타일은 이식하지 않는다.

## 사용자 흐름

### 새 공지 작성

1. `/manager/messaging/01`은 데모 초안 대신 빈 폼으로 시작한다.
2. 관리인은 긴급·생활·행사 카테고리 중 하나를 고른다.
3. 관리인은 전체·건물·호실 범위를 고르고 로그인 계정의 실제 관리 호실에서 대상을 선택한다.
4. 관리인은 한국어 제목과 본문을 입력한다.
5. `임시 저장`은 초안을 생성하거나 같은 초안을 갱신하고 현재 화면에 머문다.
6. 저장 후 URL은 `?id=<draftId>`를 포함하고 화면에 저장 완료 시각을 표시한다.
7. `검토하고 발송으로`는 최신 입력을 저장한 뒤 M-MSG-02로 이동한다.

### 기존 초안 수정

1. `?id=<draftId>`가 있으면 관리인 소유 초안을 불러온다.
2. 수정 후 임시 저장하면 새 초안을 만들지 않고 기존 초안을 갱신한다.
3. 이미 발송된 초안은 수정할 수 없다.

### 언어별 자동 번역

1. 긴급 공지를 선택하면 우측에 `English`, `中文`, `Tiếng Việt` 버튼을 표시한다.
2. 버튼 하나를 누르면 현재 한국어 제목·본문을 해당 언어 하나로만 번역한다.
3. 번역 중에는 해당 버튼만 로딩·비활성 상태가 된다. 다른 언어 결과와 입력 상태는 유지한다.
4. 번역 결과의 제목·본문은 관리인이 직접 수정할 수 있다.
5. 각 언어에는 별도의 `검수 완료` 체크가 있다.
6. 이미 번역된 언어 버튼을 다시 누르면 기존 내용을 덮어쓸지 확인한 뒤 재번역한다.
7. 한국어 원문이 바뀌면 기존 번역문은 보존하되 모든 번역의 검수 상태를 해제한다.
8. 긴급 공지는 영어·중국어·베트남어 번역이 모두 존재하고 현재 원문 기준이며 검수 완료여야 검토 단계로 이동할 수 있다.
9. 번역 API 실패 시 기존 번역을 보존하고 해당 언어 카드에 오류와 재시도 동작을 제공한다.
10. 자동 번역을 사용할 수 없어도 관리인이 번역 제목·본문을 직접 입력하고 검수할 수 있다. 가짜 번역이나 성공 상태는 만들지 않는다.

생활·행사 공지는 이번 범위에서 강제 번역 대상이 아니다. 기존 원문/선택 언어 제공 원칙을 유지하고 긴급 공지만 D21 검수 게이트를 적용한다.

## 화면 구조

페이지 전역 셸은 기존 `ManagerAppShell`/`ManagerShell`이 담당한다. M-MSG-01 내부만 다음 구조로 교체한다.

- 상단: breadcrumb, `공지 작성` 제목, `허브` 보조 버튼
- 좌측 본문
  - 카테고리 pill 버튼
  - 대상 범위 라디오와 실제 건물·호실 선택 카드
  - 공지 제목·상세 내용 입력
  - `검토하고 발송으로` 기본 버튼과 `임시 저장` 보조 버튼
- 우측 패널
  - 다음 화면에서만 발송된다는 확인 게이트 안내
  - 긴급 공지 언어별 번역 버튼과 번역 편집·검수 카드
  - 현재 저장 상태와 오류 상태

데스크톱에서는 약 8:4 비율의 2열을 사용하고 좁은 화면에서는 한 열로 쌓는다. 모든 색상·간격·radius·shadow는 `packages/ui/src/tokens.css` 변수로 표현한다. raw hex, Tailwind CDN, 외부 폰트, 외부 프로필 이미지는 추가하지 않는다.

## 공유 타입 계약

`packages/types`를 단일 소스로 삼아 다음 계약을 추가하거나 확장한다.

### 공지 번역

`AnnouncementTranslation`에 다음 상태를 둔다.

- `lang`: `en | zh | vi`
- `langLabel`
- `title`
- `body`
- `reviewed`
- `sourceHash`: 번역이 생성·검수된 한국어 원문 식별자

### 초안 수정 입력

초안 생성·수정은 다음 필드를 공통으로 받는다.

- `category`
- `scope`
- `targetLabel`
- `targetRoomIds`
- `title`
- `body`
- `translations`

`confirmRequired`는 클라이언트가 임의로 결정하지 않는다. 서버가 카테고리에서 파생하며 긴급은 항상 `true`다.

### 번역 요청·응답

- 요청: `title`, `body`, `targetLang`
- 응답: `lang`, `langLabel`, `title`, `body`, `reviewed: false`, `sourceHash`

## API 설계

### 초안 생성

`POST /manager/messaging/announcement-drafts`

- 관리인 인증을 요구한다.
- 대상 호실이 모두 해당 관리인의 관리 범위인지 확인한다.
- 전체·건물·호실 scope와 `targetRoomIds` 정합성을 검사한다.
- 긴급 여부에 따라 `confirmRequired`를 서버에서 결정한다.

### 초안 수정

`PATCH /manager/messaging/announcement-drafts/:draftId`

- 관리인 인증과 초안 소유권을 검사한다.
- `sent` 초안 수정은 거부한다.
- 생성과 같은 대상 권한·범위 검사를 수행한다.
- 제목 또는 본문이 이전 값과 달라지면 전달된 번역을 포함해 검수 상태를 해제한다.
- 수정된 초안과 갱신 시각을 반환한다.

### 언어별 자동 번역

`POST /manager/messaging/announcement-translations`

- 관리인 인증을 요구한다.
- 빈 제목·본문과 지원하지 않는 언어를 거부한다.
- 기존 `OPENAI_API_KEY`를 서버에서만 사용한다.
- 모델은 `OPENAI_TRANSLATION_MODEL`, `OPENAI_CHAT_MODEL`, `gpt-5.4-mini` 순서로 선택한다.
- OpenAI Responses API에 strict JSON Schema를 전달해 `{ title, body }`만 받는다.
- 프롬프트는 의미·시간·수치·고유명사를 보존하고 내용을 추가하거나 책임을 추정하지 않도록 제한한다.
- 결과를 trim·길이 검증한 뒤 `reviewed: false`와 현재 원문 `sourceHash`를 붙여 반환한다.
- 키 누락, upstream 오류, 거부·불완전 응답, JSON 파싱·스키마 오류는 명확한 503/502 계열 오류로 반환한다.
- 오류 시 초안이나 기존 번역을 변경하지 않는다.

### 검토·발송 게이트

긴급 공지의 검토 및 발송 전에 서버에서 다음 조건을 다시 확인한다.

- `en`, `zh`, `vi`가 각각 정확히 하나 존재한다.
- 제목·본문이 비어 있지 않다.
- `sourceHash`가 현재 한국어 제목·본문으로 계산한 값과 같다.
- 모든 번역의 `reviewed`가 `true`다.

현재처럼 빈 번역 배열이 긴급 발송 검사를 통과하지 않도록 수정한다.

## OpenAI 경계

OpenAI 호출은 브라우저가 아닌 NestJS API 내부에서 수행한다. API 키는 응답·로그·클라이언트 번들에 포함하지 않는다. 기존 Roomlog의 Responses API 호출 방식과 `OpenAI-Safety-Identifier` 패턴을 재사용한다.

구조화 출력은 Responses API의 `text.format`에 `type: json_schema`, `strict: true`, 명시적 schema를 전달한다. 번역은 도구 호출이나 대화 상태가 필요 없는 단발성 요청으로 처리한다.

로컬 Docker API는 기존 `docker-compose.yml`의 `.env` 전달 경로를 그대로 사용한다. compose, workflow, Dockerfile, AWS 설정은 수정하지 않는다. 배포 전 담당자는 배포 API 컨테이너의 `OPENAI_API_KEY` 존재 여부만 확인한다.

## 웹 상태와 데이터 흐름

서버 페이지는 관리인 세션과 선택 초안을 읽고 `AnnouncementComposer` 클라이언트 컴포넌트에 초기값과 `managedRooms`를 전달한다.

클라이언트 컴포넌트는 다음 상태를 관리한다.

- 카테고리, 범위, 선택 호실 ID
- 한국어 제목·본문
- 언어별 번역·검수 상태
- 언어별 번역 로딩·오류
- 저장 중·저장 성공·저장 오류

실제 건물 선택은 `managedRooms`를 `buildingName`으로 그룹화해 해당 그룹의 room ID를 전송한다. 호실 선택은 한 개 이상의 실제 room ID를 전송한다. 전체 선택은 관리 중인 전체 room ID를 전송한다. 사람이 읽는 `targetLabel`은 동일한 선택 데이터에서 파생한다.

새 초안을 처음 저장하면 반환된 ID로 현재 URL을 `router.replace`하고 이후 저장은 PATCH를 사용한다. 검토 버튼은 저장 성공 응답의 ID로 M-MSG-02에 이동한다.

## 오류 처리와 접근성

- 필수 입력과 대상 누락은 API 호출 전에 인라인으로 표시하고 서버에서도 재검증한다.
- 저장 실패는 입력을 유지하고 재시도할 수 있게 한다.
- 번역 실패는 해당 언어에만 표시하고 다른 언어 상태를 보존한다.
- 버튼에는 로딩 텍스트와 `aria-busy`를 제공한다.
- 카테고리·범위는 실제 radio semantics를 유지한다.
- 저장·번역 성공 메시지는 `role=status`, 오류는 `role=alert`로 노출한다.
- 키보드만으로 대상 선택, 번역, 검수, 저장, 검토가 가능해야 한다.

## 비범위

- 실제 공지 발송 채널 자체의 교체
- 생활·행사 공지의 강제 번역
- 지원 언어를 관리자가 임의 추가하는 기능
- 번역 메모리, 용어집, 배치 번역
- OpenAI 키 생성·교체
- compose, CI/CD, AWS, S3, RDS 등 인프라 변경
- Azure Horizon 전역 셸·브랜드 이식

## 승인 기준

- 첨부 시안의 공지 작성 내부 구조가 Roomlog 셸 안에서 재현된다.
- 전체·건물·호실 선택이 실제 관리 호실 ID와 일치한다.
- 임시 저장은 같은 초안을 갱신하고 화면에 머문다.
- 각 언어 버튼은 해당 언어만 번역하며 다른 상태를 보존한다.
- 자동 번역 결과를 수정하고 검수할 수 있다.
- 긴급 공지는 세 언어의 현재 원문 기준 검수 없이는 검토·발송되지 않는다.
- API 장애 시 거짓 성공이나 데이터 유실이 없다.
- 인프라 추적 파일과 기존 미추적 사용자 파일을 변경하지 않는다.
