# 거래 계약·청구 연결 내구성 보강 설계

## 목표

거래에서 수락된 계약이 정확한 호실의 룸로그 계약과 관리자 청구 흐름으로 연결되는 현재 기능을 유지하면서, JSON 저장이나 RDS 프로젝션이 일시적으로 실패해도 재시도·재기동으로 안전하게 복구되게 한다.

이번 작업은 신규 기능이나 화면 개편이 아니라 기존 연결의 내구성을 보강하는 API 작업이다. 현재 관리자 청구서 생성 화면과 세입자 납부 화면이 이미 계약·청구 API를 사용하므로 UI 계약은 변경하지 않는다.

## 범위

- 수락된 매물의 `계약완료` 상태가 Trade 매물 RDS 프로젝션에 반영될 때까지 호출자가 실패를 인지할 수 있게 한다.
- Trade 계약 수락은 먼저 JSON에 확정한 뒤 룸로그 계약 연결과 각 프로젝션을 기다린다.
- 동일한 수락 이벤트를 재시도해도 계약·체결 메시지·룸로그 연결을 중복 생성하지 않는다.
- 서버 재기동 시 JSON의 수락 계약과 RDS 매물 상태가 어긋났다면 최신 수락 상태를 다시 프로젝션한다.
- 관리자가 계약을 확정할 때 룸로그 RDS 프로젝션 완료를 기다리고, 실패하면 같은 확정을 재시도할 수 있게 한다.
- Trade와 Roomlog 프로젝션은 상태 세대(generation)를 구분하여 오래된 완료 콜백이 최신 상태를 완료 처리하지 못하게 한다.
- Trade의 모든 mutation은 JSON 저장 실패 시 마지막으로 저장에 성공한 스냅샷으로 되돌린다.
- 관련 API 테스트와 빌드를 통과시킨다.

## 제외 범위

- 관리자 청구서 작성 화면과 세입자 납부 화면의 UI 변경
- 청구서 자동 확정·자동 공개 정책 변경
- Prisma 스키마, SQL, 마이그레이션 변경
- Trade 계약·채팅 전체를 RDS 테이블로 이전하는 작업
- 공통 RDS 영속성 구조 리팩터링
- 매물·채팅·취소 등 기존 기능의 제품 정책 변경
- `origin/dev`의 OCR 및 다른 작업자가 만든 화면 변경
- 주소·호실 파서의 별도 개선

## 현재 데이터 경계

- Trade 매물은 기존 RDS 프로젝터가 영속화하며, 계약과 채팅은 기존 JSON 저장을 유지한다.
- Roomlog 계약·청구 상태는 기존 Roomlog 스토어와 RDS 프로젝터를 그대로 사용한다.
- 새 컬럼이나 테이블을 추가하지 않고 현재의 매물 상태, 계약 필드, 추출 데이터만 사용한다.
- 프로젝션 실패는 데이터 모델 문제가 아니라 비동기 저장 완료를 기다리지 않거나 실패 세대를 구분하지 않는 제어 흐름 문제로 다룬다.

## 처리 흐름

### 거래 계약 수락

1. 임차인이 제안된 계약을 수락한다.
2. Trade는 계약 상태, `respondedAt`, 매물의 `계약완료` 상태, 체결 메시지를 한 mutation으로 구성한다.
3. Trade JSON 저장에 성공하면 해당 스냅샷을 마지막 확정 스냅샷으로 갱신한다.
4. 매물 RDS 프로젝션을 큐에 등록하고 해당 generation의 완료를 기다린다.
5. 룸로그 브리지는 동일한 수락 식별자로 정확한 호실의 계약 연결을 보장하고 Roomlog 프로젝션 완료를 기다린다.
6. 모든 필수 저장이 끝난 뒤 성공을 반환하고 알림을 보낸다.

Trade JSON 저장 이후 RDS 또는 Roomlog 프로젝션이 실패하면 수락 자체는 이미 확정된 것으로 유지한다. 호출자는 실패를 받고 같은 요청을 재시도할 수 있으며, 재시도는 기존 `respondedAt`과 체결 메시지를 보존한 채 미완료 프로젝션만 다시 수행한다.

### 서버 재기동 복구

1. Trade는 JSON의 계약·채팅 상태를 읽고 RDS에서 매물 목록을 로드한다.
2. 수락된 계약이 가리키는 매물이 RDS에서 아직 `노출중`이면 메모리의 매물 상태를 `계약완료`로 보정한다.
3. 보정된 최신 매물 스냅샷을 RDS에 다시 프로젝션하고 완료를 기다릴 수 있게 한다.
4. 룸로그 브리지는 수락된 계약을 멱등하게 다시 연결하고 Roomlog의 최신 generation 프로젝션을 기다린다.

### 관리자 계약 확정

1. 관리자가 기존 계약 검토 API로 계약 값을 확정한다.
2. Roomlog는 현재 서비스 로직으로 계약을 `confirmed` 상태로 변경한다.
3. 확정된 스토어 스냅샷의 RDS 프로젝션이 끝날 때까지 컨트롤러가 기다린다.
4. 실패하면 오류를 반환한다. 같은 확정 요청은 이미 확정된 계약을 중복 생성하지 않고 최신 상태를 다시 프로젝션한다.
5. 성공 후 기존 청구서 생성 옵션 API가 확정 계약을 그대로 제공한다.

## 프로젝션 generation 규칙

- mutation 또는 복구로 새 프로젝션을 예약할 때마다 증가하는 generation을 부여한다.
- 각 비동기 작업은 자신이 맡은 스냅샷과 generation만 기록한다.
- 완료·실패 콜백은 자신이 최신 generation일 때만 전역 완료·오류 상태를 갱신한다.
- 호출자는 자신이 요구한 generation 이상이 성공했는지 확인하고, 오래된 성공으로 최신 변경을 완료 처리하지 않는다.
- 실패 후 같은 상태를 재시도하면 새 generation으로 다시 예약한다.

## Trade mutation 롤백 규칙

- 서비스 초기화와 DB 매물 hydrate가 끝난 시점에 `committedStore`를 설정한다.
- `persist()`는 현재 스토어의 깊은 복사본을 JSON에 저장한다.
- 저장 성공 시 그 복사본을 새 `committedStore`로 교체한다.
- 파일 저장을 사용하지 않는 환경에서는 현재 동작을 유지하기 위해 mutation을 즉시 확정 스냅샷으로 기록한다.
- 저장 실패 시 공통 `persist()` 경계에서 전체 Trade 스토어를 `committedStore`로 복구하고 원래 오류를 다시 던진다.
- 매물 CRUD, 문의·메시지, 계약 제안·거절·취소·수락 모두 같은 규칙을 사용한다.
- mutation별 수동 부분 롤백은 공통 스냅샷 롤백으로 대체해 새 mutation이 추가돼도 동일한 원자성을 갖게 한다.

## 오류 및 멱등성

- JSON 저장 실패: 해당 Trade mutation 전체를 이전 확정 상태로 되돌리고 오류를 반환한다.
- Trade 매물 RDS 프로젝션 실패: JSON의 수락 상태는 유지하고 오류를 반환하며 재시도·재기동으로 매물 상태를 복구한다.
- Roomlog 프로젝션 실패: 메모리/JSON의 연결 또는 확정 상태는 유지하고 오류를 반환하며 같은 작업으로 최신 스냅샷을 재프로젝션한다.
- 재시도는 계약 ID, 수락 시각, 체결 메시지, 룸로그 계약 ID를 재사용한다.
- 성공 알림은 모든 필수 저장이 완료된 요청에서 한 번만 보낸다.

## 커밋 구성

최종 PR은 다음 두 커밋만 남긴다.

1. `fix(contract): harden billing durability`
   - 수락 매물 상태 await·재시도·재기동 복구
   - 관리자 확정 Roomlog 프로젝션 await·재시도
   - Trade·Roomlog generation 기반 프로젝션 완료 판정
   - 이 설계 문서와 구현 계획
2. `fix(trade): roll back failed store writes`
   - Trade 전체 mutation의 공통 확정 스냅샷 롤백
   - 관련 회귀 테스트

## 테스트 전략

구현은 실패 테스트를 먼저 추가하는 TDD 순서로 진행한다.

- Trade 수락 후 첫 매물 프로젝션 실패가 호출자에게 전달되는지 검증한다.
- 같은 수락을 재시도하면 기존 수락 시각과 메시지를 유지하고 프로젝션만 성공하는지 검증한다.
- 재기동 시 수락 계약과 `노출중` RDS 매물의 불일치를 `계약완료`로 복구하는지 검증한다.
- 연속된 프로젝션에서 오래된 generation 완료가 최신 상태를 완료 처리하지 않는지 검증한다.
- 관리자 계약 확정의 첫 Roomlog 프로젝션 실패와 같은 요청의 성공 재시도를 검증한다.
- 매물 CRUD, 문의·메시지, 계약 제안·거절·취소·수락 각각에서 JSON 저장 실패 시 전체 스토어가 이전 확정 상태로 복구되는지 검증한다.
- 관련 API 테스트, API 빌드, `bash scripts/verify.sh`를 실행한다. Docker를 사용할 수 없는 현재 환경에서는 Docker 실동작 검증을 생략하고 그 사실을 보고한다.

## 병합 방식

- 최신 `origin/dev`에서 분기한 작업 브랜치에서 구현한다.
- `origin/dev`의 UI와 OCR 변경을 기준으로 유지한다.
- 완료 후 `yong`에 푸시하고 `dev` 대상 PR 하나를 Draft가 아닌 Ready 상태로 생성한다.
