# 관리인 AI 다건 독촉 및 공지 대상 후속 해석 설계

## 배경

관리인 AI 비서의 음성 흐름에는 두 가지 구조적 문제가 있다.

1. `billing.send_dunning`이 청구서 한 건만 준비할 수 있어 “두 개 전부 독촉 문자 보내줘”가 한 세대만 선택된다.
2. 공지 대상이 같은 호실을 가진 여러 건물과 일치하면 실제 수신 가능한 계약 세대가 하나뿐이어도 모호하다고 판단한다. 이후 “뒤에 거”, “플로우테스트3”처럼 후보를 좁히는 발화도 앞선 후보 상태와 결합되지 않는다.

건물명이 서로 비슷한 것은 모호성을 키우지만 근본 원인은 아니다. 실제 제품에서도 비슷한 이름과 동일 호실은 존재할 수 있으므로 건물명 변경이나 유사도 임계값 완화로 해결하지 않는다.

## 목표

- “전부”, “모두”, “두 개 다”가 포함된 독촉 요청은 현재 조회 맥락의 발송 가능한 청구 전체를 하나의 보류 작업으로 준비한다.
- 관리인은 대상과 금액을 한 번 확인하고 `승인` 또는 `진행해` 한 번으로 전체 발송한다.
- 실제 발송 직전 청구별 미납·입금·가드 상태를 다시 확인한다.
- 공지 대상은 실제 수신 가능한 계약 세대를 우선해 불필요한 질문을 없앤다.
- 정말 여러 후보가 남을 때는 후보를 구조적으로 보존하고 후속 발화로 선택할 수 있게 한다.
- 음성과 텍스트가 동일한 서버 해석 및 확인 경로를 사용한다.
- 같은 서버 결과가 AI 음성과 UI 말풍선에 중복 표시되지 않게 한다.
- 미납과 연체를 사실에 맞게 구분한다.

## 비목표

- 비슷한 건물명을 강제로 변경하거나 관리자에게 이름 변경을 요구하지 않는다.
- 유사도 기준을 낮춰 애매한 대상을 임의 선택하지 않는다.
- SMS 등 별도 독촉 채널을 추가하지 않는다. 기존 계약 관계의 일반 소통 채팅을 사용한다.
- 여러 종류의 발송을 한 보류 작업에 섞지 않는다. 독촉 묶음과 공지 한 건은 각각 별도 작업이다.

## 설계

### 1. 다건 독촉

단건 청구서를 반환하는 현재 해석기를 “독촉 대상 집합”을 반환하도록 확장한다.

- 명시적인 청구 ID가 있으면 단건으로 유지한다.
- 호실·청구월·건물명이 있으면 해당 조건에 맞는 청구만 선택한다.
- `전체`, `전부`, `모두`, `두 개 다`, `전부 다` 같은 집합 표현이 있으면 현재 요청에서 확인 가능한 발송 가능 청구를 모두 선택한다.
- 집합 표현이 없고 후보가 여러 건이면 기존처럼 후보를 보여주고 추가 지정을 요구한다.
- 준비 결과에는 청구 ID, 건물명, 호실, 세입자명, 청구월, 미납액, 납부기한, 연체일수, 발송 문구를 건별로 포함한다.

하나의 DB `AgentToolAction`에 선택된 청구 ID 배열과 준비 시점의 표시용 미리보기를 저장한다. 확인 시 배열의 각 청구를 다시 조회해 가드를 검사하고, 통과한 청구만 기존 일반 소통 채팅으로 발송한다.

결과는 다음 세 상태를 구분한다.

- 전체 성공: 발송 세대 수와 총액을 알린다.
- 일부 성공: 성공 세대와 실패 세대를 분리하고 실패 사유를 건별로 알린다.
- 전체 실패: 아무 메시지도 보내지 않고 최신 차단 사유를 알린다.

한 청구의 실패가 다른 정상 청구 발송을 롤백하지는 않는다. 대신 같은 보류 작업을 다시 확인해 중복 발송할 수 없도록 작업 자체는 한 번만 실행한다.

### 2. 공지 대상 선택 우선순위

공지 대상 후보는 관리 중인 모든 호실이 아니라 실제 공지 수신자가 있는 호실을 우선한다.

1. 관리인 권한 범위 안의 호실만 후보로 만든다.
2. 계약 중이며 공지를 받을 세입자가 연결된 호실을 우선 후보로 만든다.
3. 같은 호실 번호에서 수신 가능한 후보가 하나면 건물명이 비슷하거나 생략돼도 자동 선택한다.
4. 수신 가능한 후보가 여러 개면 호실 번호, 건물명 유사도, 직전 후보 선택 맥락 순서로 판별한다.
5. 충분한 점수 차이가 없으면 임의 선택하지 않고 후보를 요청한다.

공실이나 수신자가 없는 호실은 실제 수신 가능한 후보가 하나 이상 있을 때 모호성 계산에서 제외한다. 수신 가능한 후보가 전혀 없을 때만 “공지 수신 세대가 없습니다”로 차단한다.

### 3. 후속 후보 선택

모호한 대상은 일반 오류 문장으로 끝내지 않고 구조화된 선택 상태로 저장한다.

선택 상태에는 다음 정보가 필요하다.

- 선택 상태 ID와 만료 시각
- 원래 명령 종류
- 공지 제목과 본문
- 후보별 불투명 ID
- 화면 표시용 건물명, 호실, 주소 요약, 세입자명

후속 발화는 다음 형식을 지원한다.

- 정확하거나 일부인 건물명
- `첫 번째`, `두 번째`, `뒤에 거`, `앞에 거`
- 후보에 표시된 세입자명
- 후보에 표시된 주소 일부

선택이 끝나면 저장된 제목·본문과 선택한 불투명 후보 ID로 기존 확인 작업을 준비한다. 모델이 이전 `target="103호"`를 다시 만들어도 서버가 저장된 후보 선택 상태를 우선한다.

후보 상태는 관리인별로 하나만 활성화하고 짧은 시간 후 만료한다. 다른 관리인의 후보를 참조할 수 없으며 원시 호실 ID는 클라이언트에 노출하지 않는다.

### 4. 음성 및 텍스트 통합

음성과 텍스트는 동일한 서버 명령 입력을 사용한다.

- Realtime 전사 문장을 `text`에만 덧붙이는 데 그치지 않고 대상 해석 입력으로 사용한다.
- 모델이 보낸 `target`과 최신 사용자 전사, 저장된 후보 선택 상태를 서버에서 함께 해석한다.
- 다건 독촉과 공지 모두 서버가 보류 작업을 만든 후 `승인` 또는 `진행해` 한 번으로 확정한다.
- UI 버튼은 요구하지 않는다.

### 5. 중복 답변 제거

Realtime 도구 호출의 결과를 두 군데에서 동시에 대화 기록에 추가하지 않는다.

- 조회 명령: Realtime 모델이 서버 도구 결과를 자연어로 말하고, 프론트는 같은 요약을 별도 말풍선으로 추가하지 않는다.
- 보류·확정 명령: 프론트가 영수증과 상태를 기록하고, Realtime 모델에는 짧은 상태 값만 돌려 중복 문장 생성을 막는다.
- 같은 실행 ID의 영수증은 한 번만 렌더링한다.

### 6. 미납과 연체 표현

`미납`과 `연체`는 분리한다.

- 미납액이 있으나 납부기한이 지나지 않았거나 연체일수가 0이면 `미납`으로 표시한다.
- 납부기한이 지났고 연체일수가 1일 이상이면 `연체`로 표시한다.
- 수납 요약은 미납 건수와 연체 건수를 별도로 제공한다.
- 화면 이동 URL을 답변 본문에 출력하지 않고 기존 navigation 메타데이터만 사용한다.

## 데이터 흐름

### 다건 독촉

1. 사용자: “두 개 전부 다 독촉 문자 보내줘.”
2. 음성/텍스트 입력이 동일한 코파일럿 API로 전달된다.
3. 서버가 집합 표현과 최신 청구 상태를 기준으로 대상 청구 배열을 만든다.
4. 서버가 다건 미리보기와 명령 payload를 `AgentToolAction`에 저장한다.
5. AI가 대상별 호실·금액과 총 건수를 짧게 보여주고 승인을 요청한다.
6. 사용자: “진행해.”
7. 서버가 각 청구를 재검증하고 발송 가능한 세대에 메시지를 보낸다.
8. 성공·실패 결과를 건별로 반환하고 하나의 실행 영수증을 표시한다.

### 모호한 공지

1. 사용자: “103호에 에어컨 교체 공지 보내.”
2. 수신 가능한 103호가 하나면 즉시 그 세대로 초안을 준비한다.
3. 두 개 이상이면 서버가 후보 선택 상태를 만들고 건물명·세입자명·주소 요약을 보여준다.
4. 사용자: “뒤에 거.”
5. 서버가 두 번째 후보를 선택해 저장된 제목·본문으로 공지 확인 작업을 준비한다.
6. 사용자: “진행해.”
7. 서버가 대상 계약과 수신자를 재확인한 뒤 공지를 발송한다.

## 오류 및 안전 처리

- 대상 후보가 없으면 관리 범위 안에서 찾지 못했다는 사실만 알린다.
- 후보가 여러 개면 낮은 유사도 차이로 임의 선택하지 않는다.
- 확인 전에 계약이 종료되거나 수신자가 사라지면 발송을 차단한다.
- 다건 독촉의 각 청구는 독립적으로 가드를 통과해야 한다.
- 이미 납부됐거나 입금 확인 중인 청구에는 독촉을 보내지 않는다.
- 확인 작업과 후보 선택 상태는 관리인 권한으로 격리하고 만료시킨다.
- 같은 확인 ID 재요청은 기존 실행 결과를 반환하며 재발송하지 않는다.

## 테스트

### 대상 해석

- 동일한 103호 중 계약·수신 가능한 후보가 하나면 자동 선택한다.
- 두 후보 모두 계약 중이면 유사한 건물명을 임의 선택하지 않는다.
- 음성 오인식된 건물명과 정확한 건물명이 충분히 구분되면 올바른 후보를 선택한다.
- `뒤에 거`, `두 번째`, 세입자명, 주소 일부로 저장된 후보를 선택한다.
- 다른 관리인의 후보는 선택할 수 없다.

### 다건 독촉

- “두 개 전부”, “모두”, “전체”가 발송 가능한 청구 배열을 만든다.
- 단건 요청은 기존 단건 동작을 유지한다.
- 한 번 승인하면 각 대상에 정확히 한 메시지만 발송한다.
- 한 청구가 확인 중이면 나머지만 발송하고 일부 성공 결과를 반환한다.
- 전부 차단되면 메시지를 하나도 만들지 않는다.
- 재확인 요청이 중복 발송을 만들지 않는다.

### 음성 및 UI

- 첫 음성 요청과 후속 대상 선택이 같은 서버 후보 상태를 사용한다.
- 같은 서버 요약이 AI 말풍선과 실행 영수증에 중복되지 않는다.
- 미납 0일은 연체로 표시되지 않는다.
- 답변 문장에 내부 URL이 노출되지 않는다.

## 완료 기준

- 제공된 재현 흐름에서 “두 개 전부”가 102호와 103호를 한 번의 승인으로 처리한다.
- 실제 수신 가능한 103호가 하나면 건물명이 비슷해도 추가 질문 없이 공지 초안을 준비한다.
- 후보가 정말 둘이면 “뒤에 거”로 선택할 수 있다.
- 공지와 독촉 모두 텍스트 `승인` 또는 `진행해` 한 번으로 실행된다.
- 음성 대화에서 동일 내용이 두 번 출력되지 않는다.
- 기본 검증 스크립트와 관련 API·웹 테스트가 통과한다.
