# Manager Credit Widget Design

## Goal

관리자 크레딧 충전이 기존 Toss 결제위젯 키(`test_gck_*`)와 API 개별 결제창 키(`test_ck_*`)를 모두 지원하도록 한다.

## User flow

1. 관리자는 공통 헤더의 `충전` 버튼을 눌러 금액을 선택한다.
2. `결제수단 불러오기`를 누르면 서버가 `READY` 충전 주문을 만든다.
3. 결제위젯 키라면 같은 팝업 안에 결제수단과 약관을 렌더한다.
4. 관리자가 `N원 결제하기`를 누르면 기존 성공·실패 콜백으로 이동한다.
5. API 개별 결제창 키라면 기존처럼 주문 생성 직후 결제창을 연다.

## Failure handling

- SDK·위젯 렌더링·결제 요청이 실패하면 화면에 원문 오류를 표시한다.
- 결제 요청 전 생성된 `READY` 주문은 취소 API를 호출해 고아 주문으로 남기지 않는다.
- 관리자가 위젯 단계에서 팝업을 닫거나 금액 선택으로 돌아가도 해당 주문을 취소한다.
- 실제 승인 이후 잔액 반영, 성공·실패 콜백, 원장 처리는 기존 API를 그대로 사용한다.

## Scope

- 관리자 공통 크레딧 충전 팝업과 Toss 웹 유틸리티만 수정한다.
- API 스키마·DB 마이그레이션·세입자 결제 흐름은 변경하지 않는다.
- 기존 결제창 키 호환성을 유지한다.

## Verification

- 위젯 키가 `widgets.requestPayment`를 호출하고 결제창 API를 호출하지 않는 단위 테스트
- 결제창 키가 기존 `payment().requestPayment`를 호출하는 단위 테스트
- 관리자 팝업의 위젯 컨테이너·취소 API·2단계 문구 계약 테스트
- Web 전체 테스트와 빌드
- Docker 재빌드 후 실제 Toss 결제위젯의 결제수단·약관 표시 확인
