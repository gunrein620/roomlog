# 관리자 하자관리 더미 행 표시 설계

## 목표

`/manager/ticket/dash/00`에서 실제 티켓 데이터와 별도로 화면 확인용 더미 데이터 10개를 항상 함께 표시한다. 더미 데이터는 실제 데이터와 명확히 구분하고, 관리자 사이드바·공용 API·DB·다른 화면에는 영향을 주지 않는다.

## 적용 방식

- 하자관리 대시보드 전용 더미 행 모듈을 라우트 폴더 안에 둔다.
- 서버 페이지는 실제 티켓과 수리 정보를 먼저 행으로 변환한 뒤 더미 행 10개를 뒤에 합친다.
- 실제 데이터가 0건이어도 더미 10개가 표시되며, 실제 데이터가 있으면 `실제 행 + 더미 10개`가 표시된다.
- 공용 `MANAGER_DEMO_TICKETS`, API 응답, DB 시드 데이터는 변경하지 않는다.

## 더미 데이터 구성

- 더미 행은 정확히 10개이며 각 티켓 ID는 대시보드 안에서 고유하다.
- 모든 작업명은 `더미 · `로 시작한다.
- 하자 민원과 일반 민원을 섞고 대기·진행중·완료·취소 상태를 골고루 포함한다.
- 건물명, 호실, 작업자, 예정일시, 청구 금액을 채워 테이블과 필터 동작을 확인할 수 있게 한다.
- 일부 행은 미배정 작업자나 미정 금액을 사용해 누락 정보 표현도 확인할 수 있게 한다.

## 화면 동작

- 상태별 건수, 담당자 필터, 건물 필터, 템플릿 필터, 페이지네이션은 실제 행과 더미 행을 합친 전체 목록을 기준으로 동작한다.
- 더미 건물명을 표에 표시하고 건물 선택창에서도 선택할 수 있게 한다.
- 실제 행의 작업명과 작업 버튼은 기존 상세 화면 링크를 유지한다.
- 더미 행은 미리보기 데이터이므로 작업명과 작업 버튼을 비활성 상태로 표시해 존재하지 않는 상세 티켓으로 이동하지 않게 한다.
- 사이드바와 민원·하자 하위 메뉴는 변경하지 않는다.

## 오류 및 격리

- 더미 데이터는 정적 TypeScript 객체이므로 API 실패와 무관하게 렌더된다.
- 더미 데이터 추가를 위해 mutation, DB 기록, 인프라 설정 변경을 수행하지 않는다.
- 실제 API 데이터의 원본 객체는 변경하지 않고 새 배열에서 더미 행과 병합한다.

## 검증

- 더미 행이 정확히 10개인지 테스트한다.
- 모든 더미 작업명이 `더미 · ` 접두어를 가지는지 테스트한다.
- 상태·건물·담당자·템플릿 필터와 페이지네이션이 병합된 목록에서 동작하는지 테스트한다.
- 더미 행의 상세 작업이 링크로 렌더되지 않는지 소스 계약 테스트로 확인한다.
- 기존 사이드바 파일 해시 테스트를 유지한다.
- Next.js 프로덕션 빌드와 Docker web 재빌드 후 실제 브라우저에서 10개 행, 필터, 오류 오버레이 유무를 확인한다.

## 범위 제외

- DB 시드 또는 API 응답에 더미 데이터를 저장하지 않는다.
- 공용 관리자 데모 티켓 수를 변경하지 않는다.
- 더미 행 전용 상세 페이지를 만들지 않는다.
- 사이드바 및 다른 관리자 도메인 화면을 수정하지 않는다.
