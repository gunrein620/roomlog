# 관리자 업체 목록 동작·상태 가독성 설계

## 목표

관리자가 `내 업체` 목록에서 업체 상세 진입과 업체 해제를 바로 인지하고 실행할 수 있게 한다. 업체 찾기에서는 검증·계정 상태와 배정 차단 사유를 서로 붙이지 않고 독립된 정보로 보여준다.

이 작업은 기존 업체 원장, 업체 계정, 작업·결제 이력을 삭제하지 않는다. 관리자의 업체 연결만 `ARCHIVED`로 전환하는 기존 API를 재사용한다.

## 확정 범위

### 내 업체 목록

- `관리 상태` 열을 `관리`로 바꾼다.
- 각 행에 `상세 보기`와 `해제`를 명시해 업체명이 링크인지 추측하지 않아도 되게 한다.
- 업체명 링크는 기존 상세 경로를 유지하고 링크임을 알 수 있는 시각적 단서를 추가한다.
- `해제`는 목록에서 직접 실행한다.
- 해제 전 확인 다이얼로그에서 다음 내용을 알린다.
  - 업체 원장과 과거 작업·결제 이력은 유지된다.
  - 신규 작업 배정 후보에서만 제외된다.
  - 업체 찾기에서 다시 등록할 수 있다.
- 성공하면 서버 액션의 기존 revalidation을 이용해 해당 행을 목록에서 제거한다.
- 실패하면 같은 행 또는 확인 다이얼로그 안에서 오류를 보여주고 목록 상태는 유지한다.
- `진행 n건`과 `결제 대기 n건`은 한 줄씩 고정해 표시한다.

### 업체 찾기

- `검증 상태`와 `계정 상태`를 각각 독립된 줄 또는 배지로 표시한다.
- 배정 가능 상태는 하나의 긍정 배지로 유지한다.
- 배정 불가일 때 `운영 검증 필요`, `운영 중지 업체`, `업체 계정 미연결`, `내 업체 등록 필요`를 하나의 문장으로 합치지 않고 사유별 배지로 나눈다.
- 여러 사유는 일정한 간격으로 줄바꿈되며 서로 맞붙지 않는다.

## 선택한 접근

`내 업체` 표의 관리 열에 명시적 동작을 둔다.

- 행 전체 클릭은 내부 `해제` 버튼과 포인터·키보드 동작이 충돌할 수 있어 사용하지 않는다.
- 점 메뉴는 해제를 다시 숨기므로 사용하지 않는다.
- `상세 보기`와 `해제`를 직접 노출해 발견 가능성과 접근성을 우선한다.

## 컴포넌트 경계

- 업체 표의 데이터 렌더링은 기존 `ManagerVendorTable`을 유지한다.
- 업체별 상세·해제 동작을 전달할 수 있도록 관리 셀을 확장한다.
- 해제 확인과 서버 액션 상태는 전용 클라이언트 컨트롤에 격리한다.
- 기존 `archiveVendorAction`과 `DELETE /manager/vendor-mgmt/vendors/:vendorId/registration`을 재사용한다.
- API, 공유 타입, Prisma 스키마는 변경하지 않는다.

## 디자인 원칙

- 모든 색상·간격·테두리는 `packages/ui/src/tokens.css`의 토큰만 사용한다.
- 상세는 중립 버튼, 해제는 위험도가 드러나는 보조 위험 버튼으로 구분한다.
- 클릭 가능한 요소는 hover와 focus-visible 상태를 제공한다.
- 확인 다이얼로그는 현재 관리자 화면 위에 떠도 레이아웃을 밀어내지 않는다.
- 상태 묶음은 `display: grid` 또는 세로 flex와 명시적 gap을 사용하며 인접 형제 margin에 의존하지 않는다.

## 테스트

- 내 업체 표에 `상세 보기`, `해제`가 행마다 존재하는지 검증한다.
- 해제 확인 전에는 서버 액션이 호출되지 않는지 검증한다.
- 성공·실패 피드백과 목록 revalidation 경로를 검증한다.
- 진행·결제 대기와 검증·계정 상태가 독립된 블록으로 렌더되는지 소스 계약을 검증한다.
- 배정 차단 사유가 결합 문자열이 아니라 개별 배지로 렌더되는지 검증한다.
- 기존 업체 등록·상세·해제 API 테스트를 회귀 실행한다.

## 후속 작업

업체 목록 보완은 별도 커밋으로 완료한 뒤 `수리비 결제 마일스톤 설계`의 M5를 진행한다. M5에서는 관리자·세입자 화면의 결제 상태와 복구 동작을 공통 규칙으로 맞추고, 최신 주문 조회·재결제·취소·재조회가 새로고침 후에도 유지되도록 한다.
