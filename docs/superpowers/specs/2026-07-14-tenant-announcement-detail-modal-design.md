# 세입자 공지 상세 모달 설계

## 목표

`/tenant/messaging/02` 공지 목록에서 각 공지 카드를 누르면 페이지를 이동하지 않고 상세 모달을 열어 전체 내용을 확인할 수 있게 한다. 모달 동작은 관리자 민원 목록의 `TicketDetailDialog` 패턴과 일관되게 구성한다.

## 사용자 흐름

1. 세입자가 공지 카드 전체 영역을 누른다.
2. 현재 목록, 검색어, 필터와 스크롤 위치를 유지한 채 상세 모달이 열린다.
3. 모달에서 공지 카테고리, 대상 범위, 제목, 발신 관리인, 발송일시, 전체 본문을 확인한다.
4. 긴급 공지는 확인 필요 안내와 `확인` 동작을 제공하고, 일반 공지는 `읽음` 동작을 제공한다.
5. `이 공지 문의`를 선택하면 해당 공지를 문맥으로 사용하는 기존 세입자 문의 대화 흐름으로 이동한다.
6. 닫기 버튼, Escape 키 또는 모달 바깥 배경 클릭으로 목록에 돌아오며, 모달을 열었던 카드로 키보드 포커스가 복귀한다.

## 컴포넌트 구조

- `AnnouncementListPage`를 클라이언트 컴포넌트로 전환해 선택된 공지와 모달 열림 상태를 관리한다.
- 공지 카드는 상세 라우트 `Link` 대신 전체 영역을 차지하는 버튼을 사용한다.
- 새 `AnnouncementDetailDialog` 컴포넌트는 선택된 `Announcement | null`과 `onClose`를 입력으로 받는다.
- 모달은 관리자 `TicketDetailDialog`와 같은 네이티브 `<dialog>` 및 `showModal()` 패턴을 사용한다.
- 기존 `/tenant/messaging/02/[id]` 상세 페이지는 외부 링크나 직접 접근 호환을 위해 유지한다.
- 목록 URL은 모달을 열고 닫아도 `/tenant/messaging/02` 그대로 유지한다.

## 모달 내용과 시각 구조

- 헤더: 카테고리 배지, 대상 범위 배지, 제목, 닫기 버튼
- 메타 정보: 발신 관리인과 발송일시
- 본문: 줄바꿈을 보존한 전체 공지 내용
- 번역 공지: `originalBody`가 있으면 별도 원문 영역 표시
- 긴급 공지: 확인 필요 안내, 안전 CTA, 현재 확인 상태 표시
- 하단 동작: 긴급 공지는 `확인`, 일반 공지는 `읽음`, 공통으로 `이 공지 문의`

스타일은 `AnnouncementListPage.module.css`의 기존 토큰만 사용한다. 데스크톱에서는 최대 너비가 제한된 중앙 모달로 표시하고, 모바일에서는 화면 가장자리 여백을 줄이며 본문만 스크롤되게 한다. raw hex 색상은 추가하지 않는다.

## 상태 변경과 데이터 흐름

- 모달은 목록 페이지가 이미 받은 계정별 `Announcement` 객체를 사용하므로 열 때 추가 상세 조회를 하지 않는다.
- `읽음`과 `확인`은 인증된 `/api/tenant/messaging/announcements/:id/read|confirm` 경로를 호출한다.
- 성공하면 선택 공지와 목록의 해당 공지 상태를 함께 갱신해 배지가 즉시 반영되게 한다.
- 실패하면 모달을 유지하고 동작 영역에 오류 문구를 표시하며 재시도할 수 있게 한다.
- 문의는 기존 공지 문맥의 세입자 메시지 스레드 생성 API를 사용하고, 생성 성공 후 해당 대화로 이동한다.
- 로딩 중에는 중복 요청을 막고 해당 동작 버튼을 비활성화한다.

## 접근성 및 닫기 동작

- `<dialog aria-labelledby>`로 제목을 연결하고 브라우저의 모달 포커스 트랩을 사용한다.
- 카드 버튼에는 `${공지 제목} 공지 자세히 보기` 접근성 이름을 유지한다.
- 닫기 버튼은 명확한 접근성 이름을 제공한다.
- Escape와 배경 클릭을 지원하고 닫힌 뒤 원래 카드 버튼으로 포커스를 돌려준다.
- 모달 내부의 읽음, 확인, 문의, 닫기 버튼에 공유 `focus-ring` 토큰을 적용한다.
- `prefers-reduced-motion`에서는 모달 및 카드 전환 애니메이션을 제거한다.

## 반응형 동작

- 데스크톱: 목록 위에 반투명 배경과 중앙 정렬된 상세 모달을 표시한다.
- 태블릿: 모달 너비를 뷰포트에 맞춰 줄이고 내부 본문 높이를 제한한다.
- 모바일: 안전 여백을 포함해 화면 대부분을 사용하며 헤더와 하단 동작은 유지하고 본문만 세로 스크롤한다.

## 테스트 및 검증

1. 공지 목록 계약 테스트에서 카드가 상세 라우트 링크가 아닌 모달 트리거임을 먼저 실패 테스트로 고정한다.
2. `AnnouncementDetailDialog`가 선택 공지의 전체 본문과 메타 정보를 표시하는 단위 계약을 추가한다.
3. 닫기 버튼, Escape, 배경 클릭, 포커스 복귀 계약을 검증한다.
4. 읽음·확인 성공 시 목록 배지가 갱신되고 실패 시 오류가 표시되는 상태 계약을 검증한다.
5. 공지 문의가 기존 공지 문맥 스레드 생성 흐름을 유지하는지 검증한다.
6. web 집중 테스트와 전체 테스트를 실행한다.
7. Docker web 이미지를 다시 빌드한 뒤 데스크톱과 모바일 뷰포트에서 카드 클릭, 모달 표시, 닫기, 읽음·확인, 문의 동작을 실제 브라우저로 검증한다.

## 비범위

- 관리자 공지 작성 및 발송 흐름 변경
- 계정별 공지 수신자 결정 로직 변경
- 기존 `/tenant/messaging/02/[id]` 상세 페이지 제거
- 공지 검색 및 필터 동작 변경
- 공지 API 실패 시 데모 데이터 폴백 제거
- 인프라 및 배포 설정 변경
