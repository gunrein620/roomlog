# Manager Messaging Building Filter Design

## Goal

관리자 메시징 허브의 `채팅`, `공지` 탭을 제거하고 같은 위치에 건물 선택창을 배치한다. 관리자는 전체 티켓 또는 선택한 건물의 티켓만 볼 수 있다.

## Scope

- `/manager/messaging/00`의 채팅·공지 탭을 제거한다.
- 공지 초안·발송 이력 분기를 이 화면에서 제거한다.
- 기존 공지 작성 화면과 사이드바 `공지 작성` 링크는 유지한다.
- 티켓 카드, 답장 필요 계산, 열기·삭제 기능은 유지한다.
- 인프라 파일과 다른 관리자 화면은 수정하지 않는다.

## Interaction

기본 선택은 `전체 건물`이다. 선택창 옵션은 현재 조회된 티켓의 건물명을 중복 제거하고 정렬해 만든다.

- `전체 건물`: 모든 티켓 표시
- 각 건물명: 해당 `thread.buildingName`과 정확히 일치하는 티켓만 표시
- `건물 미지정`: `buildingName`이 없거나 공백인 기존 티켓만 표시

선택이 바뀌면 클라이언트 필터 컴포넌트가 현재 경로의 `building` 검색 매개변수를 갱신한다. `전체 건물`은 검색 매개변수를 제거한다. `router.replace`를 사용해 불필요한 브라우저 히스토리 누적을 피하면서 새로고침과 URL 공유 시 필터를 유지한다.

URL의 `building` 값이 현재 옵션에 없으면 서버가 `전체 건물`로 처리한다.

## Data Flow

1. 서버 페이지가 `listManagerThreads()`로 티켓을 조회한다.
2. `buildingName`을 정규화해 건물 옵션을 만든다.
3. URL의 `building`을 허용된 옵션과 비교해 활성 필터를 결정한다.
4. 선택된 건물로 티켓을 필터링한다.
5. 필터된 티켓을 기존 규칙인 `답장 필요 우선 → 최신순`으로 정렬한다.
6. 클라이언트 선택창은 활성 필터와 옵션을 받아 URL만 갱신한다.

공지 초안과 발송 결과는 더 이상 이 페이지에서 표시하지 않으므로 `listAnnouncementDrafts()`와 `listAnnouncementResults()` 호출도 제거한다.

## Presentation

기존 탭 위치에 레이블과 native select로 구성된 건물 선택창을 둔다. 공용 디자인 토큰만 사용하며 최소 터치 높이를 유지한다.

선택창 아래 섹션 제목은 `건물별 · 답장 필요 상단`으로 변경한다. 선택 결과가 없으면 `선택한 건물의 티켓이 없습니다.` 빈 상태를 표시한다.

## Testing

1. 순수 건물 옵션·선택·필터 함수의 실패 테스트를 작성한다.
2. 페이지 소스 계약 테스트에 탭과 공지 조회 제거, 건물 선택창 사용, 빈 상태를 추가한다.
3. 최소 구현 후 집중 테스트를 통과시킨다.
4. 전체 web 단위 테스트와 `scripts/verify.sh`를 실행한다.
5. 커밋·푸시 후 실행 중인 Docker 데몬에서 web 이미지를 재빌드한다.
6. 로컬 `/manager/messaging/00`을 열어 응답을 확인한다.

## Success Criteria

- `채팅`, `공지` 탭이 보이지 않는다.
- 같은 위치에 `전체 건물`을 기본값으로 한 건물 선택창이 보인다.
- 건물을 선택하면 해당 건물 티켓만 표시되고 선택이 URL에 유지된다.
- 미지정 데이터와 잘못된 URL 값이 안전하게 처리된다.
- 기존 티켓 정렬, 열기, 삭제, 답장 배지 동작은 변하지 않는다.
