# 세입자 공지사항 반응형 목록 디자인

## 목표

`/tenant/messaging/02`를 기존 공지 상세 화면에서 세입자용 공지 목록 화면으로 전환한다. 첨부된 Stitch 모바일 시안의 정보 구조를 따르되, 룸로그의 공유 디자인 토큰과 실제 공지 API 계약을 사용하고 데스크톱까지 자연스럽게 재배치되는 반응형 화면으로 만든다.

## 범위

- `/tenant/messaging/02`는 `id` 쿼리 유무와 관계없이 공지 목록을 표시한다.
- 공지 카드를 선택하면 `/tenant/messaging/02/[id]` 상세 화면으로 이동한다.
- 기존 상세 화면의 읽음 처리, 긴급 공지 확인, 원문 표시, 공지 문의 기능을 새 상세 경로에서 보존한다.
- 기존 받은함(`/tenant/messaging/00`)과 마이페이지 공지 카드가 새 상세 경로를 사용하도록 링크를 갱신한다.
- API, 데이터베이스, Docker, 배포 설정은 변경하지 않는다.

## 라우팅과 셸

현재 `apps/web/src/app/tenant/messaging/layout.tsx`가 모든 메시징 화면을 고정 `PhoneFrame`으로 감싸므로 `/02`만 유동 폭으로 만들 수 없다. 공용 레이아웃은 인증만 담당하도록 축소하고, 기존 `/00`, `/01`, `/e0` 화면은 재사용 가능한 폰 프레임 래퍼를 각 페이지에서 명시적으로 사용한다. `/02`와 `/02/[id]`는 새 반응형 공지 셸을 사용한다.

이 분리는 다른 메시징 화면의 390×844 외형을 유지하면서 공지 목록만 모바일 전체 폭과 데스크톱 콘텐츠 폭을 사용할 수 있게 한다. 경로 이동이나 인프라 변경은 없다.

## 목록 화면

### 헤더와 탐색

- 모바일에서는 뒤로가기, `공지사항` 제목, 검색 토글을 한 줄에 배치한다.
- 데스크톱에서는 제목과 설명을 왼쪽에, 검색 필드를 오른쪽에 배치한다.
- 검색이 활성화되면 `q` 쿼리로 제목, 본문, 발신자를 대소문자 구분 없이 검색한다.
- 모바일 하단 내비게이션은 `홈`, `공지`, `고정비`, `내 정보`를 제공하며 공지를 활성 상태로 표시한다.
- 데스크톱에서는 하단 내비게이션을 상단 보조 탐색으로 재배치한다.

### 필터와 정렬

- 필터는 현재 `Announcement` 계약에서 표현 가능한 `전체`, `긴급`, `건물`, `생활`, `행사`만 제공한다.
- `긴급`, `생활`, `행사`는 `category`, `건물`은 `scope === "building"`으로 판정한다.
- 활성 필터는 `filter` 쿼리로 유지하여 새로고침과 직접 링크에서도 상태가 보존된다.
- 결과는 긴급 공지를 먼저 배치하고, 같은 우선순위에서는 `sentAt` 최신순으로 정렬한다.

### 공지 카드

- 카드에는 유형 아이콘, 유형 라벨, 제목, 본문 한 줄, 발신자, 날짜, 미확인 상태를 표시한다.
- 긴급 또는 확인 필수 공지는 기본 카드보다 강한 기본색 강조와 워터마크 아이콘을 사용한다.
- 카드 전체가 상세 링크이며 키보드 포커스와 명확한 접근성 이름을 제공한다.
- 모바일은 한 열, 태블릿 이상은 두 열로 배치한다.
- 카드 색상, 여백, 그림자, 곡률은 `packages/ui/src/tokens.css`의 `var(--...)` 값만 사용하며 raw hex를 추가하지 않는다.

### 보조 상태

- 검색 또는 필터 결과가 없으면 조건을 초기화할 수 있는 빈 상태를 표시한다.
- API 조회 실패 시 기존 `listAnnouncements()`의 데모 폴백을 유지한다.
- 목록 아래에는 첨부 시안의 `도움이 필요하신가요?` 배너를 배치하고 일반 문의 대화 화면으로 연결한다.

## 상세 화면

- 기존 `/tenant/messaging/02/page.tsx`의 상세 UI와 서버 액션을 `/tenant/messaging/02/[id]/page.tsx`로 이동한다.
- 상세의 뒤로가기는 `/tenant/messaging/02` 목록으로 연결한다.
- 읽음/확인 후에는 동일한 `/tenant/messaging/02/[id]`로 돌아온다.
- 공지 문의 생성 후에는 기존과 동일하게 `/tenant/messaging/01?id=<threadId>`로 이동한다.
- 인증 오류와 데모 상세 폴백 동작은 유지한다.

## 컴포넌트 경계

- `announcement-list-model.ts`: 필터 값 정규화, 검색, 정렬, 상세 경로 생성을 담당하는 순수 함수 모듈.
- `AnnouncementListPage.tsx`: 목록 헤더, 필터, 카드 그리드, 도움 배너, 반응형 탐색을 렌더링.
- `AnnouncementListPage.module.css`: 토큰 기반 반응형 레이아웃과 상태 스타일.
- `MessagingPhoneFrame.tsx`: 기존 `/00`, `/01`, `/e0`의 고정 폰 셸을 보존.
- `/02/[id]/page.tsx`: 기존 공지 상세와 mutation 서버 액션 담당.

## 반응형 기준

- 767px 이하: 전체 폭 단일 열, 가로 스크롤 필터, 하단 고정 내비게이션.
- 768px 이상: 최대 콘텐츠 폭 안에서 2열 카드 그리드, 검색 필드 상시 노출, 상단 보조 탐색.
- 1180px 이상: 여백만 확장하고 카드 한 행의 정보 밀도는 유지한다.
- `prefers-reduced-motion`에서는 눌림과 전환 애니메이션을 제거한다.

## 테스트와 검증

1. `announcement-list-model.spec.ts`에서 상세 경로 인코딩, 긴급 우선 최신순 정렬, 필터, 검색, 잘못된 필터의 `전체` 폴백을 실패 테스트로 먼저 고정한다.
2. 메시징 화면 계약 테스트에서 `/02` 목록과 `/02/[id]` 상세, 기존 폰 프레임 보존, 토큰 기반 스타일을 확인한다.
3. 대상 테스트를 RED → GREEN 순서로 실행한다.
4. `pnpm test:web`과 `bash scripts/verify.sh`를 실행한다.
5. Docker web 이미지를 재빌드한 뒤 `/tenant/messaging/02?id=mann_7b7c0d2087`가 목록을 표시하는지 모바일·데스크톱 뷰포트에서 확인한다.
6. 검증이 통과한 기능 단위로 커밋하고 `kms-venant-notice`에 푸시한 뒤 원격과 `0 0` 동기화 상태를 확인한다.
