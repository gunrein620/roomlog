# 임차인 가구 배치 시뮬레이션 (RoomPlan × Object Capture)

> **상태**: 설계 제안 (2026-07-17 작성). 아직 미착수 — 현재 상태 아님.
> **한 줄**: 임차인이 자기 가구를 이사갈 방(3D 투어)에 미리 넣어보고 "들어가나 / 어떻게 보이나"를 확인하는 임차인용 도구.

---

## 문제 · 가치

이사의 가장 큰 불확실성 = **"내 침대·소파가 저 방에 들어갈까? 자리가 날까?"**. 지금은 답할 방법이 없어 직접 가서 줄자를 대거나, 이사하고 나서야 안다.

이미 있는 것: 매물의 **metric splat 3D 투어**(임대인이 캡처앱으로 촬영). 여기에 임차인이 **자기 가구를 얹어** 계약·이사 전에 확인 → 이사 결정 도구.

스코프: **개인 임대인 매물 + 임차인**. 기업 SaaS 아님.

---

## 핵심 통찰 — RoomPlan 박스가 Object Capture를 보정한다

애플 두 기능을 엮으면 서로의 약점을 메운다:

| 애플 API | 주는 것 | 약점 |
|---|---|---|
| **RoomPlan** (iOS 17+, LiDAR) | 방 스캔 → 벽·문·창 + **가구를 라벨·실치수 박스로 자동 감지** (ARKit 미터) | 회색 박스 — 미감 없음 |
| **Object Capture** (iOS 17+) | 물체 사진측량 → **포토리얼 메시(USDZ)** | **절대 스케일 모호** · 물체 주위를 빙 돌아야 함 |

**엮기**: RoomPlan 박스(실측 치수)에 Object Capture 메시를 끼워 맞추면, **박스가 메시의 스케일·방향을 캘리브레이션**한다. → Object Capture의 스케일 모호성이 공짜로 해결. 단순 편의가 아니라 두 기술이 상호 보정하는 구조.

## 흐름 — 프로그레시브 피델리티

```
① RoomPlan로 내 현재 방 1스캔
     → 침대·소파·책상이 라벨+실치수 박스로 자동 감지 → "내 가구함"(무입력)
② 아끼는 것만 선택 → Object Capture로 실물 정밀 스캔
     → 포토리얼 메시 → 해당 박스에 끼워맞춤(박스=스케일·방향 앵커)
③ 대상 매물 splat(정합=미터)에 인벤토리 배치
     → 박스=회색 프록시, 스캔한 건 포토리얼 (섞여서)
     → collision.ts로 "들어감/걸림/겹침" 판정
④ 결과·비교 — "다 들어가요 ✓ · 침대 옆 여유 40cm", 여러 후보 매물 비교
```

**모든 가구는 정확한 치수(박스), 아끼는 건 사실적(메시)** — 정확성은 전부, 미감은 골라서.

## 설계 원칙 · 제약

- **RoomPlan은 "품목"을 주지 "배치"를 주지 않는다.** 현재 방의 가구 배열은 새 방엔 무의미 → **아이템(치수·타입)만 추출**, 현재 위치는 버리고 임차인이 대상 방에서 새로 배치.
- **Object Capture는 물체를 빙 돌아야 한다** → 벽붙이 소파·침대는 뒷면·밑면을 못 찍어 부분 메시. **접근 가능한 몇 개만** 포토리얼, 벽붙이는 박스로 두는 게 실용적. (= "선택해서 채우기"의 근거.)
- **미터의 진짜 출처는 정합.** 대상 방 splat은 **2점 정합(solveSimilarity)의 배율 = 도면거리/​splat거리**로 도면의 실미터 프레임에 보정된다. 도면이 실치수여야 정확 (yong의 2D 도면 파싱 A트랙이 정확도를 받침). → **spz 네이티브 스케일을 신뢰할 필요 없음**; ARKit 네이티브 미터는 도면 없는 video-only 폴백일 뿐.
  - **[2026-07-20 실기 검증 — 가구 쪽은 이 서술이 뒤집혔다]** Object Capture USDZ는 **그 자체로 실측 미터**다. 실캡처 산출물(의자) 검사 결과 `metersPerUnit = 1` · `upAxis = "Y"` · **`xformOp` 0개**(스케일 보정 전무) · `extent = 0.727 × 0.806 × 0.775 m`, 줄자 실측과 일치 확인. → **RoomPlan 박스의 역할은 "스케일 구제"가 아니라 "교차검증"으로 내려간다.**
  - **덤으로 앵커 규약 확정**: 바닥이 정확히 `y ≈ 0`(−2.8e−8), X·Z가 원점 대칭 → **원점 = 발자국 중심, 바닥 = y0**. RoomPlan 박스도 중심 기준이라 **박스↔메시 정렬은 "xz 중심 정렬 + 바닥 안착"으로 끝나고 남는 자유도는 yaw 하나**.
  - **단, USDZ bbox를 치수로 그대로 쓰지 말 것**: X·Z 범위가 소수점 6자리까지 대칭(±0.36330712 / ±0.387259)인데 이는 물체 형태가 아니라 **촬영 시 사용자가 맞춘 바운딩 박스 벽면에 메시가 잘린 흔적**이다(= bbox는 실물의 과대추정). Y 상단만 비대칭이라 실제 높이로 신뢰할 수 있었다. → **치수의 정본은 RoomPlan 박스**(실물 실측), USDZ는 기하 제공 + 스케일 교차검증용.
- **임차인 기본 경험 = 모바일 웹.** 로그인·배치·비교는 웹. **앱은 캡처 전용**(RoomPlan/Object Capture) — 데모폰엔 이미 깔려 있음.

## 기존 자산 재사용 (신규 최소화)

| 부품 | 상태 |
|---|---|
| metric splat 투어 (걷기 1인칭) | ✅ 있음 |
| 가구 실치수 `sizeMm` | ✅ 있음 |
| 충돌·footprint·배치보정 `collision.ts` | ✅ 있음 |
| 정합 = 미터 보정 `similarity-solve.ts` | ✅ 있음 |
| 정합 도면 가구가 투어에 노출 | ✅ 있음 (검증됨) |
| **앱: RoomPlan + Object Capture 캡처** | ❌ 미착수 (현재 Record3D depth만) |
| **아이템에 `meshUrl?`(GLB) 옵션** | ❌ 신규 (없으면 박스, 있으면 포토리얼) |
| **임차인 흐름 화면 + 계정 소유 가구함** | ❌ 신규 |

## 열린 질문 · 리스크

- **Object Capture 스케일·방향 정렬**: 박스에 메시 끼워맞출 때 종횡비 왜곡·정면 정렬(자동 주축 vs 수동 회전) 확정 필요.
- **RoomPlan 감지 누락/오분류**: 라벨 편집 UI 필요.
- **렌더 파이프라인**: USDZ → GLB 변환(three.js), splat + 메시 공존 성능(모바일).
- **도면 정확도 의존**: 정합 배율은 도면이 실치수일 때만 정확 → A트랙(2D 도면 파싱) 정확도가 상류 의존.
- **디바이스**: RoomPlan = LiDAR(Pro), Object Capture on-device = 최신칩. 데모폰 충족 전제.
- **발열/캡처 시간**: RoomPlan 동일세션 + Object Capture 다각도 촬영 시간.

## 트랙 관계 (혼동 방지)

- **A (yong)**: 2D 도면 **이미지** 파싱(YOLO+객체그래프). 도면 그림 있는 매물용. 본 기능의 **미터 정확도를 받침**.
- **B**: 3D splat → 벽 추출. **도면 없는 video-only**의 자동 도면. (RoomPlan도 벽을 주므로 캡처 시엔 B 대체 가능.)
- **C (이 문서)**: 임차인 가구 배치. **A/B의 미터·벽 토대 위에 얹힘.**
