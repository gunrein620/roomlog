### 0. 공통 운영 원칙: 한 번 입력하고 역할별로 나눠 보기

- 임차인 앱과 임대인/관리자 앱은 같은 서버와 DB를 사용한다.
- 계약서, 입주 전 사진, 민원, 하자 사진, 수리 기록, 비용 기록은 중복 저장하지 않고 하나의 공용 기록으로 관리한다.
- 임차인이 입력한 내용은 관리자 화면에서 다시 입력하지 않고, 검토/수정/승인하는 방식으로 사용한다.
- 임차인이 접수한 민원/하자는 서버에서 하나의 티켓으로 생성되고, 임차인 화면과 임대인/관리자 화면에 역할별로 다르게 표시된다.
- AI 분석 결과도 역할별로 따로 생성하지 않고, 서버에서 한 번 생성한 결과를 임차인 화면과 관리자 화면에 다르게 보여준다.
- 단, 관리자 수정, 이의제기, 추가 사진, 상태 변경처럼 새 정보가 생긴 경우에만 AI 분석을 부분 갱신한다.
- AI는 책임 소재를 확정하지 않고 `임대인 책임 가능성`, `임차인 책임 가능성`, `판단 어려움`으로만 표시한다.
- 책임 가능성, 긴급도, 비용 분류, 업체 출동 여부는 관리자가 최종 확인하거나 사전 설정에 따라 처리한다.
- 자동 문자 발송, 업체 출동, 포인트 차감, 결제 처리는 반드시 관리자 설정 또는 승인 단계를 거친다.
- 모든 상태 변경, 답변, 추가 정보 요청, 업체 배정, 수리 완료 기록은 호실별 타임라인에 저장한다.

### 1. 공유 기록 검토 및 아카이빙

- 임차인이 업로드한 계약서, 입주 전 사진, 하자 사진, 민원 기록을 관리자 화면에서 확인한다.
- 관리자는 이미 등록된 자료를 다시 업로드하지 않고, 누락된 자료만 보완한다.
- 계약서가 임차인 앱에서 먼저 등록된 경우 관리자는 OCR 결과와 핵심 추출 항목을 검토/확정한다.
- 관리자가 먼저 계약서를 등록한 경우 임차인은 동일한 계약서 기록을 본인 화면에서 확인한다.
- 계약서는 원본 파일, OCR 전문, 핵심 요약본을 함께 보관한다.
- AI가 계약서의 핵심 항목을 추출한다.
    - 계약 기간
    - 보증금
    - 월세
    - 관리비
    - 납부일
    - 계좌 정보
    - 특약
    - 수리 책임 관련 조항
    - 원상복구/청소비 관련 조항
    - 계약 만료일
- 추출 결과가 불확실한 항목은 `확인 필요`로 표시한다.
- AI가 참고한 조항은 `근거 보기` 형태로 확인할 수 있게 한다.
- 임대인은 보유한 사진, 계약서, 수리 이력 등을 근거로 책임 가능성 판단에 이의제기할 수 있다.

예시 표시:

```
책임 가능성: 임대인 확인 필요
근거 후보:
- 계약서에 에어컨이 기본 옵션으로 기재됨
- 입주 전 사진에 동일 증상 없음
- 최근 동일 설비 수리 이력 없음
주의: 최종 판단은 관리자 확인 후 확정됩니다.
```

### 2. 건물/호실/임차인 기본 관리

- 관리자가 보유한 여러 임대 공간을 등록하고 건물별, 호실별로 관리한다.
- 각 호실에 연결된 임차인 정보를 관리한다.
    - 이름
    - 연락처
    - 입주일
    - 계약 상태
    - 거주 여부
    - 연결된 호실 정보
- 계약 및 납부 정보를 호실별로 관리한다.
    - 계약 기간
    - 보증금
    - 월세
    - 관리비
    - 납부일
    - 계좌 정보
    - 갱신 여부
- 호실/세대별 타임라인을 제공한다.
    - 계약서
    - 입주 전 사진
    - 하자 사진
    - 민원 기록
    - 수리 기록
    - 채팅 기록
    - 비용/영수증 기록
    - 납부 기록
    - 상태 변경 기록
    - 추가 정보 요청 기록
- 임차인이 등록한 기록과 관리자가 등록한 기록을 같은 타임라인에서 확인한다.
- 기존 관리 데이터를 엑셀/CSV로 일괄 등록할 수 있도록 확장한다.

---

### 3. 임차인 초대 및 호실 연결

- 관리자는 링크 또는 QR을 통해 임차인을 서비스에 초대할 수 있다.
- 초대된 임차인은 본인이 거주하는 호실에 연결된다.
- 임차인 연결 후 계약서, 공지, 민원, 납부 정보가 해당 호실 기준으로 표시된다.
- 잘못 연결된 경우 관리자가 호실 연결을 수정하거나 해제할 수 있다.
- 임차인이 이미 계약서나 사진을 등록했다면, 초대 후 해당 기록을 관리자 호실 데이터와 연결한다.
- 연결된 호실 기준으로 이후 민원, 사진, 청구, 납부, 수리 기록이 하나의 타임라인에 누적된다.

---

### 4. 홈 / 자산 현황 대시보드

- 관리자가 보유한 전체 자산 현황을 한눈에 확인한다.
- 전체 건물 수, 전체 호실 수, 입주 완료 호실 수, 공실 수를 표시한다.
- 총 보증금, 월 예상 수익, 이번 달 실제 수납액을 표시한다.
- 이번 달 수납률을 표시한다.
    - 수납 완료 금액
    - 연체/미납 금액
    - 수납 대기 금액
    - 공실 손실 금액
- 미납, 긴급 민원, 계약 만료, 진행 중 수리 등을 한눈에 확인한다.
- 미처리 업무 목록을 제공한다.
    - 오늘 처리해야 할 티켓
    - 확인 대기
    - 결제 대기
    - 추가 정보 요청 건
    - 사진 업로드 대기 건
    - 업체 배정 대기 건
- 월별 수익 추이 그래프를 제공한다.
    - 6개월 / 1년 단위 전환 가능
- 공실률, 입주율, 미납률, 민원 처리율, 월별 수리비, 월별 수익/비용 등을 그래프로 표시한다.
- 임대 현황 보고서로 이동할 수 있는 요약 카드를 제공한다.

---

### 5. 전체 건물 관리

- 전체 건물 목록을 카드 또는 리스트 형태로 확인한다.
- 건물명 또는 주소로 검색할 수 있다.
- 건물별 주요 상태를 표시한다.
    - 수납률
    - 공실 수
    - 연체 건수
    - 진행 중 민원 수
    - 긴급 민원 수
    - 리스크 상태
- 정렬 기능을 제공한다.
    - 리스크 많은 순
    - 수납률 높은 순
    - 월세 높은 순
    - 공실 많은 순
    - 긴급 민원 많은 순
- 필터 기능을 제공한다.
    - 담당자별 필터
    - 건물 유형별 필터
    - 지역별 필터
    - 상태별 필터
- 건물 상세 화면에서는 해당 건물의 호실, 임차인, 민원, 수납, 공실, 리포트를 확인한다.

---

### 6. 청구 관리

- 전체 청구 목록을 확인한다.
- 청구서, 지출 요청, 견적 검토 목록을 구분해 확인한다.
- 청구서별 금액, 항목 수, 작성일, 담당자/작성자를 표시한다.
- 확인이 필요한 청구 금액과 대기 중인 청구 건수를 표시한다.
- 기간, 건물, 상태 기준으로 필터링할 수 있다.
- 청구 상태를 관리한다.

```
작성됨 → 발송 대기 → 발송 완료 → 수납 대기 → 일부 납부 → 납부 완료 → 연체
```

- 관리비, 월세, 수리비, 기타 비용을 항목별로 분리해 관리한다.
- 임차인 화면에는 같은 청구 데이터를 바탕으로 이번 달 청구·납부·연체 정보가 표시된다.
- 미납 세대에는 AI가 납입 안내문 또는 독촉문 초안을 생성한다.
- 관리자가 확인 후 문자, 앱 알림, 채팅으로 발송한다.
- 발송 이력은 호실별 타임라인과 청구 기록에 저장한다.

---

### 7. 재무 / 수금 현황 관리

- 이번 달 수금 현황을 확인한다.
- 목표 금액 대비 수금률을 표시한다.
- 연체 미납 금액과 공실 손실 금액을 표시한다.
- 최근 입금 내역을 확인한다.
    - 입금자명
    - 호실
    - 건물명
    - 입금 시간
    - 입금 금액
    - 입금 상태
- 입금 내역과 호실/청구서를 자동 또는 수동으로 매칭한다.
- 입금자명이 다르거나 금액이 다른 경우 확인 필요 상태로 표시한다.
- 수금 현황, 연체 관리, 입금 내역, 관리 요청 메뉴를 분리한다.
- 초기 MVP에서는 관리자 입력 또는 엑셀/CSV 업로드 기반으로 처리하고, 이후 계좌 연동을 확장할 수 있다.

---

### 8. 연체 관리

- 전체 연체 현황을 확인한다.
- 총 미수금, 연체 건수, 연체 세대 수를 표시한다.
- 연체 상태를 단계별로 분류한다.
    - 경미: 납부일 1~7일 경과
    - 주의: 8~30일 경과
    - 심각: 31일 이상 경과
- 연체 등급별 필터를 제공한다.
- 연체 세대 목록을 표시한다.
    - 호실
    - 임차인명
    - 건물명
    - 납부일 기준 경과 일수
    - 미납 금액
    - 관련 청구서 번호
    - 일부 납부 내역
    - 담당자/처리자
- AI가 연체 임차인에게 보낼 납입 독촉문 초안을 생성한다.
- 관리자가 확인 후 문자, 앱 알림, 채팅으로 발송한다.
- 발송 이력은 호실별 타임라인과 청구 기록에 저장한다.

---

### 9. 민원/하자 티켓 검토 및 처리

- 임차인이 민원/하자를 접수하면 서버에서 하나의 티켓을 생성한다.
- 리얼타임 챗봇, 음성 챗봇, 콜봇으로 접수된 민원도 동일한 티켓으로 생성한다.
- 관리자 화면에서는 새로 내용을 입력하지 않고, 생성된 티켓을 검토하고 처리한다.
- 티켓에는 접수 채널을 표시한다.
    - 앱 직접 입력
    - 리얼타임 챗봇
    - 음성 챗봇
    - 콜봇
    - 관리자 수동 등록
- 챗봇으로 접수된 경우 음성 대화 기록, 채팅 메시지, 사진 첨부 내역을 함께 확인한다.
- 콜봇으로 접수된 경우 통화 녹음, 음성 전사 텍스트, AI 요약 결과를 함께 확인한다.
- 콜봇 접수 중 사진이 필요한 경우 임차인에게 사진 업로드 링크를 발송하고, 사진이 업로드되면 같은 티켓에 연결한다.
- AI가 임차인 입력 내용을 요약한다.
- AI가 민원 유형을 자동 분류한다.
    - 하자
    - 소음
    - 설비
    - 납부
    - 계약
    - 공용공간
    - 기타
- AI가 긴급도를 분류한다.
    - 1순위 긴급
    - 2순위 우선 처리
    - 3순위 일반 수리
    - 4순위 일반 문의
- 위험 키워드가 포함된 경우 긴급도를 강제로 상향한다.
- 임차인이 사진을 첨부한 경우 AI가 현재 사진 속 문제 후보를 분석한다.
    - 누수
    - 곰팡이
    - 벽지 훼손
    - 바닥 손상
    - 에어컨 문제
    - 창틀 문제
- 이 사진 분석은 민원/하자 접수 티켓 안에서 실행하며, 이전 사진 비교가 필요한 경우 같은 호실/공간/항목의 체크리스트 사진 또는 기존 하자 사진을 함께 조회한다.
- AI는 현재 사진과 이전 사진을 비교해 `기존 하자 가능성`, `신규 발생 가능성`, `비교 어려움`, `추가 사진 필요` 중 하나의 참고 상태를 제공한다.
- 관리자는 티켓 상세에서 이전 체크리스트 사진, 현재 접수 사진, 수리 완료 사진을 나란히 확인할 수 있다.
- 사진 각도, 조명, 해상도, 촬영 위치가 크게 다르면 비교 결과를 확정하지 않고 추가 사진 요청 대상으로 처리한다.
- 계약서, 입주 전 사진, 기존 수리 이력 등을 참고해 책임 가능성을 표시한다.
- 관리자는 AI가 생성한 접수 요약, 민원 유형, 긴급도, 사진 분석/이전 사진 비교 결과를 검토한다.
- 관리자는 AI 결과를 그대로 확정하거나 수정할 수 있다.
- AI 요약이 부정확한 경우 관리자가 직접 수정할 수 있다.
- 임차인이 AI 판단에 이의제기한 경우 관리자 화면에서 이의제기 사유를 확인한다.
- 관리자가 수정한 결과는 티켓 이력과 모델 개선용 피드백으로 저장한다.
- 반복 민원/수리 기록 요약 카드를 제공한다.
    - 최근 3개월 내 동일 유형 민원 2회 이상 발생 시 반복 민원으로 표시

접수 채널 표시 예시:

```
접수 채널: 리얼타임 챗봇
입력 방식: 음성 + 채팅 + 사진
AI 요약: 305호 화장실 천장에서 물이 떨어진다는 민원
사진 여부: 첨부됨
사진 비교 결과: 입주 전 사진에는 동일 증상 없음
비교 상태: 비교 가능
긴급도: 1순위 긴급
관리자 확인 필요: 예
```

콜봇 접수 표시 예시:

```
접수 채널: 콜봇
통화 기록: 저장됨
전사 내용: 화장실 천장에서 물이 계속 떨어지고 있습니다.
사진 필요 여부: 필요
처리 상태: 사진 업로드 링크 발송 대기
주의: 사진 업로드 후 최종 검토 필요
```

### 10. 티켓 상태 관리 및 답변 처리

- 민원/하자 티켓의 처리 흐름을 관리한다.
- 챗봇 또는 콜봇으로 접수된 티켓도 동일한 상태 흐름으로 관리한다.
- 상태값은 다음과 같이 관리한다.

```
접수 → 검토 → 추가 정보 요청 → 업체 배정 → 견적 확인 → 수리 중 → 완료 → 재요청
```

- 각 티켓에는 담당자, 관련 호실, 임차인, 긴급도, 접수 채널, 처리 기한을 표시한다.
- 관리자는 티켓별로 답변, 추가 정보 요청, 업체 배정, 견적 승인, 완료 확인을 처리할 수 있다.
- 사진이 부족하거나 정보가 불명확한 경우 임차인에게 추가 사진 또는 추가 설명을 요청한다.
- 콜봇으로 접수된 티켓에서 사진이 필요한 경우 사진 업로드 링크를 발송한다.
- 임차인이 사진을 업로드하면 기존 티켓에 자동 연결하고 AI 사진 분석 및 이전 사진 비교 결과를 부분 갱신한다.
- AI가 상황에 맞는 관리자 답변 초안을 생성한다.
    - 접수 안내
    - 추가 사진 요청
    - 추가 설명 요청
    - 방문 일정 조율
    - 업체 배정 안내
    - 수리 완료 안내
- 관리자가 수정 후 임차인에게 전송한다.
- 임차인에게 전송되는 답변은 앱 알림, 채팅, 문자 등으로 확장할 수 있다.
- 실제 문자 발송 또는 외부 채널 발송은 관리자 확인 후 진행한다.
- 모든 상태 변경과 답변 기록은 호실별 타임라인에 저장된다.
- 챗봇 대화 기록, 콜봇 통화 전사, 사진 업로드 기록도 같은 티켓 타임라인에 연결한다.
- 임대인/관리자 내부 상태와 임차인에게 보여줄 상태 문구는 다르게 표시할 수 있다.
- 임차인 앱에는 같은 티켓의 상태가 `접수됨`, `검토중`, `추가정보 요청`, `업체 배정`, `수리중`, `완료`, `재요청` 형태로 표시된다.

상태 표시 매핑 예시:

| 관리자 상태 | 임차인 표시 |
| --- | --- |
| 접수 | 접수됨 |
| 검토 | 검토중 |
| 추가 정보 요청 | 추가정보 요청 |
| 업체 배정 | 업체 배정 |
| 견적 확인 | 처리 준비중 |
| 수리 중 | 수리중 |
| 완료 | 완료 |
| 재요청 | 재요청 |

추가 정보 요청 예시:

```
요청 유형: 추가 사진 요청
요청 내용: 천장 누수 부위를 가까이서 한 장, 화장실 전체가 보이게 한 장 촬영해주세요.
전송 채널: 앱 알림 + 채팅
상태 변경: 추가정보 요청
```

처리 흐름:

```
티켓 접수
→ AI 요약/분류 확인
→ 관리자 검토
→ 필요 시 추가 정보 요청
→ 임차인 사진/설명 추가
→ AI 사진 분석/이전 사진 비교 부분 갱신
→ 업체 배정 또는 답변 처리
→ 수리 중
→ 완료 확인
→ 기록 저장
```

### 11. 업체/수리기사 관리

- 자주 쓰는 업체, 연락처, 담당 분야, 견적/수리 이력을 저장한다.
- 견적 비교/승인 기능을 제공한다.
- 임차인 책임이 아닌 하자이면서 긴급한 사항일 경우 협력업체 연결과 연계한다.
- 협력업체 부재 또는 파견 불가 시 여러 업체에게 견적을 요청할 수 있다.
- 업체에게 전달되는 정보는 수리에 필요한 범위로 제한한다.
- 업체에게 전달되는 정보는 하자 사진, 증상 요약, 방문 가능 시간, 대략적 위치를 기본으로 한다.
- 계약서 전문, 상세 주소, 연락처는 필요한 경우에만 단계적으로 제공한다.
- 업체 전달 전 관리자 또는 임차인은 전달될 정보 범위를 확인하고 동의한다.
- 관리자 승인을 받고 처리되는 것을 기본값으로 한다.
- 긴급도와 관리자 설정에 따라 승인 없이 바로 업체 선정 및 결제가 가능하도록 설정할 수 있다.
- 수리 완료 후 관리자 또는 임차인이 완료 확인을 진행한다.
- 업체별 응답 속도, 견적 금액, 완료 이력, 만족도 기록을 저장할 수 있다.

### 12. 비용/영수증 기록 정리

- 수리비, 관리비, 기타 비용 영수증을 업로드한다.

- 온라인 영수증 발행 및 파일 업로드도 지원한다.

- 비용 유형을 자동 분류한다.
    - 수리비
    - 관리비
    - 청소비
    - 공용설비 비용
    - 기타 비용
- 추출된 비용은 월간 리포트와 호실별 기록에 반영된다.
- 비용 분류 결과가 불확실한 경우 `확인 필요` 상태로 표시한다.
- 임차인에게 공개 가능한 관리비 사용 내역은 같은 비용 데이터를 기반으로 임차인 앱에 표시한다.

### 13. 공지사항 및 커뮤니티/채팅 기능

- 관리자는 전체 건물, 특정 건물, 특정 호실, 미납 세대에 공지를 발송할 수 있다.
- AI가 공지문 또는 납입 독촉문 초안을 생성한다.
    - 관리자가 주제나 내용을 간단히 입력
    - AI가 문장 형태로 정리
    - 관리자가 수정 후 발송

### 14. 주간 / 월간 / 분기별 관리 리포트 생성

- AI가 관리 현황을 양식에 맞게 정리한다.
- 리포트에는 다음 내용을 포함한다.
    - 체결/만료 계약
    - 월세 납부 내역
    - 연체 현황
    - 민원 처리 현황
    - 수리비용
    - 공실 현황
    - 수익/비용 요약
    - 반복 민원
    - 다음 조치 사항
- 생성된 리포트는 관리자 대시보드에서 확인할 수 있다.
- PDF 또는 Excel로 내보낼 수 있다.
- 리포트는 별도 수기 입력이 아니라 누적된 공용 기록을 기반으로 생성한다.
- 챗봇/콜봇으로 접수된 민원도 동일한 민원 처리 통계에 포함한다.
- 추가 정보 요청, 업체 배정, 완료까지 걸린 시간도 처리 지표로 활용할 수 있다.

예시 요약:

```
이번 달 수납률은 91%이며, 연체 세대는 4건입니다.
에어컨 관련 민원이 3건 반복되어 여름철 사전 점검이 필요합니다.
다음 달 계약 만료 예정 호실은 2건입니다.
```

### 16. 관리자 질의 챗봇

- 관리자는 자연어로 운영 데이터를 조회할 수 있다.
- 챗봇은 DB에 저장된 계약, 납부, 민원, 수리, 비용 기록을 기반으로 답변한다.
- 관리자 질의 챗봇은 임차인 민원 접수 챗봇과 역할이 다르다.
- 임차인 챗봇/콜봇은 민원을 접수하는 기능이고, 관리자 질의 챗봇은 이미 저장된 운영 데이터를 조회하거나 정리하는 기능이다.
- 관리자는 챗봇 또는 콜봇으로 접수된 민원을 조건별로 조회할 수 있다.
- 예시 질문:

```
이번 달 미납된 호실 알려줘
302호 최근 민원 내역 보여줘
연체 30일 넘은 세대만 정리해줘
이번 달 수리비 많이 나온 건물 알려줘
미납 세대에 독촉 문자 초안 작성해줘
콜봇으로 접수된 미처리 민원만 보여줘
사진 첨부가 아직 안 된 하자 티켓만 보여줘
긴급도 1순위 민원 중 아직 업체 배정 안 된 건 보여줘
이번 주 챗봇으로 접수된 민원 요약해줘
추가 정보 요청 상태에서 멈춘 티켓만 보여줘
```

- 챗봇이 독촉문, 공지, 답변 초안을 생성하더라도 실제 발송은 관리자 확인 후 진행한다.
- 챗봇 답변에는 조회 기준 기간과 사용한 데이터 범위를 함께 표시한다.
- 이미 생성된 AI 요약, 티켓 분류, 비용 분류 결과가 있으면 다시 분석하지 않고 저장된 결과를 우선 활용한다.
- 챗봇 답변에서 바로 티켓 상세, 호실 상세, 임차인 상세 화면으로 이동할 수 있게 한다.
- 관리자 질의 결과는 참고용이며, 상태 변경이나 발송 같은 실제 조치는 관리자 확인 후 실행한다.

### 17. 권한 및 데이터 관리

- 임대인/관리자는 본인이 소유하거나 담당하는 건물과 호실만 조회할 수 있다.
- 직원 또는 담당자 계정은 역할에 따라 접근 권한을 제한한다.
    - 전체 관리자
    - 건물 담당자
    - 협력업체
- 계약서, 계좌, 연락처, 주소 등 민감정보는 마스킹 또는 접근 제한을 적용한다.
- 기록 수정 시 누가, 언제, 어떤 항목을 변경했는지 로그로 남긴다.
- 계약 종료 후 보관 기간이 지난 데이터는 삭제 또는 비식별화할 수 있다.
- 하나의 원천 데이터에 대해 임차인 화면, 관리자 화면, 업체 화면이 서로 다른 권한으로 접근한다.
- 임차인의 이의제기, 관리자 수정값, AI 판단 결과는 변경 이력으로 저장한다.
- 업체 전달 전 전달 정보 범위와 동의 여부를 기록한다.

### 19. 티켓 운영 기준 및 처리 우선순위 보강

- 관리자 화면에는 티켓의 긴급도, 접수 채널, 책임 가능성, 반복 민원 여부를 함께 표시한다.
- 긴급도별 권장 처리 기한을 두고, 기한이 가까워지거나 초과된 티켓은 대시보드 상단에 노출한다.
- 담당자가 지정되지 않은 티켓, 추가 정보 요청 후 멈춘 티켓, 업체 배정 대기 티켓은 별도 미처리 목록으로 관리한다.
- 긴급도 1순위 티켓은 관리자 확인 전이라도 즉시 알림 대상으로 분류한다.
- 긴급도 2순위 티켓은 빠른 일정 조율 또는 업체 확인 대상으로 분류한다.
- 긴급도 3~4순위 티켓은 일반 처리 큐에서 담당자와 처리 예정일을 지정한다.
- 관리자가 긴급도나 책임 가능성을 수정하면 수정 사유를 남기고, 임차인 화면에는 필요한 범위만 반영한다.
- 반복 민원은 단순 개별 수리가 아니라 건물/설비 개선 필요 이슈로 리포트에 포함한다.

권장 처리 기준 예시:

```
1순위 긴급: 즉시 확인 / 당일 조치 여부 결정
2순위 우선 처리: 1~2일 내 확인 및 일정 조율
3순위 일반 수리: 3~7일 내 처리 예정일 지정
4순위 일반 문의: 확인 후 답변 또는 예약 처리
```

### 20. 협력업체 배정, 견적, 완료 승인 흐름 보강

- 관리자는 티켓 상세에서 협력업체를 배정하거나 여러 업체에 견적을 요청할 수 있다.
- 업체 배정 전 전달할 정보 범위와 개인정보 포함 여부를 확인한다.
- 협력업체에는 수리에 필요한 최소 정보만 제공한다.
- 업체는 배정된 작업의 증상 요약, 사진, 방문 가능 시간, 대략적 위치, 관리자 요청 사항을 확인한다.
- 업체가 견적을 제출하면 관리자는 금액, 방문 가능 시간, 예상 소요 시간, 추가 확인 사항을 비교한다.
- 견적 승인 전에는 임차인 부담 가능성, 임대인 부담 가능성, 보증금/예치금/직접 결제 여부를 확인한다.
- 업체가 방문 일정을 확정하면 임차인에게 방문 안내가 발송된다.
- 수리 진행 중 상태 변경은 업체 또는 관리자가 업데이트할 수 있고, 최종 완료 승인은 관리자 또는 임차인 확인을 거친다.
- 수리 완료 후 사진, 영수증, 비용, 완료 메모를 티켓과 호실 타임라인에 저장한다.
- 임차인이 완료를 거절하거나 미해결을 신고하면 티켓을 재요청 상태로 되돌리고 업체 재방문 또는 관리자 재검토 대상으로 표시한다.

업체 처리 흐름:

```
업체 후보 선택
→ 전달 정보 범위 확인
→ 견적 요청
→ 견적 비교
→ 관리자 승인
→ 방문 일정 확정
→ 임차인 안내
→ 수리 진행
→ 완료 사진/영수증 등록
→ 완료 확인
→ 비용 기록 및 리포트 반영
```

### 21. 운영 예외, 감사 로그, 데이터 정정 보강

- 잘못 연결된 호실, 잘못 접수된 티켓, 중복 티켓, 취소된 티켓은 관리자 화면에서 정정할 수 있어야 한다.
- 정정 시 기존 기록을 삭제하지 않고 변경 이력으로 남긴다.
- 티켓 병합, 담당자 변경, 긴급도 수정, 책임 가능성 수정, 업체 재배정, 완료 취소는 모두 감사 로그에 기록한다.
- 임차인이 데이터 삭제를 요청한 경우 관리자는 보관 필요 기록과 삭제 가능 기록을 구분해 처리한다.
- 정산, 분쟁, 회계 보관이 필요한 기록은 삭제 대신 마스킹 또는 비식별화 상태로 전환할 수 있다.
- 계약서, 계좌, 연락처, 상세 주소, 얼굴, 우편물 등 민감정보가 포함된 자료는 원본 열람 권한과 마스킹 열람 권한을 분리한다.
- 직원 계정은 담당 건물/호실 범위 안에서만 티켓과 임차인 정보를 조회하도록 제한한다.
- 협력업체 계정은 배정된 작업 외의 임차인 정보, 계약 정보, 납부 정보에 접근할 수 없다.
- 관리자 질의 챗봇이 조회한 결과에도 권한 제한을 동일하게 적용한다.
- 상태 변경이나 발송 같은 실제 조치는 챗봇 답변만으로 자동 실행하지 않고 관리자 확인 단계를 거친다.

감사 로그 예시:

```
변경 항목: 긴급도
기존 값: 3순위 일반 수리
변경 값: 1순위 긴급
변경자: 관리자
변경 사유: 누수 진행 중으로 피해 확산 가능성 확인
반영 범위: 관리자 티켓 / 임차인 상태 안내 / 리포트
```

### 22. 에스크로 결제, 송금, 환불 운영

- 관리자는 견적 승인 후 수리비 결제 주체를 확정한다.
- 결제 주체는 `임대인`, `임차인`, `판단 대기`로 구분한다.
- 확정된 견적 금액은 수리 완료 전까지 안전결제 또는 예치금 상태로 관리할 수 있다.
- 임대인 부담 수리비는 임대인이 예치하고, 임차인 부담 수리비는 임차인이 예치한다.
- 긴급 수리에서 임대인 부담 가능성이 높고 사전 설정이 켜져 있으면 임대인 선결제 또는 자동결제를 적용할 수 있다.
- 수리 완료 후 관리자 또는 임차인 확인을 거쳐 예치금을 업체에 지급한다.
- 취소, 업체 미방문, 수리 미완료, 분쟁 발생 시 예치금을 환불 처리할 수 있다.
- 예치, 송금, 환불, 결제 실패, 지급 보류 기록은 티켓 타임라인과 감사 로그에 저장한다.
- 결제 상태는 `예치완료`, `수리중`, `지급대기`, `송금완료`, `환불완료`, `분쟁중`으로 관리할 수 있다.

에스크로 처리 흐름:

```
견적 승인
→ 비용 주체 확정
→ 예치금 결제
→ 업체 수리 진행
→ 완료 확인
→ 업체 송금 또는 환불/분쟁 처리
→ 비용 기록 및 감사 로그 저장
```

### 23. 데이터 삭제, 마스킹, 비식별화 처리

- 임차인이 데이터 삭제를 요청하면 관리자는 삭제 가능 항목과 보관 필요 항목을 구분해 검토한다.
- 삭제 가능 항목은 승인 후 삭제 처리하고, 보관이 필요한 항목은 보관 사유를 기록한다.
- 정산, 분쟁, 회계, 법적 보관이 필요한 기록은 즉시 삭제하지 않고 마스킹 또는 비식별화할 수 있다.
- 삭제 요청은 `접수`, `검토중`, `승인`, `반려`, `일부 보관`, `완료` 상태로 관리한다.
- 계약서, 사진, 계좌, 주소, 얼굴, 우편물 등 민감정보가 포함된 첨부파일은 마스킹 처리할 수 있다.
- 원본 파일 열람 권한과 마스킹본 열람 권한을 분리한다.
- 비식별화 처리 시 개인을 특정할 수 있는 이름, 연락처, 상세 주소, 계좌 정보 등을 제거하거나 대체한다.
- 삭제 승인, 반려, 마스킹, 비식별화 처리 결과는 감사 로그에 저장한다.
- 임차인 화면에는 삭제/보관 결과와 보관 사유를 필요한 범위에서 안내한다.

삭제 검토 예시:

```
요청 항목: 계약서 OCR, 하자 사진, 채팅 기록
삭제 가능: 일반 채팅, 중복 사진
보관 필요: 정산 관련 하자 티켓
보관 사유: 퇴실 정산 및 분쟁 대비
처리 방식: 원본 접근 제한 + 얼굴/주소 마스킹
```

### 24. 중복 티켓 후보 조회 및 병합

- 같은 호실, 같은 위치, 같은 유형의 민원이 짧은 기간 안에 반복 접수되면 중복 후보로 표시한다.
- 관리자는 중복 후보 목록에서 기존 티켓과 새 티켓의 내용, 사진, 접수 시각, 긴급도, 상태를 비교한다.
- 중복으로 판단되면 여러 티켓을 하나의 대표 티켓으로 병합할 수 있다.
- 병합 시 사진, 채팅, 추가 정보, 업체 배정, 상태 변경 이력은 대표 티켓 타임라인으로 연결한다.
- 병합된 티켓은 별도 삭제하지 않고 병합 이력을 남긴다.
- 임차인 화면에는 중복 접수로 인해 기존 티켓에 연결되었다는 안내를 표시한다.
- 병합 이력은 감사 로그와 리포트 통계에 반영한다.
- 이미 완료되었거나 분쟁 중인 티켓은 병합 제한 상태로 표시할 수 있다.

중복 병합 흐름:

```
중복 후보 감지
→ 관리자 후보 확인
→ 대표 티켓 선택
→ 중복 티켓 병합
→ 사진/채팅/상태 이력 연결
→ 임차인 안내
→ 감사 로그 저장
```

### 25. 티켓 담당자 배정 및 재배정

- 관리자는 티켓별 담당자를 지정할 수 있다.
- 담당자는 전체 관리자, 건물 담당자, 수리 담당자 등 역할에 따라 지정한다.
- 담당자가 없는 티켓은 미배정 상태로 대시보드에 표시한다.
- 긴급도 1순위 또는 처리 기한 초과 티켓은 담당자 지정이 우선 필요 업무로 표시된다.
- 담당자 변경 시 변경 전 담당자, 변경 후 담당자, 변경 사유를 기록한다.
- 담당자 변경 이력은 티켓 타임라인과 감사 로그에 저장한다.
- 담당자는 본인에게 배정된 티켓을 우선 처리 목록에서 확인할 수 있다.
- 담당자 권한 범위를 벗어난 건물/호실 티켓은 조회하거나 처리할 수 없도록 제한한다.

담당자 배정 예시:

```
티켓 ID: 5001
긴급도: 1순위 긴급
기존 담당자: 미배정
신규 담당자: 김관리자
배정 사유: 누수 긴급 확인 필요
처리 기한: 당일 확인
```
